# Bilibili WebOS TV UI 规范

## 1. 目标

本规范用于约束后续所有页面和功能的 UI 落地方式。  
设计来源以 `ui_ref/` 为准，落地实现以“遥控器优先、TV 场景优先、可复用组件优先”为准。

## 2. 总原则

- 所有新功能必须先判断是否符合 TV 大屏观看和遥控器操作习惯，不能套用移动端信息密度。
- 所有视觉实现优先复用现有 token 和通用组件，不允许每个页面各写一套颜色、圆角、阴影和焦点态。
- 所有 hover 语义必须先翻译成 D-pad 焦点语义，再考虑 Pointer 补充。
- 优先做“清晰、稳定、可聚焦”的 UI，不为了视觉效果破坏性能和遥控器体验。
- 浏览器端增强效果只能算辅助观察，最终视觉基线必须以 `webOS Simulator` 和真机可见效果为准，不能依赖较新的桌面浏览器能力单独成立。
- 对关键布局间距要优先采用 `webOS Simulator / 真机` 稳定可见的实现方式；页面主 section 堆叠、主按钮组、抽屉操作区、wrap chip 组等关键区域，禁止只依赖 `flex gap` 承担唯一间距语义，优先使用显式 sibling margin 或容器 margin 方案兜底。

## 3. 视觉语言

### 3.1 色彩

- 主品牌色：`#FB7299`
- 功能强调色：`#00A1D6`
- 主背景：`#131314`
- 低层表面：`#1C1B1C`
- 中层表面：`#201F20`
- 高层表面：`#2A2A2B`
- 最高层表面：`#353436`
- 主文本：`#E5E2E3`
- 次文本：`#DDBFC4`

### 3.2 表面层级

- 页面底色、导航、内容区、浮层必须通过表面层级区分，不使用 1px 实线分割区域。
- 浮层、控制层、登录面板优先使用半透明深色表面，可配合轻量 blur。
- 仅在必要时使用极弱描边，不能依赖高对比细线做结构分割。

### 3.3 版式

- 默认按 `1920x1080` 布局，外边距必须预留 TV 安全区。
- 首页和内容页优先使用横向泳道，不做密集纵向信息墙。
- Hero 区优先采用左文右图或左文右氛围背景的非对称布局。
- 文本层级必须足够大，保证 10-foot viewing 可读。

### 3.4 图标

- 全局操作图标统一使用 `Material Symbols Rounded`，禁止在同一套 TV UI 中混用多套图标库。
- 应用内图标名统一登记到 `src/app/iconRegistry.ts`；业务组件、页面组件和路由配置不直接散落书写原始图标字符串。
- 默认态图标以未填充态为基线；当前页、焦点态和主操作允许切到填充态并适度加重字重。
- 侧边导航、播放器控制项、搜索和筛选等关键操作优先使用“图标 + 文本”组合，不能只靠图标让用户猜语义。
- 复用的“图标 + 文本”按钮统一通过 `src/components/TvIconButton.tsx` 承载，避免在多个页面重复拼装相同结构。
- 禁止使用字母缩写、临时占位字符或截图占位替代正式图标进入可交付界面。

### 3.5 App 图标

- webOS 应用资源必须提供正式的 `public/icon.png` 和 `public/largeicon.png`，不能使用 `1x1` 占位图或透明空白图提交。
- `icon.png` 按 `80x80` 导出，`largeicon.png` 按 `130x130` 导出；两者应来自同一主视觉，不允许风格割裂。
- App 图标优先使用高识别度的几何符号和品牌色块，避免细字、小字、复杂插画和过强材质细节，确保在 TV 启动器里远看仍能认出。
- 图标应与应用内视觉语言保持一致，优先延续深色底、品牌粉、功能蓝等既有识别元素，但不能直接把页面截图或复杂界面缩成图标。
- 如果使用 AI 生成图标，只能把它当作初稿；入库前必须人工筛选，并导出为尺寸、背景和边缘都稳定的最终 PNG 资源。

## 4. 焦点与交互规则

- 每个主操作必须可通过遥控器稳定到达。
- 焦点态必须至少包含以下三项中的两项：缩放、提亮、外发光。
- 默认焦点反馈以 `1.05x` 放大和主品牌色外发光为基线。
- 不允许出现“视觉像按钮，但无法聚焦”的主操作。
- 返回键行为统一遵循：`弹窗 > 面板 > 页面 > 退出应用`。
- 搜索框、筛选项、播放器控制项都必须先设计焦点流，再设计样式。
- 焦点基础设施统一采用 `Section + Spatial Navigation` 模型，不再新增 `row / col` 坐标式焦点规则。
- 页面只声明焦点结构，不在业务组件里手写“按左跳到谁、按下猜到谁”的运算逻辑。
- 每个页面进入时，默认焦点必须先落在主内容区 section，而不是左侧导航。
- 左侧导航不能主动抢焦点；只有内容区 section 显式声明 `leaveFor.left = '@side-nav'` 时，才允许切回导航。
- Hero、按钮组、chip 网格、卡片网格、播放器控制条、抽屉和弹窗都必须先拆成 section，再定义 section 间关系。
- section 必须明确 `defaultElement` 或 `defaultFocus`；需要恢复上下文的列表区默认使用 `enterTo = 'last-focused'`。
- section 内优先做“当前区域内的稳定 4-way 导航”；跨 section 只能依赖显式 `leaveFor`，不能靠几何距离误判。
- 抽屉、弹窗、输入面板这类临时层一旦打开，必须接管焦点；关闭后必须恢复到触发它的元素。
- 焦点邻接关系优先通过统一属性声明，例如 `focusId`、`sectionId`、`focusLeft/right/up/down`，禁止散落使用临时 DOM selector 回焦点。
- Pointer 只做补充，不得成为任何主操作的唯一可达方式。

## 5. 组件复用规则

后续新增页面或功能时，优先复用或扩展以下组件，而不是重写：

- `AppShell`
- `SideNavRail`
- `FocusSection`
- `FocusButton`
- `TvIconButton`
- `HeroBanner`
- `SectionHeader`
- `MediaCard`
- `SearchComposer`
- `TvProgressBar`
- `PlayerControlBar`
- `PlayerSettingsDrawer`
- `TopbarProfilePill`
- `FollowingSummaryChips`
- `HomeChannelTabs`

如果现有组件不够用，应先抽象成新通用组件，再接入页面。

### 5.1 同类控件统一规则

- 顶部频道切换器、收藏夹切换器这类“页内切换 chip”必须复用同一套圆角矩形语义，不再页面各自长出胶囊形、细描边形、纯文本形多个近似版本。
- 主 CTA、搜索确认、Hero 主操作这类“主按钮”必须继续复用 `FocusButton` 的主按钮语义，不允许页面单独发明一套只在本页出现的主操作样式。
- 图标 + 文本的操作入口优先复用 `TvIconButton` 或基于 `FocusButton` 的统一结构，避免每页手拼一遍图标、标签和焦点态。
- 进度条、设置抽屉、账号摘要、搜索输入面板等复合控件，应优先扩展已有通用组件，不在业务页复制一份布局和样式。

### 5.2 UI Debug 页治理规则

- `UI Debug` 页的职责是陈列项目里真实存在的 UI 控件，并用文字标注“它用在什么页面、什么场景、来源是什么”。
- `UI Debug` 页本身不是新的设计系统试验场，不为它单独新增一套调试专用按钮、卡片、输入框、chip 或焦点态。
- 如果某个控件需要在 `UI Debug` 页展示，应优先直接复用真实组件；如确实需要包裹容器，只允许新增说明性布局，不允许改出新的控件视觉语义。
- 每次新增可复用 UI 组件、或把散落样式收口成通用组件后，都应同步把对应展示补进 `UI Debug` 页，避免控件资产只存在于业务页面里不可盘点。

## 6. 页面级规则

### 6.1 首页

- 必须保留 Hero、主 CTA、推荐泳道、媒体卡片的核心结构。
- 首页主视觉允许更强氛围化，但不能影响标题和 CTA 可读性。

### 6.2 播放器页

- 优先保证返回、播放/暂停、快进快退、清晰度、设置的操作稳定。
- 控制层可以做玻璃态，但不能为了视觉牺牲焦点清晰度。
- 播放进度、当前时间和下一条提示必须在远距离下清晰可读。
- 播放器路由进入后必须使用真正的沉浸式全屏壳，不显示左侧导航和顶部状态栏，不保留页面式外边距、圆角卡片或可见背景条。
- 进入播放器后默认直接开始播放视频，不显示封面式初始页；标题、进度和普通 OSD 默认隐藏，只在用户按遥控器方向键或确认键后显示。
- 普通 OSD 只在 `overlayMode = none` 时出现；播放中 3 秒无输入自动隐藏，返回键在 OSD 可见时优先隐藏 OSD，而不是直接退出播放器。
- 推荐视频区和分P/选集区必须作为底部固定横条 overlay 打开，背景全屏视频保持原位继续播放，不允许通过页面滚动把视频区顶走。
- 横条 overlay 的一屏容量固定为 6 项，左右在当前屏内移动，`下键` 切下一屏 6 项，`上键` 切上一屏 6 项，返回键优先收起横条并把焦点还给触发按钮。
- 推荐视频点击后直接切换到新视频并立即播放，不再经过详情页中转；分P/选集点击后直接切到对应 `cid` 的播放器实例。
- 播放器的返回链路必须统一遵循：`推荐横条 / 分P横条 / 设置抽屉 > 普通 OSD > 播放器页面`。

### 6.3 搜索/分类/登录/历史页

- 必须沿用同一套表面层级、焦点态和卡片风格。
- 不允许为了页面差异另起一套按钮或卡片样式。
- 搜索页里的输入入口、历史词条和操作按钮必须保留清晰间距，不能为了压缩布局把相邻按钮挤成难以分辨的一团。
- 搜索页不同 section 之间必须保留稳定的纵向留白；标题、副标题、按钮组与上一块内容的分隔，不允许只在桌面浏览器里成立而在 `webOS Simulator` 中塌陷。

## 7. 性能约束

- blur、阴影、渐变可以使用，但必须克制，优先保证 webOS 真机稳定。
- 长列表必须考虑渐进渲染或后续虚拟化，不允许整页无上限堆内容。
- 页面首屏优先清晰和稳定，不做高成本装饰性动画。

## 8. 开发流程要求

以后做任何 UI 相关功能时，必须按下面顺序执行：

1. 先对照 `ui_ref/` 和本规范确认页面是否延续同一视觉语言。
2. 先判断能否复用现有组件和 token。
3. 如果需要新组件，先抽组件再接页面。
4. 改交互时同步检查 section 划分、焦点流、抽屉接管与返回链路。
5. 如果改动涉及通用控件、统一样式来源或焦点语义，要同步检查 `src/features/debug/UiDebugPage.tsx` 是否需要补展示与用途标注。
6. 完成后至少执行 `npm run lint`、`npm run typecheck`、`npm run build`。
7. 如果改动了 App 图标或 `appinfo.json` 资源路径，还要执行 `npm run build:webos` 确认打包产物中的图标文件正确。
8. 如果新增或替换了应用内图标，必须同步更新 `src/app/iconRegistry.ts`，不能只在组件里临时写一个字符串。

## 9. 禁止事项

- 禁止直接把浏览器 hover 稿原样搬进 TV 页面。
- 禁止新增与现有规范冲突的按钮、卡片、导航样式。
- 禁止混用不同风格的系统图标、线性图标和占位字母，破坏整套界面的语义一致性。
- 禁止绕过 `icon registry` 直接在页面或业务组件里新增原始图标字符串，导致图标治理和子集打包失控。
- 禁止把临时生成图、未裁切主稿或非标准尺寸图片直接当作 webOS 应用图标入库。
- 禁止在业务组件里散落定义一套页面专属 token 来绕过全局样式体系。
- 禁止为了“先跑起来”牺牲主操作的可聚焦性。
- 禁止继续新增 `data-focus-row / data-focus-col` 一类坐标驱动焦点实现。
- 禁止用硬编码 DOM selector 在页面里做抽屉关闭回焦点，必须走统一的焦点接管与恢复机制。
- 禁止把 `UI Debug` 页当成新增控件样式的借口，先在 Debug 页发明样式、再反向扩散到业务页面。
