# PiliPlus 到 Bilibili WebOS 的功能迁移计划

## 1. 目标

以 `F:\CodeProjects\bilibili_tv_android\PiliPlus` 为功能参考源，逐步把一个功能完备的哔哩哔哩客户端能力，迁移到当前 `bilibili_webos` 项目中。  
迁移目标不是逐行复刻 Flutter 实现，而是在 WebOS TV 场景下，复用其成熟的功能结构、页面关系和业务闭环，做出一套适合遥控器和大屏观看的版本。

本计划默认建立在以下现状之上：

- WebOS 真机连接、打包、安装、启动链路已经跑通
- 当前 WebOS 仓库已有基础页面栈、遥控器按键映射、焦点移动和 TV UI 骨架
- 后续工作重点从“能部署”切换到“逐步补齐功能闭环”

## 2. 迁移原则

### 2.1 不机械抄页面，优先抄业务闭环

`PiliPlus` 的 Flutter 页面非常多，既包含主路径，也包含移动端和桌面端长尾功能。  
如果按页面名一页页照搬，最终很容易得到一个结构混乱、TV 操作割裂、维护成本很高的 WebOS 客户端。

因此迁移顺序必须遵循：

1. 先打通 TV 用户最高频主路径
2. 再补用户资产相关能力
3. 最后再评估社交和长尾能力

### 2.2 所有能力都要先过 TV 适配判断

参考项目的很多交互默认面向触摸、鼠标和桌面窗口。  
WebOS 端新增任何能力前，都要先回答三个问题：

- 遥控器能否稳定操作？
- 焦点路径是否清晰？
- 大屏远距观看是否仍然易读？

不满足这三个问题的能力，不进入当前阶段。

### 2.3 先搭基础设施，再铺页面

当前 WebOS 仓库的短板并不只是页面少，更关键的是业务基础设施还没起来。  
在首页、搜索、详情、历史这些页面真正可用之前，至少要逐步补齐：

- 更完整的页面/路由体系
- 统一请求层与接口模型
- 登录态与用户态
- 视频详情与播放数据流
- 收藏/历史/稍后再看等状态同步能力
- 严格遵守本项目的UI规范

## 4. 当前 WebOS 项目基线

当前 `bilibili_webos` 已有：

- `src/platform/webos.ts`：webOS 平台能力边界
- `src/platform/remote.ts`：遥控器按键到 TV 操作语义的映射
- `src/platform/focus.ts`：基础焦点移动
- `src/app/usePageStack.ts`：轻量页面栈
- `src/features/home/HomePage.tsx`：首页静态试点页
- `src/features/player/PlayerPage.tsx`：播放器静态试点页
- `scripts/webos-cli.mjs`：真机构建部署脚本

当前明显缺少：

- 真实业务路由与页面模型
- 接口请求层与 B 站数据模型
- 视频详情页
- 搜索页与搜索结果页
- 登录状态与用户信息状态
- 历史/收藏/稍后再看等资产页
- 播放器真实数据绑定
- 焦点分区与复杂页面的跨区导航策略

结论很明确：下一阶段不是“继续堆静态页面”，而是进入“业务功能迁移”阶段。

## 5. 总体迁移策略

建议按“五段式”推进，而不是一次性摊开所有功能。

### 阶段 A：补业务基础设施

目标：让 WebOS 项目具备承接真实功能的骨架。

建议完成：

- 将当前轻量页面栈扩展为可承载多页面的 TV 路由模型
- 建立 `src/services/` 或 `src/lib/` 层，统一放请求、鉴权、数据转换
- 抽离 `src/features/video/`，把“视频详情”和“播放器”拆成两个可协作页面
- 建立全局应用状态：
  - 登录状态
  - 当前用户简要信息
  - 播放上下文
  - 最近播放记录缓存
- 规划更清晰的焦点分区：
  - 全局导航区
  - 页面主内容区
  - 弹窗/面板区
  - 播放器控制区

这一阶段不追求功能多，重点是后面不返工。

### 阶段 B：打通首个核心观看闭环

目标：让用户从首页走到播放，形成最小可用链路。

建议顺序：

1. 首页推荐数据化
2. 视频卡片进入详情页
3. 详情页进入播放器
4. 播放器支持真实播放控制
5. 返回链路完整闭环

本阶段的完成标准：

- 用户能在 TV 上浏览首页推荐
- 用户能进入视频详情页
- 用户能开始播放、暂停、继续、返回
- 用户能看到相关推荐并继续跳转

这是整个项目最关键的里程碑。

### 阶段 C：补齐搜索与历史

目标：让用户不依赖首页，也能主动找到并继续看内容。

建议实现：

- 搜索入口页
- 热搜/默认搜索词
- 搜索历史
- 搜索结果页
- 观看历史页
- 从历史页回到详情/播放

TV 特化要求：

- 搜索框要做成“可聚焦入口”，而不是默认键盘输入框
- 首版优先接入 webOS 输入方案或简化输入面板
- 热搜、历史、结果列表都要支持稳定的 5-way 焦点流

完成这一阶段后，应用才不再只是“推荐流播放器”。

### 阶段 D：补齐登录与用户资产

目标：让用户开始使用自己的账号内容。

建议优先级：

1. 扫码登录
2. 个人中心首页
3. 稍后再看
4. 收藏夹列表与详情
5. 用户基础信息展示

这里建议明确取舍：

- 扫码登录优先，Cookie/账号密码输入后置
- 收藏与稍后再看优先做查看和播放，不先做复杂管理操作
- 个人中心先展示必要信息和入口，不急着复制移动端完整“我的”页

这一阶段完成后，WebOS 客户端才真正具备“常用账号观看工具”的形态。

### 阶段 E：内容扩展与平台化能力

目标：在主路径稳定后，再扩大内容覆盖面。

可依次评估：

- 热门页
- 番剧/PGC
- 分区/排行榜
- 直播入口
- 更多播放器设置
- 基础评论浏览

这一阶段开始，已经不是单纯“能看视频”，而是在补齐完整内容生态。

### 阶段 F：社交与长尾功能

目标：只在前面都稳定后再考虑。

后置能力包括：

- 动态
- 消息/私信
- 用户主页深链
- 关注/粉丝
- SponsorBlock
- DLNA
- 离线缓存
- 多账号
- WebDAV

这些能力不应该阻塞前几个主阶段。

## 6. 推荐的实际实施顺序

为了便于开工，建议直接按照下面的任务序列推进：

1. 扩展页面模型，把当前 `home/player` 两页栈升级成多页面 TV 路由
2. 新建请求层和基础 API 模块，先覆盖首页推荐、搜索、视频详情、播放地址
3. 把首页静态数据替换为真实接口数据
4. 新建视频详情页，承接选集、简介、相关推荐的最小版本
5. 把播放器从“静态演示页”升级成“真实播放页”
6. 新建搜索页与搜索结果页
7. 新建观看历史页
8. 接入扫码登录
9. 新建个人中心、稍后再看、收藏夹
10. 再做热门、番剧/PGC、直播等扩展内容页

如果我们按这个顺序做，几乎每一步都能产生可见成果，也便于真机回归。

## 7. 模块映射建议

不要直接把 Flutter 目录结构照搬到 React 项目，而建议做下面的对应关系：

### 参考项目到 WebOS 仓库的映射

- `lib/http/*`
  - 对应 `src/services/api/*`
  - 负责请求封装、接口参数、错误处理

- `lib/models/*` 与 `lib/models_new/*`
  - 对应 `src/features/*/types.ts` 或 `src/services/models/*`
  - 只迁移当前阶段真正用到的模型

- `lib/pages/home/*`
  - 对应 `src/features/home/*`

- `lib/pages/video/*`
  - 对应 `src/features/video-detail/*` 与 `src/features/player/*`

- `lib/pages/search/*`、`lib/pages/search_result/*`
  - 对应 `src/features/search/*`

- `lib/pages/history/*`
  - 对应 `src/features/history/*`

- `lib/pages/login/*`
  - 对应 `src/features/auth/*`

- `lib/pages/fav/*`、`lib/pages/later/*`、`lib/pages/member/*`
  - 对应 `src/features/library/*` 与 `src/features/profile/*`

- `lib/plugin/pl_player/*`
  - 不直接照搬
  - 只提炼成 WebOS 需要的播放器状态与控制层抽象

## 8. 各阶段的完成标准

### A 阶段完成标准

- 新页面可以稳定加入路由/页面栈
- 有统一请求层，不再在页面里直接乱写 `fetch`
- 有统一错误态、空态、加载态
- 复杂页面开始具备焦点分区规则

### B 阶段完成标准

- 首页内容来自真实接口
- 视频详情页可打开
- 播放器能真实播放
- 返回键链路稳定
- 真机可完成“首页 -> 详情 -> 播放 -> 返回”

### C 阶段完成标准

- 用户可主动搜索内容
- 用户可从历史中继续观看
- 搜索与历史页面在遥控器下可顺畅操作

### D 阶段完成标准

- 用户可扫码登录
- 用户可查看自己的收藏、稍后再看、个人信息
- 登录态切换不会破坏页面栈和焦点

### E/F 阶段完成标准

- 新增内容域不会破坏前面主闭环
- 页面性能与焦点链路在真机上仍稳定
- 功能是“扩展”，不是“返工”

## 9. 明确暂不做的内容

在前四个阶段完成前，不建议投入以下方向：

- 手机手势播放交互
- 桌面窗口、托盘、PIP 等桌面专属能力
- 离线缓存与本地文件管理
- 私信、动态发布、复杂富文本编辑
- 过多设置项和播放器高级开关

这些功能会显著拉高复杂度，但对 TV MVP 的价值不够高。

## 10. 风险与控制

### 风险 1：抄得太散，导致目录和状态失控

控制策略：

- 每次只迁移一个闭环，不同时铺太多页面
- 先抽公共服务层，再写页面

### 风险 2：移动端交互照搬到 TV 导致难用

控制策略：

- 每个页面先设计 5-way 焦点流，再写样式
- 一切 hover、手势、长按行为都要重新审视

### 风险 3：播放器链路过早做大

控制策略：

- 播放器先做最小必需子集
- 高级能力按真实需求逐项加

### 风险 4：接口模型过度照搬 Flutter 导致类型臃肿

控制策略：

- 只迁移当前阶段用到的字段
- 先做 ViewModel，再决定是否保留原始响应结构

## 12. 下一步建议

如果我们准备直接开工，建议只拿 `Task List` 里的前 3 项做第一轮，不要一次摊太大：

1. 扩展 `src/app` 页面模型与导航状态
2. 新建 `src/services/api` 基础层
3. 接入首页推荐接口，把首页变成真实数据页

这三步做完后，第二轮就能顺势进入“视频详情页 + 播放器真实链路”。
