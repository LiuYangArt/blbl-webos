# WebOS 播放器分 P 播完后自动续播下一条视频方案

## 1. 背景

当前 `webOS` 播放器已经支持“同一视频内的下一 P 自动续播”，但只覆盖当前 `bvid` 的 `parts` 列表。  
如果用户把一个多 P 视频全部播完，当前实现会直接停住，不会继续播放下一条视频。

这和电视端的真实使用习惯不完全一致。TV 场景里，用户往往希望内容连续播放，尤其是在：

- 正在长时间挂机看视频
- 用遥控器操作，不希望每次都手动回到推荐区重新点选
- 当前视频本身已经有相关推荐数据，天然具备“继续播”的候选来源

用户提出的直觉方案是“从推荐视频里找第一个继续播”。这个方向是可行的，但需要先看参考安卓项目有没有更完整的规则，避免我们把逻辑做得过于草率。

## 2. 调研结论

## 2.1 当前 webOS 仓库已有能力

当前仓库的播放器页 `src/features/player/PlayerPage.tsx` 已经具备下面几项能力：

- 进入播放器时会并行请求：
  - `fetchPlaySource(bvid, cid, qualityPreference)`
  - `fetchPlayInfo(bvid, cid)`
  - `fetchRelatedVideos(bvid)`
  - `fetchVideoDetail(bvid)`
- `fetchVideoDetail` 已经能拿到当前视频的 `parts`
- `fetchRelatedVideos` 已经能拿到相关推荐卡片列表
- `PlayerPage` 内已经维护了 `nextEpisodeRef`
- 视频 `ended` 事件里已经会读取 `nextEpisodeRef`，并自动跳到下一 P

也就是说，当前仓库并不是完全没有续播基础，而是已经完成了“同一 BV 内部续播”的第一层。

现状的关键断点是：

1. `nextEpisodeRef` 只会指向当前 `parts` 的下一项。
2. 当当前 `cid` 已经是最后一 P 时，`nextEpisodeRef` 会变成 `null`。
3. `ended` 事件在拿不到 `nextEpisodeRef` 后会直接返回，不再尝试任何下一条视频来源。

换句话说，webOS 端现在缺的不是“自动播放框架”，而是“最后一 P 播完后的下一条解析策略”。

## 2.2 参考项目 PiliPlus 的做法

本次最有参考价值的是 `PiliPlus`，不是 `JKVideo`。

### 2.2.1 播放结束时的总入口

`PiliPlus/lib/pages/video/view.dart` 在播放器完成播放时，会根据 `PlayRepeat` 决定后续行为：

- `singleCycle`：单个循环
- `listOrder`：顺序播放
- `listCycle`：列表循环
- `autoPlayRelated`：自动连播
- `pause`：播完暂停

它不会把“相关推荐连播”写死在播放器结束事件里，而是先经过一个“播放顺序模式”的总开关。

### 2.2.2 UGC 视频的 nextPlay 优先级

`PiliPlus/lib/pages/video/introduction/ugc/controller.dart` 的 `nextPlay()` 非常关键，它的优先级不是“先推荐”，而是：

1. 先播当前视频的下一 P
2. 如果当前处于 `播放全部` 上下文，则继续播列表里的下一条
3. 如果当前视频属于 `ugcSeason`/合集，则继续播合集里的下一条
4. 只有在上面都走不通时，且播放模式是 `autoPlayRelated`，并且开启了 `showRelatedVideo`，才会调用 `playRelated()`

`playRelated()` 的行为也很直接：

- 如果相关推荐控制器还没准备好，先异步拉数据
- 拉到数据后取相关推荐列表的第一项
- 第一项为空就停止连播
- 第一项存在就切过去继续播放

这说明安卓参考项目的核心思想是：

**显式连续上下文优先，相关推荐最后兜底。**

这比“最后一 P 直接跳推荐第一条”更合理，因为它尽量维持原来的内容语境。

### 2.2.3 PGC 的行为

`PiliPlus/lib/pages/video/introduction/pgc/controller.dart` 的 `nextPlay()` 更保守：

- 有下一集就播下一集
- 到头后如果是列表循环就回到第一集
- `autoPlayRelated` 在这里并不会额外跳相关推荐

这进一步说明 `PiliPlus` 并不是无脑追求“永不停播”，而是优先尊重内容本身的连续关系。

### 2.2.4 默认值

`PiliPlus/lib/utils/storage_pref.dart` 里 `PlayRepeat` 的默认值是 `pause`。  
也就是它虽然支持自动连播，但默认不会强制所有用户都开启“播完接相关推荐”。

这个点对我们很重要，因为它说明：

- “自动续播能力”是一个可配置播放模式
- “相关推荐第一条”只是其中一种终态策略，不是播放器唯一规则

## 2.3 参考项目 JKVideo 的情况

`JKVideo` 目前能看到：

- 视频详情页展示了合集/分集区和相关推荐区
- 有季/合集的横向列表展示
- 没有找到明确的“播放完成后自动切下一条视频”的落地逻辑

因此本轮方案应以 `PiliPlus` 为主要参考，而不是 `JKVideo`。

## 3. 结论：对 webOS 最合适的方案是什么

## 3.1 不建议直接照搬“播完就取相关推荐第一条”

原因不是这个方案不能用，而是它的优先级太低层：

- 它忽略了“当前内容是否还有更自然的下一条”
- 它把推荐流当成唯一连续来源
- 后续如果我们补上合集 / `ugcSeason` / 播放全部，上层逻辑会被迫重写

如果今天只是为了尽快出结果，这个方案当然能跑；但它不是参考项目里的最佳实践。

## 3.2 推荐采用“分层解析下一条”的方案

结合当前仓库现状和 `PiliPlus` 的思路，我建议 webOS 版本采用下面这个规则：

1. 当前视频还有下一 P：继续播下一 P
2. 当前视频已经播到最后一 P：
   - 如果未来拿得到显式连续上下文（如 `ugcSeason`、播放列表、合集），优先播该上下文里的下一条
   - 如果当前没有显式连续上下文，就退化到相关推荐第一条
3. 如果相关推荐为空，则停止播放，回到“播完暂停”的状态

这个方案本质上是：

**Android 的优先级结构 + webOS 当前阶段可落地的数据源收敛。**

也就是说，MVP 版本的真实落地形态会是：

**下一 P优先，最后一 P 结束后自动播放相关推荐第一条。**

但代码结构上不要把它写成“硬编码推荐第一条”，而要写成“解析下一播放目标”的统一入口，为后续补齐合集上下文留位置。

## 4. 为什么这是当前阶段最合适的折中

## 4.1 符合现有仓库能力

当前 `PlayerPage` 已经同时拿到了：

- `detail.parts`
- `related`

所以不需要新增接口，也不需要改播放器核心加载链路，就能完成最小闭环。

## 4.2 符合 TV 使用习惯

用户看完最后一 P 后直接停住，对电视端来说割裂感很强。  
自动接一条相关推荐，至少能让长观看路径持续下去。

## 4.3 与参考项目思路一致

虽然 webOS 当前还没有 `PiliPlus` 那么完整的“播放全部 / 合集 / autoPlayRelated”体系，但如果我们把逻辑抽象成统一解析器，方向就是一致的：

- 先找最自然的下一条
- 再找相关推荐第一条
- 找不到就停

## 4.4 不引入当前阶段不必要的复杂设置

`PiliPlus` 有完整的 `PlayRepeat` 配置体系，但当前 webOS 项目仍是 MVP，播放器能力还在持续补齐。  
如果本轮同时引入：

- 播放顺序设置
- 自动连播开关
- 列表循环
- 单个循环

会明显超出这次需求本身，也会增加 TV 焦点流、设置面板和状态持久化复杂度。

因此本轮建议：

- **先不新增播放顺序设置**
- **直接把“最后一 P 后接相关推荐第一条”作为当前产品行为**
- 但内部实现保留升级为 `PlayRepeat` 的扩展点

## 5. 建议的实现设计

## 5.1 新增统一的“下一播放目标解析”逻辑

不要把判断逻辑散在 `ended` 事件回调里，建议抽成一个纯函数或轻量工具，形态类似：

```ts
type AutoPlayNextTarget =
  | { kind: 'episode'; bvid: string; cid: number; title: string; part?: string }
  | { kind: 'related'; bvid: string; cid: number; title: string; part?: string }
  | null;
```

建议输入至少包含：

- 当前 `bvid`
- 当前 `cid`
- 当前标题 `title`
- 当前 `episodeEntries`
- 当前 `currentEpisodeIndex`
- 当前 `related`

后续如果补合集/播放列表，只需要继续往这个解析器的输入里加上下文，而不是回头改 `ended` 事件本体。

## 5.2 解析优先级

第一版建议解析顺序如下：

### Step 1：先找下一 P

- 若 `currentEpisodeIndex >= 0`
- 且 `episodeEntries[currentEpisodeIndex + 1]` 存在
- 返回该 P 作为下一目标

这部分当前仓库已经有 `nextEpisodeRef` 逻辑，后续可以：

- 保留 `nextEpisodeRef`
- 或统一收敛到新的解析器里

二选一即可，但不要同时保留两套判断源。

### Step 2：最后一 P 时找相关推荐第一条

只在下面条件同时满足时进入：

- 当前没有下一 P
- `related.length > 0`
- 第一条相关推荐具备可播放所需字段：`bvid`、`cid`、`title`

返回：

- `kind: 'related'`
- `bvid: related[0].bvid`
- `cid: related[0].cid`
- `title: related[0].title`

### Step 3：没有候选则停止

如果：

- 当前没有下一 P
- `related` 为空
- 或第一条相关推荐缺少必要字段

则返回 `null`，播放器保持当前播完暂停状态。

## 5.3 ended 事件中的行为

当前 `handleEnded` 里是：

1. 先 `setIsPlaying(false)`
2. 先显式把控制层显示出来
3. 再尝试读取 `nextEpisodeRef`
4. 有下一 P 就 `onOpenPlayer(nextEpisode)`
5. 否则直接结束

改造后建议变成：

1. 调用统一解析器拿到 `nextTarget`
2. 如果 `nextTarget === null`
   - 保持当前结束态
   - 控制层显示出来
   - 不跳转
3. 如果 `nextTarget !== null`
   - 清空续播进度缓存（与当前下一 P 逻辑一致）
   - 根据 `kind` 写一条简短提示文案
   - 调用 `onOpenPlayer(nextTarget)`

建议提示文案区分来源：

- 下一 P：`正在继续播放下一 P：xxx`
- 相关推荐：`当前视频已播完，正在为你播放下一条推荐视频`

## 5.4 不建议本轮做倒计时弹层

虽然很多 TV 产品会做“3 秒后自动播放下一条，可取消”的交互，但本轮不建议上来就做，理由：

- 当前需求是补齐自动续播闭环，不是重做片尾交互系统
- 倒计时会引入新的暂停/取消/返回键语义
- 还会影响 OSD、焦点和 `ended` 后状态管理

所以本轮建议：

- 先做立即续播
- 只给一条轻量提示文案
- 如果后续用户觉得需要“取消自动连播”，再单独设计片尾倒计时面板

## 5.5 不建议本轮新增播放顺序设置

虽然 `PiliPlus` 的 `PlayRepeat` 很完整，但当前 webOS MVP 不建议同步引入。  
本轮只做能力闭环，不做模式系统。

为了后续可演进，代码层面可以轻量预留一个内部枚举，例如：

```ts
type PlayerAutoNextMode = 'off' | 'related-after-last-part';
```

但如果当前实现并不会真正暴露第二种以上模式，这个内部枚举也可以先不加，避免过早抽象。

我的建议是：

- **文档层面保留未来可升级为 `PlayRepeat` 的方向**
- **代码实现层面只抽“下一目标解析器”，不要提前做完整模式系统**

## 6. 文件级改造建议

本轮如果进入实现，预计主要涉及以下位置：

### `src/features/player/PlayerPage.tsx`

职责：

- 从当前 `ended` 逻辑中抽离下一目标判断
- 将“下一 P”与“相关推荐第一条”统一接入自动续播
- 复用现有 `onOpenPlayer` 路由替换链路

建议改动点：

- 新增一个解析下一目标的 `useMemo` 或纯函数
- 在 `handleEnded` 中改为读取统一解析结果
- 区分来源文案

### 可选新增：`src/features/player/playerAutoNext.ts`

职责：

- 放纯解析函数，避免 `PlayerPage.tsx` 继续膨胀

如果这次想把逻辑写清楚、后续方便加合集上下文，我更推荐单独拆文件。

### `src/services/api/types.ts`

本轮大概率不需要改。  
当前 `VideoCardItem` 已经包含相关推荐续播所需的基础字段。

### `src/services/api/bilibili.ts`

本轮大概率也不需要改。  
因为 `fetchRelatedVideos` 已经存在，且播放器页已经在请求它。

## 7. 后续增强位

这部分不是本轮必须做，但文档里要先标明，避免将来推翻当前实现。

## 7.1 合集 / ugcSeason 上下文

`PiliPlus` 的 UGC 续播不是只靠相关推荐，它会优先沿着：

- 当前视频 `pages`
- `play all` 列表
- `ugcSeason`

去找下一条。

当前 webOS API 层还没有把 `ugcSeason` 暴露到播放器链路里，所以本轮无法完整复刻这层优先级。  
后续如果要补齐，应优先做：

1. `fetchVideoDetail` 映射 `ugcSeason`
2. 进入播放器时把“来源列表上下文”带进来
3. 把解析器的优先级扩展成：
   - next part
   - next in explicit playlist / season
   - first related

## 7.2 自动连播设置

如果后面用户反馈分歧明显，再考虑引入播放器设置项，例如：

- 播完暂停
- 播完接推荐

是否需要做成完整 `PlayRepeat`，要看届时是否已经具备：

- 播放列表
- 合集连续播放
- 列表循环

如果没有这些前置能力，直接照搬 `PiliPlus` 的五种播放顺序，收益并不高。

## 8. 风险与注意事项

## 8.1 第一条相关推荐不一定总是最优内容

这是这个方案天然存在的风险，但在当前仓库能力下是可接受的。  
因为我们本轮的目标不是“最聪明的推荐”，而是“最后一 P 播完后有可用下一条”。

## 8.2 相关推荐可能缺少可播放 cid

虽然当前 `VideoCardItem` 里有 `cid`，但还是建议在解析时做一次必要字段校验。  
不要假设接口永远完整。

## 8.3 ended 事件要避免重复触发导致多次跳转

实现时要注意：

- `ended` 有些场景可能重复触发
- 路由替换前后状态切换可能有短暂窗口

建议在自动跳转时增加一次轻量保护，例如：

- 通过 `ref` 标记当前这一轮 `ended` 是否已经处理过
- 当 `bvid/cid` 改变时再重置

当前文档先把这个作为实现注意事项，是否需要真正加锁要以联调结果为准。

## 8.4 提示文案不要依赖旧页面残留状态

如果后续发现 `setPlaybackNotice()` 在路由替换后不稳定，就不要继续强依赖旧页面本地状态。  
更稳妥的做法是：

- 简化为无需提示也能完成续播
- 或把“自动续播来源”作为新路由的瞬时参数带过去

这不是本轮必须项，但实现时要注意不要把提示文案做成阻塞逻辑。

## 9. 实施建议

如果后续进入开发，我建议按下面顺序做：

1. 先抽出“下一目标解析器”，只覆盖：
   - 下一 P
   - 第一条相关推荐
2. 用它替换 `handleEnded` 里对 `nextEpisodeRef` 的直接判断
3. 补一个纯逻辑测试文件，覆盖：
   - 有下一 P 时优先下一 P
   - 最后一 P 时退到第一条相关推荐
   - 没有相关推荐时返回 `null`
   - 相关推荐缺字段时返回 `null`
4. 再跑播放器页手动验证，重点确认：
   - 最后一 P 结束后是否能无缝跳转
   - 焦点和控制层是否没有异常闪动
   - 进度记录是否不会串到下一条视频

## 10. 最终建议

本轮建议采用下面这条结论，作为后续实现基线：

**先沿用当前“同一视频下一 P 自动续播”，在最后一 P 播完后，自动播放相关推荐中的第一条；但实现方式必须抽成统一的“下一播放目标解析”逻辑，为未来接入合集 / 播放列表 / ugcSeason 连续播放保留扩展点。**

这是当前仓库能力、电视端使用习惯、以及安卓参考项目思路三者之间最稳妥的交集。
