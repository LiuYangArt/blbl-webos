# LG C1 真机交互性能优化方案

## 1. 背景

当前项目在 `LG C1` 真机上的主要问题，不是“完全无法操作”，而是遥控器操作时存在明显的迟滞感。  
从现有反馈看，问题集中在三类高频场景：

- 视频列表内使用方向键移动焦点时，有轻微到中等程度的发黏和掉帧感
- 焦点从视频列表切到左侧菜单栏时，体感卡顿更明显
- 播放器页呼出控制条、设置抽屉、推荐条和分 P 条后，菜单导航存在拖泥带水的感觉

这些问题都直接影响电视端体验，因为 TV 端交互主要依赖 5-way 导航，用户对“按一下就立刻响应”的期待非常高。  
本方案的目标不是盲目做一轮泛化优化，而是围绕 `LG C1` 上最常见、最可感知的交互路径，优先降低主线程负担、减少昂贵样式效果、收敛焦点导航链路中的同步开销。

## 2. 本次范围

本次优化范围只覆盖“交互性能”与“TV 真机体感”相关内容，重点包括：

- 首页、热门、历史、搜索结果、关注、收藏等列表页的焦点移动性能
- 内容区与左侧导航栏之间的焦点切换性能
- 播放器页普通 OSD、控制条、设置抽屉、字幕抽屉、推荐条、分 P 条的操作性能
- 长列表逐页加载后的导航稳定性
- `LG C1` 真机与 webOS Simulator 的性能差异验证

本次不在范围内的内容：

- 网络接口耗时优化
- 播放清晰度策略或解码成功率优化
- 新功能开发
- 大规模视觉改版

## 3. 用户可感知的问题拆解

### 3.1 视频列表内移动焦点卡顿

用户在视频列表里连续按左右或上下键时，焦点移动不是完全丢失，而是呈现“跟手度不够”的感觉。  
这类问题通常有两种来源：

- 单次按键处理过重，导致每次导航都要等待一轮同步计算
- 列表渲染规模持续增长后，焦点系统和样式系统的每次开销被放大

从现有实现看，这两种情况同时存在。

### 3.2 焦点从内容区切到左侧菜单栏时卡顿

这个场景已经确认纳入优化范围，而且优先级较高。  
它不是简单的“向左移动一次焦点”，而是至少叠加了四类成本：

- 焦点引擎要重新计算目标元素
- 焦点高亮层要同步更新位置
- 左侧导航栏会从收起态切到展开态，伴随宽度、文字显隐、位移、阴影变化
- 内容区和导航栏都处在大面积半透明表面之上，合成层压力上升

因此这个问题比“列表内部左右移动”更容易在真机上被感知出来。

### 3.3 播放器菜单操作卡顿

播放器页的问题更像“持续热更新页面 + 菜单本身又不轻”，典型表现为：

- 播放中呼出控制条后，左右切换按钮时不够利索
- 打开设置抽屉、字幕抽屉后，操作 chip 和按钮时有发黏感
- 推荐条或分 P 条分页切换时，页面明显比首页更重

播放器页因为有 `timeupdate`、历史同步、进度刷新、overlay 切换、焦点接管与恢复，所以比普通页面更容易出现交互迟滞。

## 4. 现状证据与根因分析

### 4.1 焦点引擎在每次导航时存在较重的同步 DOM 查询与布局测量

当前焦点引擎会在导航时重新查找 section 内的全部可聚焦元素，并结合 `getBoundingClientRect()` 计算候选目标。  
此外，元素聚焦后还会做“舒适区滚动”和“首行判断”，这些逻辑本身也会触发布局读取。

重点位置：

- `src/platform/focus/engine.ts`
  - `getSectionElements`
  - `isFirstRowFocused`
  - `ensureFocusedElementComfort`
  - `findNextInSection`
  - `moveFocus`

影响：

- 列表项越多，单次方向键的计算成本越高
- 连按时更容易感知到响应滞后
- 内容区切到左侧导航时，这套计算会与侧栏动画开销叠加

### 4.2 FocusOverlay 当前是常驻全局观察模式

当前应用根节点始终挂载了 `FocusOverlay`，它会：

- 对 `document.body` 建立 `MutationObserver`
- 监听 `focusin`、`focusout`、`keydown`、`keyup`、`resize`、`scroll`
- 每次调度时读取活动焦点元素的 `getBoundingClientRect()` 与 `getComputedStyle()`
- 通过 React state 驱动额外渲染

重点位置：

- `src/components/FocusOverlay.tsx`
- `src/App.tsx`

影响：

- 每次方向键导航，都会带着一个旁路的调试高亮系统一起工作
- 焦点越频繁变化，额外开销越明显
- 该实现对浏览器开发期帮助较大，但对 TV 真机日常运行没有必要保持全时开启

### 4.3 左侧导航栏切换时伴随较重的宽度、位移和阴影变化

当前左侧导航在 `focus-within` 或展开态时，会发生以下变化：

- 宽度从收起态切到展开态
- 文本标签从透明 + 位移态切入
- 容器阴影增强
- 导航项自身还有焦点缩放和光效

重点位置：

- `src/components/SideNavRail.tsx`
- `src/styles/app.css`
  - `.side-nav-rail`
  - `.side-nav-rail:focus-within`
  - `.side-nav-rail--expanded`
  - `.side-nav-item[data-focus-active="true"]`

影响：

- 内容区切到左侧菜单时，不只是焦点在移动，整个左栏都在做结构变化
- 宽度动画会推动布局与合成变化，比单纯颜色变化更贵
- 老设备上，这个场景比列表内部移动更容易出现明显卡顿

### 4.4 页面和按钮样式使用了较多 blur / 阴影 / scale 效果

当前样式体系里存在多处对 TV 设备较贵的效果：

- `page-shell` 使用 `backdrop-filter: blur(16px)`
- 全局 token 中定义了 `--tv-blur: blur(22px)`
- `focus-button--glass` 使用 blur
- `player-settings-drawer` 使用 `backdrop-filter: blur(22px)` 和较重阴影
- 焦点态普遍带缩放和光晕

重点位置：

- `src/styles/app.css`

影响：

- 大面积半透明层与 blur 在 webOS 设备上通常是高成本项
- 这些效果与导航切换、抽屉打开、焦点移动重叠时，成本会叠加
- 浏览器端看起来“高级”的视觉语义，在真机上可能正是交互发涩的来源之一

### 4.5 播放器页存在持续性的状态更新与全局 Context 广播

播放器页在 `timeupdate` 中会持续执行：

- `setProgressView`
- `setWatchProgress`
- 历史心跳与上报相关逻辑

而 `watchProgress` 当前挂在全局 `AppStore` Context 中，一旦更新就会触发所有 `useAppStore()` 消费者重新参与渲染。

重点位置：

- `src/features/player/PlayerPage.tsx`
- `src/app/AppStore.tsx`
- `src/App.tsx`

影响：

- 用户打开播放器菜单时，页面本身仍在持续热更新
- 菜单导航卡顿不一定只来自菜单本身，也来自后台的进度写入和 Context 广播
- 这类问题在低端或老一些的 TV SoC 上会特别明显

### 4.6 PlayerControlBar 焦点变化会触发整条控制栏重渲染

控制条内部使用 `focusedActionId` 追踪当前获得焦点的按钮。  
每次 `onFocus` / `onBlur` 都会改变局部 state，从而重渲染整条控制栏。

重点位置：

- `src/components/PlayerControlBar.tsx`

影响：

- 单看不是灾难性问题
- 但在播放器页这个本来就比较热的页面里，会进一步放大焦点切换成本

### 4.7 长列表默认是“全量保留渲染”，会放大焦点扫描和样式成本

`VideoGridSection` 默认 `visibilityMode = 'loaded'`，而 `usePagedCollection` 会把更多页 merge 进同一个数组。  
热门、历史、搜索结果等页面随着加载页数增加，会保留越来越多卡片节点。

重点位置：

- `src/components/VideoGridSection.tsx`
- `src/features/shared/usePagedCollection.ts`
- `src/features/hot/HotPage.tsx`
- `src/features/history/HistoryPage.tsx`
- `src/features/search/SearchResultsPage.tsx`

影响：

- DOM 节点数量增加后，焦点系统的候选元素搜寻成本上升
- 卡片焦点态、阴影、图片等样式开销一起上升
- 用户会感知为“刚进页面还行，翻几屏后变卡”

### 4.8 图片加载与解码是次级问题，但仍值得纳入优化尾项

当前卡片封面和 Hero 图没有统一看到显式懒加载策略。  
这不是当前体感卡顿的第一根因，但在长列表页面会加重图片解码和内存压力。

重点位置：

- `src/components/MediaCard.tsx`
- `src/components/HeroBanner.tsx`
- `src/features/video-detail/VideoDetailPage.tsx`

### 4.9 遥控器输入节流参数可能放大“吞键感”

`remote.ts` 当前有 `KEY_GUARD_MS = 140`，并显式过滤 `event.repeat`。  
这不一定是渲染卡顿的根因，但可能让用户主观感受更像“按了没立刻走”。

重点位置：

- `src/platform/remote.ts`

建议：

- 先把渲染与样式问题处理完，再单独评估是否需要微调输入节流
- 不建议在没有证据的情况下，直接把它当作唯一根因

## 5. 优先级排序

### P0：交互链路基础治理

目标：先解决最影响体感的焦点导航链路。

包含内容：

- 让 `FocusOverlay` 退出生产常驻模式，只在明确的 debug 场景启用
- 为焦点引擎建立 section 级可聚焦元素缓存与失效机制
- 降低焦点移动过程中的重复 DOM 查询与重复测量
- 针对“内容区 -> 左侧菜单栏”建立单独的性能回归验证场景

预期收益：

- 首页、热门、历史、搜索结果的列表 navigate 体感明显改善
- 向左切到菜单栏时的顿挫感明显下降

### P1：左侧导航切换成本收敛

目标：让左侧导航从“结构变化明显的动画块”回到“低成本稳定锚点”。

包含内容：

- 减少或替换宽度动画
- 降低展开态阴影与文字显隐动画成本
- 保持导航层级与可读性，但避免在焦点切入时引发大范围布局变化

预期收益：

- 解决从内容区切到菜单栏的明显卡顿
- 左侧导航成为稳定锚点，而不是切换时的性能热点

### P2：播放器热路径隔离

目标：减少播放中对整页和全局状态的连带更新。

包含内容：

- 将播放进度刷新与全局 `AppStore` 热更新解耦
- 控制 `watchProgress` 写入频率与广播范围
- 让播放器菜单、控制条、抽屉尽量局部更新

预期收益：

- 播放器控制条、设置抽屉、字幕抽屉的操作更利索
- 播放中打开 overlay 时不再叠加全树重渲染

### P3：TV 样式降成本

目标：保留统一视觉语言，但把最贵的视觉语义换成 TV 更友好的实现。

包含内容：

- 收敛 `backdrop-filter`
- 降低重阴影范围
- 控制焦点缩放幅度与使用范围
- 把“高级感”优先放到纯色层级、渐变、边框和局部光效，而不是大面积 blur

预期收益：

- 所有页面的基础交互帧率更稳定
- 尤其改善左侧导航、抽屉、玻璃按钮等高频区域

### P4：长列表渐进渲染

目标：避免页面越用越重。

包含内容：

- 热门、历史、搜索结果等页面改为渐进显示或分段保留
- 避免无限保留已翻页内容造成 DOM 规模线性增长

预期收益：

- 翻更多屏后仍能维持较稳定的焦点导航
- 降低列表页后期退化

### P5：图片与细节补刀

目标：处理次级但累计可见的负担。

包含内容：

- 图片懒加载或延迟解码策略
- 进一步检查不必要的状态更新和日志写入路径
- 最后评估遥控器输入节流参数是否需要按 TV 调整

## 6. 建议实施顺序

建议按以下顺序推进，而不是并行散改：

1. `P0` 焦点链路治理
2. `P1` 左侧导航切换成本收敛
3. `P2` 播放器热路径隔离
4. `P3` TV 样式降成本
5. `P4` 长列表渐进渲染
6. `P5` 图片与次级补刀

原因：

- `P0 + P1` 最直接对应当前最明显的体感问题
- `P2` 直接对应播放器菜单操作卡顿
- `P3` 能为前几项提供通用收益，但不应在没解决热路径前只做“削特效”
- `P4` 适合在基础链路稳定后做，以免列表行为和焦点行为一起变动，增加排查难度

## 7. 建议拆分的实施任务

### Task A：聚焦焦点引擎热路径

- 为 section 元素建立缓存
- 只在必要时刷新候选元素列表
- 减少一次导航中的重复 `getBoundingClientRect()` 调用
- 为内容区与导航区切换建立专项回归用例

### Task B：FocusOverlay 改为真正的 Debug 能力

- 默认不在正常运行路径中启用
- 保留 debug 能力，但不让它参与真机日常交互链路

### Task C：左侧导航减重

- 调整展开方式，优先避免宽度驱动的大布局变化
- 降低进入导航时的阴影与位移动画成本
- 验证“收起态 -> 展开态 -> 焦点态”三者不要同时成为热点

### Task D：播放器状态隔离

- 将高频进度更新从全局热路径中拆开
- 控制菜单和控制条的重渲染范围
- 让 overlay 打开后仍保持稳定响应

### Task E：列表渲染治理

- 热门 / 搜索 / 历史页引入渐进展示
- 评估是否需要保留固定数量的已浏览项
- 验证翻页后的焦点定位和回焦点链路

### Task F：视觉层性能收口

- 全局梳理 blur、阴影、scale 的使用位置
- 为 webOS 真机设更保守的视觉基线

## 8. LG C1 上的验收标准

本次优化完成后，至少要在 `LG C1` 真机上满足以下体感标准：

- 首页、热门、历史、搜索结果中，连续方向键导航时，焦点移动明显比当前顺滑
- 焦点从视频列表切到左侧菜单栏时，不应再出现明显的停顿或“拖一下才过去”的感觉
- 左侧菜单栏展开时，用户主观上不应感到整个页面“顿住一下”
- 播放器呼出控制条后，左右切换按钮应保持稳定跟手
- 设置抽屉、字幕抽屉、推荐条、分 P 条打开后，按键响应不应明显劣于首页列表
- 长列表翻页 3 至 5 屏后，导航性能不应出现明显退化

## 9. 验证方式

### 9.1 手动验证场景

#### 场景 A：首页列表导航

1. 进入首页
2. 在推荐区连续按左右键移动 10 次以上
3. 再连续按下键、上键切换行或 section
4. 观察是否存在明显卡顿、吞键、焦点延迟

#### 场景 B：内容区切到左侧导航

1. 进入首页或热门页
2. 将焦点移动到内容卡片区中部
3. 连续按左键切回左侧菜单栏
4. 观察焦点切入时，是否出现明显停顿
5. 再在左侧导航中上下移动，观察展开态是否流畅

#### 场景 C：长列表退化

1. 进入热门页或搜索结果页
2. 触发加载更多，至少翻到第 3 至第 5 屏
3. 重复进行左右、上下和回左导航
4. 对比第一页与后几屏的体感差异

#### 场景 D：播放器菜单

1. 进入播放器并开始播放
2. 呼出普通 OSD
3. 左右切换控制条按钮
4. 打开设置抽屉和字幕抽屉
5. 打开推荐条和分 P 条
6. 观察各类 overlay 内的焦点移动和分页切换是否顺滑

### 9.2 构建与基础校验

实施阶段每轮改动后至少执行：

- `npm run lint`
- `npm run test`
- `npm run typecheck`
- `npm run build`

涉及真机部署与验证时，再执行：

- `npm run build:webos`
- `npm run webos:package`
- `npm run webos:update -- --device <deviceName>`
- 等待 8 秒
- `npm run webos:verify-install -- --device <deviceName>`
- `npm run webos:launch -- --device <deviceName>`

## 10. 风险与控制

### 风险 1：为了追求性能，误伤焦点稳定性

控制策略：

- 所有焦点系统优化都必须保留既有 section 语义
- 先做缓存与减少测量，不直接改业务焦点规则

### 风险 2：为了减轻样式开销，破坏现有 UI 统一性

控制策略：

- 视觉降成本必须优先复用现有 token 和组件
- 不允许页面各自临时“砍效果”

### 风险 3：播放器状态隔离改动范围较大

控制策略：

- 先只处理高频进度更新路径
- 不和播放链路、编码选择、字幕功能改动混在一起

## 11. 结论

“列表导航卡顿”“内容区切到左侧菜单栏卡顿”“播放器菜单卡顿”不是三个独立问题，而是同一类 TV 交互性能问题在不同场景下的表现。  
当前最值得优先处理的不是网络请求，也不是单一组件，而是以下三条主线：

- 焦点导航热路径减重
- 左侧导航切换成本收敛
- 播放器高频状态更新隔离

只要这三条主线处理到位，`LG C1` 上的体感通常就会有一轮比较明显的改善。  
本文件作为后续实施和 issue 拆分的基线文档。
