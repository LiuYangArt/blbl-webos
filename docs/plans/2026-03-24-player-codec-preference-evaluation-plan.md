# 播放器编码偏好策略评估与真机验证计划

## 1. 背景

当前播放器已经支持在设置面板中切换：

- `自动`
- `AVC`
- `HEVC`
- `AV1`

用户提出两个方向：

1. 既然 `HEVC / AV1` 在同等主观画质下通常能以更低码率传输，是否应该优先选择这些编码
2. 当用户手动选择某种编码后，是否应将其作为默认值持久化保存

本计划先做评估与验证，不直接修改默认策略。

## 2. 当前代码已确认的事实

## 2.1 用户手动选择 codec 已具备持久化能力

当前播放器页已实现：

1. 初始进入播放器时，从 `player_settings` 读取 `codecPreference`
2. 当 `codecPreference` 变化时，自动写回本地存储
3. 下次进入播放器时继续使用该偏好

涉及位置：

- `src/features/player/PlayerPage.tsx`
- `src/features/player/playerSettings.ts`

结论：

- “用户手动选择后持久化”这件事，基础能力已经存在
- 当前更需要确认的是：
  - 这个偏好是否真的影响了后续播放计划
  - 在电视真机上是否能稳定起播

## 2.2 当前 `auto` 明确偏向 `AVC`

当前自动优先级定义在：

- `src/features/player/playerCodec.ts`

现状是：

1. `AUTO_CODEC_PRIORITY = ['avc', 'hevc', 'av1']`
2. 评分上额外给：
   - `avc +3`
   - `hevc +2`
   - `av1 +1`

因此在“设备都声明支持、历史记忆也没有明显倾向”的情况下：

1. `auto` 会先尝试 `AVC`
2. 然后才是 `HEVC`
3. `AV1` 最后

这意味着：

- 即使 LG C1 理论支持 `HEVC / AV1`
- 当前自动策略也不会默认先冲这些编码

## 2.3 UI 选择 codec 会真实重建播放计划

当前设置面板点击 codec 后，链路是：

1. 用当前 `play + capability + codecMemory` 先预览一遍 `buildPlaybackAttempts(...)`
2. 根据预览结果更新 `codecPreference`
3. 更新 notice
4. `reloadNonce + 1`
5. 播放器重新加载

结论：

- 这不是只改了按钮态
- 当前 UI 的 codec 选择会真实驱动播放计划重算与播放器重载

## 3. 方案评估

## 3.1 方案 A：保持 `auto` 现状，重点支持“手动选择并记住”

结论：

- 可行
- 风险最低
- 最适合作为第一阶段落地方案

优点：

1. 不会影响当前默认起播稳定性
2. 用户知道自己电视更适合 `HEVC` 时，可以手动固定
3. 与电视端“偏好设置”习惯一致

风险：

1. 用户未必理解“这是当前视频生效”还是“后续也会默认使用”
2. 需要把设置文案讲清楚

建议：

1. 保持当前持久化逻辑
2. 后续只在 UI 上明确提示“编码偏好会保存为默认值”

## 3.2 方案 B：把 `HEVC` 提到 `AVC` 前面，作为特定设备的自动优先策略

结论：

- 可行
- 比方案 A 更激进，但仍属于可控范围

适用前提：

1. 目标设备已确认支持 `HEVC`
2. 真机实测表明 `HEVC` 起播稳定
3. 起播失败时能快速回退到 `AVC`

建议形态：

- 不要全平台直接改
- 先只对 `webos-2021 / LG C1` 这类设备做定向优先

推荐顺序：

- `HEVC > AVC > AV1`

而不是：

- `AV1 > HEVC > AVC`

原因：

1. `HEVC` 对当前电视端链路更容易先验证
2. `AV1` 虽然压缩率更好，但 webOS Web Runtime + MSE 的长期稳定性还需要更多样本

## 3.3 方案 C：全局默认优先 `AV1 / HEVC`

结论：

- 技术上能做
- 当前阶段不建议直接这么改

原因：

1. 设备“声明支持”不等于整个 webOS 播放链路都稳定
2. `AV1` 可能比 `HEVC` 更容易踩到 Web Runtime / MSE / 分片兼容边角
3. 当前项目刚把 `1080P DASH 优先` 修稳，不适合马上再引入全局更激进的默认 codec 策略

## 4. 推荐结论

本轮推荐分两层：

### 第一层：现在就可以采用

1. 保持 `auto` 现状不变
2. 明确把“用户手动选择 codec 后，该偏好会持久化”作为正式能力
3. 在电视真机上优先验证：
   - `HEVC` 是否稳定起播
   - `AV1` 是否真的能播、且体验可接受

### 第二层：等真机数据回来再决定

如果 LG C1 真机验证表明：

1. `HEVC` 稳定
2. 起播体验不差于 `AVC`
3. 回退链路也正常

则下一阶段可以考虑：

- 仅对 `webos-2021 / LG C1` 把 `auto` 调整为 `HEVC > AVC > AV1`

`AV1` 是否进入自动优先的更前位置，本轮不建议直接承诺。

## 5. 真机验证目标

本轮真机主要验证三件事：

1. 当前电视是否真的能起播 `HEVC`
2. 当前电视是否真的能起播 `AV1`
3. 当前 UI 中选择不同 codec 后，播放器是否真的切到了对应编码计划，而不是只改了显示状态

## 6. 真机验证步骤

### 6.1 部署准备

按仓库标准流程：

1. `npm run webos:list -- --device lgtv`
2. `npm run webos:deploy -- --device lgtv`

### 6.2 推荐验证视频

优先选同时可能包含多 codec 分轨的公开样本视频，例如：

- `BV1Sf4y1q7H9`
- `BV1wqoGYxEox`

如果某条视频没有 `AV1` 或 `HEVC`，不能据此直接判定电视不支持，需要换样本继续验证。

### 6.3 电视端操作

进入播放器后，按下面顺序测：

1. 先用 `自动`
2. 打开设置面板，切到 `HEVC`
3. 观察播放器是否重载，并确认：
   - 当前执行 codec
   - 当前执行画质
   - 是否正常起播
4. 再切到 `AV1`
5. 同样确认：
   - 是否真的触发重载
   - 是否有 `AV1` 分轨可用
   - 如果没有，是否明确回退到 `auto`

### 6.4 验证判定标准

要认定“UI 选择 codec 确实生效”，至少需要满足：

1. 选择 codec 后播放器发生重载
2. 诊断面板里的“当前执行 codec”跟着变化
3. telemetry 中首个 `attempt-switch` 的 `attemptCodec` 与选择一致
4. 若目标 codec 不可用，必须出现清晰可解释的回退提示，而不是静默失败

## 7. 当前阶段不做的事

本轮先不做：

1. 改 `auto` 的默认 codec 顺序
2. 针对不同内容类型做 codec 策略画像
3. 把 `AV1` 设为 LG C1 的自动首选

这些都要建立在真机验证数据之上。

## 8. 后续落地顺序建议

如果本轮电视验证结果良好，建议下一步顺序是：

1. 明确 UI 文案：提示 codec 偏好会被保存
2. 只对 `LG C1 / webos-2021` 评估是否将 `auto` 调整为 `HEVC > AVC > AV1`
3. 待更多样本稳定后，再决定是否让 `AV1` 进入更高优先级
