# 播放器请求 1080P 实际仅 720P 专项改造计划

## 1. 背景

Issue `#9` 指出：当前播放器在用户明确选择 `1080P` 后，最终仍可能稳定在 `720P`，状态提示类似：

- `720P 准高清（已请求 1080P 高清）`

Issue 同时给出了几个重要事实：

- 请求画质：`1080P`
- 实际返回/执行：`720P`
- 接口限制码：`0`

这意味着问题不像“会员权限被明确拒绝”那样简单，更像是：

1. 高画质取流参数组合还不够理想
2. 播放器在 DASH 与兼容流并存时，把低画质候选过早当成“当前实际画质”
3. UI 和调试信息没有把“请求画质 / 接口返回画质 / 当前执行画质 / 回退原因”区分清楚

用户还补充说明：**本地模拟器上也能复现相同问题**。  
因此，本计划默认把它视为“仓库内部决策逻辑问题优先”，而不是只在真机上才会出现的设备兼容问题。

## 2. 结论先行

先说结论：

- 当前问题的第一优先级，不是先去猜会员权限或 LG 设备限制，而是先修正当前仓库里对播放源质量的归一和选择逻辑。
- 参考项目 `JKVideo` 的价值，不在于逐行照搬，而在于它提供了三个明确方向：
  - 初次请求就使用更激进的高画质参数
  - 画质列表与当前实际档位分开管理
  - DASH 轨道按目标 `qn` 选择，兼容流只作为独立回退层
- 本轮实施必须把问题拆成四层，而不是试图在一个函数里一次性“蒙对”：
  - 取流请求策略
  - 播放源归一策略
  - 播放器尝试选择策略
  - UI / 日志可观测性

## 3. 当前仓库已确认的事实

### 3.1 取流层事实

当前 `src/services/api/bilibili.ts` 中：

- `requestDashPlaySource()` 只轮询 `fnval=[4048, 16]`
- `fetchCompatibleSources()` 会按高到低请求兼容流，并把接口返回的 `actualQuality` 记录为兼容流档位
- `fetchPlaySource()` 在 DASH 成功时，`currentQuality` 当前取值优先来自 `compatibleSources[0]?.quality`

这里最危险的一点是：

- 只要兼容流列表第一个返回的是 `720P`
- 即使 DASH 响应里已经拿到了 `1080P`
- `currentQuality` 也会被覆盖成 `720P`

这会把后续所有依赖 `currentQuality` 的逻辑一起带偏。

### 3.2 播放器尝试层事实

当前 `src/features/player/playerCodec.ts` 中：

- `getResolvedDashStreams()` 会优先按 `playSource.currentQuality` 过滤 DASH 视频轨
- `buildCompatibleAttempts()` 也会按 `playSource.currentQuality` 选择兼容流来源

因此，一旦 `currentQuality` 被兼容流污染：

- 如果 DASH 中同时存在 `1080P` 和 `720P`，播放器会真的优先拿 `720P` 分轨
- 即使最终仍能回退到更高档位，当前执行策略和 UI 语义也会不一致

这说明问题不是“只显示错了”，而是**可能真的选错了轨**。

### 3.3 UI 可观测性事实

当前 `src/features/player/PlayerPage.tsx` 已经展示：

- `请求画质`
- `实际返回画质`
- `接口限制码`
- `当前执行策略`

但这些信息仍然依赖 `play.currentQuality` 和 `play.qualityLabel`。  
因此只要上游归一逻辑不准，整个诊断面板也会一起误报。

### 3.4 测试覆盖事实

当前测试已经覆盖：

- DASH 不可用时回退 durl
- DASH 与兼容流返回同一档位时的正常情况

当前测试未覆盖：

- DASH 返回 `1080P`
- 兼容流只返回 `720P`
- 最终不应让兼容流覆盖 DASH 当前档位

这正是本次回归最关键的测试空洞。

## 4. `JKVideo` 参考实现可借鉴的策略

### 4.1 高能力请求参数

`JKVideo/services/bilibili.ts` 中，安卓端 `getPlayUrl()` 使用：

- `fnval = 16 | 64 | 128 | 256 | 1024`
- `fourk = 1`

这说明参考项目在第一跳请求时就尽量争取更多高画质 / 高音质 / 杜比相关能力，而不是只靠保守参数探测。

### 4.2 初次就用高 qn 拉清晰度

`JKVideo/hooks/useVideoDetail.ts` 中：

- 首次拉取播放数据直接请求高档位 `qn=126`
- 登录态变化后会刷新清晰度列表

这背后的产品含义是：

- 清晰度列表应尽量一次拿全
- 用户登录前后能拿到的画质可能不同，列表需要刷新

### 4.3 DASH 按目标 qn 选轨

`JKVideo/utils/dash.ts` 中：

- 视频轨优先选 `id === qn`
- 音频轨优先选杜比，否则选带宽最高的普通音轨

可提炼出的核心原则是：

- DASH 实际选轨要由“目标档位”驱动
- 兼容流只是兜底，不应反向污染 DASH 当前档位

## 5. 改造目标

## 5.1 必达目标

1. 当接口确实返回 `1080P` DASH 分轨时，播放器优先以 `1080P` 作为当前执行档位。
2. 当接口没有返回目标档位时，明确区分：
   - 请求画质
   - DASH 实际返回画质
   - 兼容流可回退画质
   - 当前执行策略
3. 若最终只能回退到 `720P`，必须能解释原因，而不是只展示模糊状态。
4. 新增测试覆盖“DASH 高档位 + 兼容流低档位并存”的场景。

## 5.2 非目标

本轮不直接承诺：

- HDR / 杜比视界 / HDR Vivid 真机稳定播放
- 全量重写播放器架构
- 重新设计整套播放器 UI

这些能力可以在本轮把数据入口和诊断信息补齐，但不作为第一阶段验收前提。

## 6. 实施方案

## 6.1 Phase A: 修正播放源语义模型

目标：

- 把“请求档位”“DASH 实际档位”“兼容流档位”“当前执行档位”彻底拆开

建议改造：

1. 扩展 `PlaySource` 类型，明确补充质量语义字段。
2. 让 `fetchPlaySource()` 在 DASH 可用时：
   - 优先根据 DASH 响应推导 `dashReturnedQuality`
   - 不再让 `compatibleSources[0]` 覆盖 DASH 当前档位
3. 为兼容流单独保留：
   - `compatibleSources`
   - `compatibleBestQuality`
   - `compatibleRequestedQuality` 或等效调试字段
4. 增加“质量决策摘要”字段，供日志和 UI 复用，避免页面层重复猜测。

建议涉及文件：

- `src/services/api/types.ts`
- `src/services/api/bilibili.ts`

验收要点：

- 当 DASH 返回 `1080P`、兼容流返回 `720P` 时，`PlaySource` 里的 DASH 当前档位仍为 `1080P`

## 6.2 Phase B: 提升高画质请求策略

目标：

- 提高拿到高画质 DASH 元信息的成功率

建议改造：

1. 在 `requestDashPlaySource()` 中引入“高能力优先、现有策略保底”的 `fnval` 候选序列。
2. 序列中至少纳入 `JKVideo` 已验证使用的高能力组合。
3. 保留当前仓库已有的 `4048` 和 `16` 作为回退候选，避免直接砍掉现有兼容路径。
4. 视接口行为补充请求元信息记录：
   - `qn`
   - `fnval`
   - `fourk`
   - `platform`
   - `high_quality`

设计建议：

- 本轮不要只保留一个“理论上最强”的 `fnval`
- 应采用多候选尝试，并把最终命中的参数写进调试数据，方便后续真机复盘

建议涉及文件：

- `src/services/api/bilibili.ts`
- `src/services/api/types.ts`

验收要点：

- 同一视频在权限允许时，应优先拿到包含目标档位的 DASH 元信息，而不是一开始就落到低档兼容流语义

## 6.3 Phase C: 重构 DASH / 兼容流的选择优先级

目标：

- 让播放器尝试列表真正体现“DASH 优先，兼容流兜底”

建议改造：

1. `getResolvedDashStreams()` 优先使用“DASH 实际返回档位”而不是混合后的 `currentQuality`。
2. 兼容流选择逻辑只在：
   - DASH 不可用
   - DASH 起播失败
   - 设备明确不支持当前 DASH 组合
   时才介入。
3. 兼容流参与回退时，也不要覆盖 DASH 的“接口实际返回画质”字段。
4. 播放尝试对象里显式记录：
   - `requestedQuality`
   - `resolvedQuality`
   - `mode`
   - `fallbackReason`

建议涉及文件：

- `src/features/player/playerCodec.ts`
- `src/features/player/playerCodec.test.ts`

验收要点：

- 当 DASH 已有 `1080P` 时，尝试列表首项应是 `1080P DASH`
- 兼容流只能作为后续兜底尝试，而不是首个质量来源

## 6.4 Phase D: 修正 UI 与日志的诊断语义

目标：

- 让播放器显示的信息与内部真实决策一致

建议改造：

1. 设置面板中把以下字段拆开显示：
   - 请求画质
   - DASH 实际返回画质
   - 兼容流最高可回退画质
   - 当前执行画质
   - 当前执行模式
   - 回退原因
2. 顶部状态文案不再简单使用 `play.currentQuality`，而是使用“当前执行尝试”的质量。
3. `describeQualityAvailability()` 改为优先引用结构化原因，而不是基于旧字段拼句子。
4. 调试事件中新增：
   - 请求参数组合
   - DASH 返回档位列表
   - 兼容流返回档位列表
   - 最终选中的 attempt

建议涉及文件：

- `src/features/player/PlayerPage.tsx`
- `src/features/player/playerDebug.ts`

验收要点：

- 当 UI 提示“已请求 1080P，但当前执行 720P”时，旁边必须能说明是：
  - 接口没返回 1080P
  - 会员/权限限制
  - 设备能力不支持当前 DASH 组合
  - 已主动回退兼容流

## 6.5 Phase E: 补强自动化测试

目标：

- 防止后续再次出现“高画质被兼容流覆盖”的回归

必须补的测试：

1. `fetchPlaySource()`：
   - DASH 返回 `1080P`
   - 兼容流返回 `720P`
   - 结果应保留 DASH 当前档位为 `1080P`
2. `fetchPlaySource()`：
   - 请求 `1080P`
   - 接口只返回 `720P`
   - 应给出明确的返回档位和限制原因语义
3. `buildPlaybackAttempts()`：
   - 当 DASH 已有目标档位时，首选 attempt 不应退到兼容流或低档位
4. `PlayerPage` 相关逻辑：
   - 若拆出了新的诊断字段，至少补纯函数或数据映射测试，避免 UI 文案再混淆

建议涉及文件：

- `src/services/api/bilibili.test.ts`
- `src/features/player/playerCodec.test.ts`

## 7. 实施顺序建议

建议严格按下面顺序推进：

1. 先修 `PlaySource` 数据模型和 `fetchPlaySource()` 归一逻辑
2. 再修 `playerCodec.ts` 的选轨和尝试优先级
3. 然后补 `PlayerPage.tsx` 的诊断展示与文案
4. 最后统一补测试和调试日志

原因：

- 如果先改 UI，不解决上游字段污染，只会把错误信息换一种形式显示出来
- 如果先改选轨，不修数据模型，后面仍然会被模糊语义反噬

## 8. 验证计划

## 8.1 自动化验证

完成改动后至少运行：

- `npm run lint`
- `npm run test`
- `npm run typecheck`
- `npm run build`

专项关注：

- 新增的高画质请求与回退测试必须全部通过

## 8.2 手动验证

### 本地模拟器

目标：

- 验证“当前质量被兼容流覆盖”的逻辑问题是否已消失

步骤：

1. 打开任意已知支持 `1080P` 的视频
2. 在播放器设置中选择 `1080P`
3. 观察顶部状态文案和设置面板中的诊断字段
4. 记录是否仍出现：
   - 请求 `1080P`
   - 实际却显示或执行 `720P`

预期：

- 若 DASH 已返回 `1080P`，模拟器下的当前执行档位应显示 `1080P`
- 不应再因为兼容流列表存在 `720P` 就把当前执行档位降成 `720P`

### 真机 webOS

目标：

- 验证真实设备上的高画质链路是否稳定，同时确认兼容流回退仍可用

步骤：

1. 按仓库既定流程部署到真机
2. 打开同一视频并选择 `1080P`
3. 检查：
   - 请求参数是否为高能力组合
   - DASH 是否返回目标档位
   - 首次 attempt 是否为 `1080P DASH`
   - 若 DASH 失败，是否能自动回退兼容流继续播放

预期：

- 权限允许时优先命中 `1080P`
- 回退发生时，UI 和日志能解释原因

## 9. 风险与注意事项

1. `fnval` 候选组合变得更激进后，某些视频或账号状态下可能返回更复杂的轨道集合，必须保证现有低风险回退路径不被破坏。
2. 如果把质量语义拆得过于复杂，但页面层继续自己推导，会再次形成“双重解释”；因此需要尽量把诊断字段收口在服务层。
3. 模拟器可复现只说明逻辑 bug 存在，不代表真机就一定全部修好；真机仍需验证实际播放稳定性与 DASH 兼容性。
