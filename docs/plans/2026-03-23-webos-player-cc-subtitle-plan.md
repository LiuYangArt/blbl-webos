# WebOS 播放器 CC 字幕功能计划

## 1. 背景

当前 `bilibili_webos` 已经具备基础播放、画质/编码设置抽屉、底部 OSD 控制条和本地播放器设置持久化能力，但还没有把哔哩哔哩视频里的 CC 字幕链路接起来。  
这会直接影响电视场景下的可用性，尤其是：

- 远距离观看时，缺少字幕会明显降低可理解性。
- 部分内容本身依赖字幕辅助理解。
- 用户已经明确要求：
  - 默认开启 CC 显示
  - 用户关闭后持久化开关状态
  - 在视频下方 OSD 按钮区增加字幕 `CC` 设置按钮

本计划先基于安卓参考项目 `PiliPlus` 的真实实现，结合当前 WebOS 播放器结构，定义一个适合 TV 场景的 MVP 方案，再据此创建实施 Issue。

## 2. 安卓参考项目结论

### 2.1 数据来源不是只靠 `playurl`

`PiliPlus` 并不是只靠 `/x/player/playurl` 做字幕。  
它会额外请求：

- `/x/player/wbi/v2`

这条接口由 `VideoHttp.playInfo()` 承担，用来获取字幕与其他播放附加信息。  
字幕内容本身仍然需要通过字幕 URL 拉取 JSON，再转换成 `WEBVTT` 供播放器加载。

对 WebOS 端的含义非常明确：

- 不能继续假设现有 `fetchPlaySource()` 足够承接字幕。
- 需要单独补一条 `fetchPlayInfo()` 或同等语义的 API 层。

### 2.2 安卓项目里的字幕能力分了三层

安卓参考项目的字幕功能可以分成三层：

1. 全局默认偏好
   - `off`
   - `on`
   - `withoutAi`
   - `auto`
2. 当前视频的字幕轨切换
   - 关闭字幕
   - 切换语言/轨道
3. 字幕样式设置
   - 字号
   - 全屏字号
   - 字重
   - 描边
   - 边距
   - 背景不透明度

此外它还支持：

- 加载外部字幕文件
- 导出字幕 JSON

这些能力对安卓/桌面端是合理的，但不应直接全部塞进当前 WebOS MVP。

### 2.3 安卓底部控制区确实有独立的 CC 入口

参考项目播放器底部控制区已经放了独立的 CC 按钮，点击后直接弹出轨道列表。  
这点和用户要求完全一致，说明我们不需要重新发明字幕入口形态。

## 3. 当前 WebOS 仓库现状

### 3.1 已有可复用基础

当前 WebOS 仓库已经有以下基础能力：

- [PlayerControlBar](/F:/CodeProjects/bilibili_webos/src/components/PlayerControlBar.tsx)
  - 已有稳定的 OSD 按钮链
- [PlayerPage](/F:/CodeProjects/bilibili_webos/src/features/player/PlayerPage.tsx)
  - 已有 overlay 模式和焦点捕获/恢复逻辑
- [playerSettings.ts](/F:/CodeProjects/bilibili_webos/src/features/player/playerSettings.ts)
  - 已有本地持久化模式
- [local.ts](/F:/CodeProjects/bilibili_webos/src/services/storage/local.ts)
  - 已封装 `localStorage` 读写

这意味着 CC 不需要从零起一套新基础设施，完全可以沿用现有播放器结构逐步接入。

### 3.2 当前缺口

当前播放器还缺少这些关键能力：

- 没有字幕轨数据模型
- 没有字幕信息 API 请求
- 没有字幕 JSON -> VTT 转换链路
- 没有播放器文本轨加载逻辑
- 没有 CC 状态持久化
- 没有 OSD 级字幕入口
- 没有字幕焦点链与返回链路

## 4. 设计目标

本次规划的目标是做出一个“TV 能用、可稳定验证、不过度工程化”的 CC 字幕 MVP：

1. 如果视频有字幕轨，默认开启字幕显示。
2. 用户手动关闭后，后续视频沿用关闭状态。
3. OSD 底部按钮区提供清晰的 `CC` 入口。
4. 用户能通过遥控器切换字幕语言/轨道。
5. 用户至少能调整少量最影响电视观看体验的字幕样式。
6. 返回键与焦点行为符合现有播放器 overlay 规则。

## 5. 范围划分

### 5.1 首期必须实现

- 字幕信息请求与解析
- 默认开启字幕
- 关闭状态持久化
- 底部 OSD `CC` 按钮
- 字幕开关
- 字幕语言/轨道切换
- 无字幕视频的禁用/空态反馈
- 基础字幕样式项：
  - 字号
  - 底部边距
  - 背景不透明度

### 5.2 首期明确不做

- 外部字幕导入
- 字幕导出
- 全量照搬安卓所有样式滑杆
- AI 字幕四档智能偏好
- 按静音状态自动切字幕
- 复杂字幕缓存管理或跨视频语言偏好记忆

## 6. 方案设计

### 6.1 数据层

新增一条专门的字幕信息请求：

- 目标接口：`/x/player/wbi/v2`
- 请求参数至少包括：
  - `bvid`
  - `cid`
  - 若未来支持 PGC，再补 `season_id / ep_id`

新增数据模型建议：

- `PlaySubtitleTrack`
  - `id`
  - `lang`
  - `langDoc`
  - `subtitleUrl`
  - `isAi`
- `PlayInfo`
  - `subtitles`

建议不要把字幕轨硬塞进现有 `PlaySource` 的语义里。  
更清晰的做法是：

- `fetchPlaySource()` 继续负责播放地址、画质、音视频轨
- `fetchPlayInfo()` 负责字幕与其他附加信息

### 6.2 字幕内容转换

参考安卓项目，B 站字幕内容需要走：

1. 请求字幕 JSON
2. 转换成 `WEBVTT`
3. 注入 HTML5 `<track>` 或播放器文本轨

WebOS 端建议封装一个独立工具，例如：

- `playerSubtitle.ts`

职责包括：

- 拉取字幕 JSON
- 转成 `WEBVTT`
- 缓存当前视频已转换的字幕内容
- 管理“关闭 / 已选中某一条字幕轨”的切换状态

### 6.3 默认开启与持久化

首期遵循用户要求，规则简化为：

- 默认值：`subtitleEnabled = true`
- 如果用户手动关闭字幕：
  - 写入播放器设置存储
  - 后续视频继续默认关闭
- 如果用户重新打开：
  - 也应覆盖写回持久化

首期强制持久化的只包括：

- `subtitleEnabled`
- 基础字幕样式设置

首期不强制全局持久化的包括：

- 语言/轨道选择

原因：

- 不同视频可用字幕语言差异很大
- 先把总开关做好，更符合 MVP

### 6.4 默认轨道选择规则

当 `subtitleEnabled = true` 且视频存在字幕轨时，首期建议按以下顺序选择默认轨道：

1. 优先中文人工字幕
2. 其次非 AI 字幕
3. 最后回退到第一条可用字幕

这样既保留了对安卓“跳过 AI”思路的借鉴，也不会把首期做成完整的四档策略系统。

### 6.5 OSD 入口与交互

在底部 [PlayerControlBar](/F:/CodeProjects/bilibili_webos/src/components/PlayerControlBar.tsx) 中新增独立 `CC` 按钮。

推荐位置：

- `分P / 选集` 与 `设置` 之间

原因：

- 这两个功能都属于播放时的附加控制
- 用户能比较直觉地在设置旁边找到字幕

按钮行为：

- 有字幕时：打开字幕浮层
- 无字幕时：
  - 按钮禁用，或
  - 可点后提示“当前视频暂无字幕”

首期更推荐“可聚焦但有禁用视觉态 + 点击提示”，避免 OSD 焦点链突然断裂。

### 6.6 字幕浮层

不建议把字幕功能继续塞进现有“画质与编码”抽屉。  
建议新增独立 overlay，例如：

- `overlayMode = 'subtitles'`

浮层内容分三组：

1. 显示
   - 开 / 关
2. 语言
   - 关闭字幕
   - 字幕轨列表
3. 样式
   - 字号
   - 底部边距
   - 背景不透明度

这样做的好处：

- 焦点结构清晰
- 用户心智简单
- 不会把“编码、画质、线路、字幕”混成一个大杂烩抽屉

### 6.7 焦点与返回链路

字幕浮层需要遵循现有播放器 overlay 规则：

- 打开时捕获到底部 `CC` 按钮的来源焦点
- 浮层默认聚焦：
  - 若字幕已关闭，优先聚焦“开启字幕”
  - 若字幕已开启，优先聚焦当前选中的字幕轨
- 关闭时恢复到 `CC` 按钮
- 返回键顺序保持：
  - 字幕浮层 > 其他面板 > 页面返回

### 6.8 样式范围控制

安卓参考项目的样式项很多，但 TV 首期建议只保留最影响观看体验且适合遥控器调整的 3 项：

- 字号
- 底部边距
- 背景不透明度

后续如果真机验证证明还需要，再扩：

- 描边
- 字重
- 左右边距
- 全屏单独字号

## 7. 实施分期

### Phase A: 字幕数据链路

目标：

- 让播放器能拿到字幕轨和字幕内容

任务：

- 在 API 层新增 `fetchPlayInfo()`
- 定义字幕轨类型
- 实现 B 站字幕 JSON -> WEBVTT 转换
- 为播放器接入文本轨加载与切换

交付标准：

- 给定含字幕的视频，代码层能拿到字幕列表
- 能成功加载任意一条字幕并显示在视频上

### Phase B: 持久化与默认策略

目标：

- 实现默认开启与关闭记忆

任务：

- 在 `playerSettings` 中新增字幕配置字段
- 播放器初始化时读取字幕偏好
- 根据偏好与可用轨道自动选中默认字幕

交付标准：

- 首次打开含字幕视频时默认显示字幕
- 用户关闭后，进入下一条视频仍保持关闭
- 用户重新打开后，下一条视频再次默认开启

### Phase C: OSD 与字幕浮层

目标：

- 提供可通过遥控器稳定操作的 CC 入口

任务：

- 在底部控制条加入 `CC` 按钮
- 新增 `subtitle` overlay 模式
- 实现显示开关、语言列表、基础样式项
- 接好焦点恢复与返回键链路

交付标准：

- 遥控器能从 OSD 聚焦到 `CC`
- 能进入字幕浮层并切换开关/语言
- 返回键能稳定关闭浮层并回到 `CC` 按钮

### Phase D: 验证

目标：

- 保证浏览器、Simulator、真机三条链路都能完成基本验证

任务：

- 补充单元测试：
  - 字幕偏好读写
  - 字幕 JSON -> VTT 转换
  - 默认轨道选择规则
- 运行基础校验：
  - `npm run lint`
  - `npm run test`
  - `npm run typecheck`
  - `npm run build`
- 如进入部署阶段，再补真机验证：
  - `npm run build:webos`
  - `npm run webos:deploy -- --device <deviceName>`

交付标准：

- 静态检查与测试通过
- 真机上能看到字幕、关闭字幕、再次进入保持关闭状态

## 8. 验收标准

本需求完成时，至少需要满足：

1. 含字幕视频进入播放器后，字幕默认显示。
2. 用户通过 OSD 的 `CC` 按钮可以关闭字幕。
3. 用户关闭字幕后，切换到下一条含字幕视频时仍默认关闭。
4. 用户重新开启字幕后，当前视频可立即显示字幕。
5. 用户可以切换字幕语言/轨道。
6. 当前视频没有字幕时，`CC` 按钮有明确的无字幕反馈。
7. 遥控器焦点和返回键在字幕浮层中保持稳定。

## 9. 风险与注意事项

### 9.1 接口层风险

`/x/player/wbi/v2` 的字幕数据结构和 `playurl` 不同，接入时要注意：

- UGC / PGC 参数差异
- 未登录与登录态返回差异
- 某些视频没有字幕字段

### 9.2 播放器层风险

HTML5 `<video>` 文本轨、Shaka、以及 webOS 真机对字幕轨的支持细节可能不完全一致。  
因此首轮要优先确认：

- 浏览器里能不能正常显示
- 真机里 `<track>` 是否能稳定渲染
- 若原生文本轨表现不稳定，是否需要回退到自绘字幕层

本计划不预先承诺自绘字幕层，先按最简单可行路径验证。

### 9.3 TV 交互风险

字幕样式项如果一口气上太多，会显著增加遥控器操作成本。  
因此首轮必须控制设置项数量，先验证真实观看体验，再决定是否继续扩。
