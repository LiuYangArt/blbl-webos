# Bilibili WebOS TV UI 重构计划

## 1. 目标

基于 `ui_ref` 提供的整套 TV Mockup，建立一套可持续扩展的电视端 UI 规范与组件层，并按阶段替换当前页面骨架。  
本轮优先目标不是一次性完整还原所有页面，而是先完成“设计系统落地 + 首页试点 + 播放器页试点”，验证视觉方向、焦点流和播放器交互骨架。

## 2. 为什么不直接整页替换

当前项目 UI 仍处于 MVP 骨架阶段，现有页面主要承担焦点导航、页面栈和播放器入口验证，视觉层非常薄。  
如果直接按 `ui_ref` 逐页硬替换，会出现三个问题：

- 页面样式会高度重复，后续搜索、登录、历史页仍要回头统一。
- `ui_ref` 中大量浏览器 `hover` 表现并不等价于 TV 焦点态，直接翻译会导致遥控器体验不稳定。
- 静态稿中的字体、阴影、模糊和远程资源，需要按 webOS 真机能力重新收敛。

因此本次重构采用“规范先行、组件先行、页面渐进替换”的方式。

## 3. 总体策略

### 方案

先沉淀一层轻量 UI 规范，再抽取通用电视端组件，并用 `UI Debug` 页集中盘点真实控件来源，最后分阶段替换页面。

### 为什么选这个方案

- 能把颜色、表面层级、焦点态、间距这些稳定规则沉到全局，避免每页手写。
- 能把 mockup 中反复出现的导航、按钮、卡片、控制条沉成组件，后续页面只做组合。
- 能把已经收口过的真实控件统一陈列到 `UI Debug` 页，后续讨论“改哪一个按钮 / 哪一种进度条”时不再靠口头猜。
- 能优先验证首页和播放器页这两个最关键的 TV 场景，尽早发现焦点流或性能问题。

## 4. 本次要沉淀的 UI 规范

### 4.1 Design Tokens

- 颜色：背景、低层表面、高层表面、主品牌粉、功能蓝、正文、弱化文本、描边替代色
- 字体：展示标题、页面标题、正文、标签、按钮
- 圆角：卡片圆角、面板圆角、矩形 chip 圆角、主按钮圆角
- 间距：页面安全边距、区块间距、泳道间距、卡片信息间距
- 阴影：常规阴影、主色焦点光晕、功能色进度光晕
- 动效：焦点缩放、阴影变化、背景过渡

### 4.2 TV 交互规则

- 所有主操作必须可聚焦，不允许“可见但遥控器无法到达”的主操作。
- 焦点态优先表现为放大、提亮和外发光，而不是仅改变边框颜色。
- 返回键行为保持 `弹窗 > 面板 > 页面 > 退出应用`。
- 搜索输入优先设计为“可聚焦的输入入口”，后续再对接 webOS 输入方案。
- 播放器控制层优先保证返回、播放/暂停、快进快退、设置入口的稳定操作。

### 4.3 布局规则

- 左侧导航作为全局锚点，保持稳定宽度与激活态。
- 内容区使用 1080p 安全边距，不贴边堆内容。
- 首页 Hero 采用左文右图的非对称布局。
- 长内容优先横向泳道，不做密集纵向信息墙。

### 4.4 UI Debug 治理规则

- `UI Debug` 页只展示项目里已经存在、或已收口为通用组件的真实控件，不发明调试专用视觉体系。
- 每个展示项都要标注“用于什么页面 / 什么场景 / 来源是什么组件或样式语义”。
- 新增或统一通用控件后，要同步更新 `UI Debug` 页，保持它是当前 UI 资产的真实对照台。
- 如果某个样式只存在于 `UI Debug` 页而不服务真实页面，说明这个样式不该存在。

## 5. 组件提取清单

第一批要落地的通用组件如下：

- `AppShell`：应用整体骨架，负责侧边导航和主内容容器
- `SideNavRail`：左侧导航栏，支持当前页高亮、焦点态、底部用户入口
- `FocusButton`：统一主按钮、次按钮和玻璃按钮风格
- `TvIconButton`：统一图标 + 文本操作按钮
- `HeroBanner`：首页主视觉、标题区、元信息、CTA、分页点
- `SectionHeader`：泳道标题与右侧行动入口
- `MediaCard`：视频封面卡片，支持时长角标、标题、副标题和焦点态
- `SearchComposer`：搜索页与结果页输入面板
- `TvProgressBar`：播放器进度与缓冲层
- `PlayerControlBar`：播放器底部控制区
- `PlayerSettingsDrawer`：播放器右侧设置抽屉
- `TopbarProfilePill`：应用壳右上角账号状态入口
- `FollowingSummaryChips`：关注页顶部账号摘要标签
- `HomeChannelTabs`：首页频道切换器，并与收藏夹切换风格统一

后续扩展组件：

- `CategoryTile`
- `QrLoginPanel`
- `ProfilePanel`
- `GlassPanel`

## 6. 页面替换顺序

### Phase A：基础层

- 重构全局样式为 design tokens
- 抽出基础布局与按钮/card 组件
- 将当前 `FocusButton` 升级为可承载多种 TV 视觉变体的基础聚焦组件
- 建立 `UI Debug` 页作为真实控件总览与用途标注面板

### Phase B：首页试点

- 按 mockup 重做首页布局
- 落 Hero Banner、CTA、推荐泳道、媒体卡片
- 保持现有页面栈与进入播放器逻辑不变

### Phase C：播放器页试点

- 重做播放器顶部信息层、底部控制层、进度条、相关推荐区
- 保持返回链路和焦点逻辑稳定
- 先做演示交互骨架，不在本轮引入真实播放 SDK 复杂逻辑

### Phase D：扩展页

- 搜索/分类
- 登录二维码
- 历史/用户中心

## 8. 每阶段验收标准

### 基础层验收

- 全局颜色、间距、圆角、焦点光效已从页面散样式中抽离
- 组件可在首页和播放器页复用
- 不引入业务组件直接访问平台 API 的问题
- `UI Debug` 页能对照展示已收口的真实控件，并明确标注使用场景

### 首页验收

- 首页视觉明显对齐 `home_featured` 风格
- 所有 CTA、卡片、导航项可通过遥控器稳定聚焦
- 焦点移动时不出现“跳列”或不可达区域

### 播放器页验收

- 返回、播放/暂停、快进快退、设置项可顺畅聚焦与触发
- 控制层视觉与首页共用同一套 token
- 页面仍可通过页面栈正常回退

## 9. 验证方式

每轮改动后至少执行：

- `npm run lint`
- `npm run test`
- `npm run typecheck`
- `npm run build`

涉及 Simulator 重新验证 UI / 焦点 / 调试面板时，优先执行：

- `npm run webos:simulator`

如果 Simulator 看起来仍是旧包，先核对：

- `build/webos/index.html` 的入口脚本
- `build/webos/assets` 是否已同步到最新 legacy bundle

不要只看 `dist/assets` 就下结论。

涉及真机联调时再执行：

- `npm run build:webos`
- `npm run webos:package`
- `npm run webos:install -- --device <deviceName>`
- `npm run webos:launch -- --device <deviceName>`

## 10. 本轮实施范围

本轮会先完成以下内容：

1. 写入本计划文档与过程记录文件
2. 落地全局 design tokens 与基础 TV 组件样式底座
3. 建立 `UI Debug` 页作为样式与组件治理入口
4. 改造首页为第一阶段试点页面
5. 若进度允许，同步改造播放器页基础结构

## 11. 暂不在本轮做的内容

- 不接入真实 B 站接口
- 不接入真实扫码登录链路
- 不实现复杂搜索输入法
- 不提前做会员/版权内容适配

## 12. 风险与控制

### 风险

- 参考稿偏浏览器展示风格，直接照搬会破坏 TV 焦点体验
- 大量 blur 与阴影可能对 webOS 真机性能不友好
- 页面一次性全量替换会让焦点流调试成本过高

### 控制策略

- 先做首页和播放器页，确保关键路径可控
- 所有视觉反馈先按 D-pad 焦点态改写，再考虑 Pointer 补充
- 样式优先使用本地 CSS token，不引入额外样式框架
