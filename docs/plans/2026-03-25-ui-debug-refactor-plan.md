# UI Debug 重构方案文档

## 1. 背景

当前 `UI Debug` 页已经承担了“真实 UI 资产对照台”的职责，但现有内容组织方式更接近“按近期补充过哪些展示块来堆页面”，而不是按 UI 资产层级来组织。

这直接带来了几个问题：

- 上层基础控件与下层组合模块混排，阅读路径不稳定。
- 用户很难快速回答“这个样式到底是基础件，还是某个页面里的组合件”。
- 某些组合展示块隐含依赖了上方未先定义的 UI 语义，后续容易继续长出 `UI Debug` 专用样式。
- `UI Debug` 页虽然在展示真实组件，但还没有把“资产层级”和“依赖关系”表达清楚，讨论改动时仍然容易口头错位。

结合当前实现、仓库里的 `UI Debug` 治理规则，以及用户提出的目标，本次重构需要把 `UI Debug` 页从“展示页”收敛为“有层级约束的 UI 资产目录页”。

## 2. 本次重构目标

### 2.1 主目标

将 `UI Debug` 页重构为清晰的双层结构：

1. 上半区只展示基础单体元素。
2. 下半区只展示组合元素。
3. 每个组合元素必须显式声明其依赖的基础单体元素。
4. 组合元素中出现的 UI 语义，必须都能在上半区找到对应来源。

### 2.2 用户目标

用户进入 `UI Debug` 后，应当能在尽量少的滚动和解释成本下，完成下面几件事：

- 快速找到“基础按钮”“标题样式”“Tab / Chip”“视频缩略图 / 视频卡片”等单体元素。
- 快速找到“设置页片段”“菜单栏”“播放器控制条”“搜索入口区”等组合模块。
- 在看到组合模块时，明确知道这个模块由哪些基础元素构成。
- 讨论改 UI 时，可以先指认基础元素，再定位组合模块，不再靠模糊描述猜测。

### 2.3 治理目标

- 明确 `UI Debug` 不是业务页面样板间，而是当前项目真实 UI 资产的对照台。
- 防止“先在组合区拼一个新样式，再反向补到基础区”的倒挂式扩散。
- 防止同一种视觉语义在不同区块中被重复展示成多个看似不同、实则相近的来源。

## 3. 非目标

本次不做下面这些事情：

- 不为 `UI Debug` 单独发明新的按钮、卡片、输入框、抽屉或导航样式。
- 不把所有业务页面都缩略搬进 `UI Debug`，只保留高价值、可复用的组合模块。
- 不因为“展示更完整”而引入真实项目里尚未存在的“设置页面标准稿”或“未来页面原型”。
- 不修改业务页面视觉语义，只重整 `UI Debug` 的组织方式和展示边界。

## 4. 现状评估

当前 `UI Debug` 页大致分为：

- 顶部 Hero 说明区
- 基础控件
- 导航与入口模块
- 壳层与摘要标签
- 卡片与内容模块
- 输入与轻量说明
- 播放器复合模块

现状的主要问题如下。

### 4.1 分类维度不统一

有些区块按“控件粒度”组织，例如按钮状态；有些区块按“业务场景”组织，例如播放器模块；有些区块则按“漏补项”组织，例如壳层与摘要标签。

这导致：

- `SectionHeader`、`TopbarProfilePill`、`FollowingSummaryChips`、`SearchComposer`、`TvProgressBar` 等可独立复用元素，没有被统一归入“基础单体元素层”。
- 用户往下浏览时，需要不断切换判断标准，认知负担偏高。

### 4.2 基础件定义不完整

用户明确希望上方先列出：

- 基础按钮
- 标题样式
- Tab
- 视频缩略图样式

但当前页面里：

- 按钮被展示了
- 标题样式没有被独立定义成一个稳定的“文本 / 标题元素区”
- Tab / Chip 只作为“顶部切换器”的一部分出现
- 视频缩略图 / 视频卡片虽然展示了，但被放在后段内容模块里，没有进入基础资产层

### 4.3 组合区依赖关系是隐式的

当前组合区虽然能看到真实模块，但没有表达“它依赖哪些基础件”。

例如：

- 侧边导航预览依赖 `TvIconButton`、导航项样式、品牌区样式
- 搜索入口区依赖 `SectionHeader`、`FocusButton`、`SearchComposer`
- 播放器设置抽屉依赖 `FocusButton`、抽屉标题、section label、信息行样式

这些依赖关系目前都要靠阅读代码才能知道，不符合 `UI Debug` 页作为“对照台”的目标。

### 4.4 组合区存在“带出新语义”的风险

如果下方组合区里出现了上方没有先定义过的控件语义，后续讨论就会出现两个问题：

1. 用户会默认把它当作一个新的基础来源。
2. 实现时很容易围绕组合区临时补样式，造成 Debug 页反向孵化真实样式。

用户这次补充的要求，实际上就是为了堵住这个口子：

> 需要确保下面组合元素里面引用的 UI 元素都是在上面单体元素里的。

这条约束应当被写成显式规则，而不是只保留在口头讨论中。

## 5. 关键定义

### 5.1 什么叫“基础单体元素”

基础单体元素不要求在代码结构上是最原子组件，但必须满足以下条件：

- 具备独立、稳定、可复用的视觉语义。
- 在讨论中可以被单独点名修改。
- 可以不依赖特定页面上下文完成展示。
- 能作为组合元素的组成部分被复用。

按这个定义，以下内容都属于基础单体元素：

- `FocusButton` 的各类按钮变体
- `TvIconButton`
- `SectionHeader`
- `page-helper-text`
- `detail-chip` / `library-folder-chip`
- `MediaCard`
- `SearchComposer`
- `TopbarProfilePill`
- `FollowingSummaryChips`
- `TvProgressBar`

### 5.2 什么叫“组合元素”

组合元素指的是由多个基础单体元素，加上一层真实业务容器关系后组成的高阶模块。它们的重点不是单个视觉语义，而是“这些基础元素如何组合成一个真实场景”。

例如：

- 左侧菜单栏
- 首页频道切换条
- 搜索入口区
- 播放器底部控制条
- 播放器设置抽屉

### 5.3 组合元素引用约束

本次重构后，组合元素必须遵守以下规则：

1. 组合元素中使用到的可见 UI 语义，必须都能在基础单体元素区找到对应来源。
2. 如果某个组合元素需要一种上方尚未列出的新语义，必须先补到基础单体元素区，再允许进入组合区。
3. 如果某个组合元素只是业务内容消费者，而没有引入新的 UI 原语，不应再单独并列展示成新的来源。
4. 组合元素展示块必须标注“由哪些基础单体元素组成”。

## 6. 新的信息架构

推荐将 `UI Debug` 页重构为下面三层结构。

### 6.1 顶部轻量导览区

保留一个明显缩小的说明区，用来传达 3 件事：

- 这里是“真实 UI 资产对照台”，不是业务页面样板间。
- 阅读顺序是“先看基础单体元素，再看组合元素”。
- 下方组合元素只能引用上方已定义的单体元素。

这里不再保留当前大面积的“统一来源 / 当前规则”双卡摘要，避免说明信息压过内容本体。

### 6.2 第一大区：基础单体元素

这是整个页面的资产字典层，优先展示最容易被点名修改、也最值得统一治理的 UI 单元。

建议按下面顺序组织。

#### A. 按钮类

- 主按钮 / 次按钮 / 玻璃按钮
- 聚焦态 / 按下态 / 禁用态
- 图标按钮状态

对应来源：

- `FocusButton`
- `TvIconButton`

#### B. 文本与标题类

- 页面主标题样式
- `SectionHeader`
- 辅助说明文案 `page-helper-text`

说明：

- 用户点名“标题样式”时，应该能在这里直接找到，而不是从某个组合块里猜哪个标题才算基准。

#### C. Tab / Chip 类

- 通用轻量标签
- 首页频道 Tab
- 收藏夹切换 Chip

对应来源：

- `detail-chip`
- `library-folder-chip`
- `HomeChannelTabs` 当前所复用的按钮语义

说明：

- 这里不应只展示“顶部切换器组合”，而要先把“Tab / Chip 是什么”定义出来。

#### D. 内容入口类

- 视频卡片

对应来源：

- `MediaCard`

说明：

- 用户提到的“视频缩略图样式”，在当前项目里最稳定、最真实的承载体其实就是 `MediaCard`，不建议再人为拆成“只有封面图的 Debug 专用样本”。

#### E. 输入类

- 搜索输入面板

对应来源：

- `SearchComposer`

#### F. 轻量信息类

- 顶栏账号胶囊
- 关注摘要标签组

对应来源：

- `TopbarProfilePill`
- `FollowingSummaryChips`

#### G. 反馈类

- 播放进度条

对应来源：

- `TvProgressBar`

### 6.3 第二大区：组合元素

这里只展示真实项目中高频讨论、且确实需要看组合关系的模块。

建议保留下面这些组合块。

#### A. 左侧菜单栏

用途：

- 对照全局导航的整体样子和焦点流入口。

基础依赖：

- 图标按钮
- 品牌区文案 / 标识
- 导航项容器样式

建议标注文案：

- 由：`TvIconButton`、品牌标识、导航容器组成

#### B. 首页频道切换条

用途：

- 对照首页频道切换和收藏夹切换类条带结构。

基础依赖：

- `SectionHeader`
- Tab / Chip

建议标注文案：

- 由：`SectionHeader`、频道 Tab / Folder Chip 组成

#### C. 搜索入口区

用途：

- 对照搜索页顶部“标题 + 主操作按钮 + 输入面板”的组合关系。

基础依赖：

- `SectionHeader`
- 基础按钮
- `SearchComposer`
- 辅助文案

建议标注文案：

- 由：`SectionHeader`、按钮、`SearchComposer`、辅助文案组成

#### D. 播放器底部控制条

用途：

- 对照播放器主操作区的按钮编排方式。

基础依赖：

- 图标按钮
- 焦点按钮基线

建议标注文案：

- 由：`TvIconButton`、`FocusButton` 的播放器控制条编排组成

#### E. 播放器设置抽屉

用途：

- 对照设置类抽屉的标题、section 节奏、按钮组、信息行关系。

基础依赖：

- 基础按钮
- 标题 / 标签文本
- 辅助说明文案

建议标注文案：

- 由：按钮、标题 / section label、信息行、辅助说明组成

#### F. 设置页片段

这项只有在满足下面条件时才应保留：

- 项目中已经存在真实可复用的“设置页”或“设置面板”页面结构来源。

如果当前没有独立的真实设置页来源，就不要为了满足“看起来完整”而在 `UI Debug` 中额外拼一个“设置页面样板”。

当前阶段更合理的做法是：

- 先保留播放器设置抽屉作为设置类面板的真实基线。
- 等项目里出现真实通用设置页结构后，再把它加入组合区。

## 7. 建议移除、合并与重命名

### 7.1 顶部说明区

当前：

- Hero 区 + 两张治理摘要卡

建议：

- 合并为一个轻量导览区
- 说明入口方式即可
- 将“规则摘要”压缩为 2 至 3 条短文案

### 7.2 “导航与入口模块”

当前问题：

- 名称偏宽泛
- 会和“基础按钮 / 图标按钮”边界重叠

建议：

- 重命名为“左侧菜单栏”
- 明确它是组合元素，不再承担基础按钮解释职责

### 7.3 “壳层与摘要标签”

当前问题：

- `TopbarProfilePill` 和 `FollowingSummaryChips` 本质上更接近基础单体元素

建议：

- 从组合区移到基础单体元素区

### 7.4 “卡片与内容模块”

当前问题：

- “顶部切换器”与“视频卡片”不属于同一层级

建议：

- “视频卡片”移到基础单体元素区
- “顶部切换器”重构为“Tab / Chip 单体元素”展示
- 如需保留频道条整体效果，则在组合区以“首页频道切换条”方式展示

### 7.5 “输入与轻量说明”

当前问题：

- `SearchComposer` 与 `page-helper-text / detail-chip` 都应先归入基础层

建议：

- 拆分到“输入类”和“文本 / Chip 类”
- 如果要保留搜索页顶部结构，则在组合区新增“搜索入口区”

### 7.6 “播放器复合模块”

建议：

- 保留“播放器底部控制条”
- 保留“播放器设置抽屉”
- `TvProgressBar` 移到基础单体元素区

原因：

- 进度条本身是稳定单体元素，不应只作为播放器区子项出现

## 8. 建议的页面顺序

最终页面建议按下面顺序排布：

1. 轻量导览区
2. 基础单体元素
3. 组合元素

其中：

- 基础单体元素内部再按“按钮 -> 文本标题 -> Tab / Chip -> 内容入口 -> 输入 -> 轻量信息 -> 反馈”排序
- 组合元素内部按“导航 -> 首页频道条 -> 搜索入口区 -> 播放器控制条 -> 播放器设置抽屉”排序

这样能保证用户的阅读节奏始终是：

先认识资产，再看拼装方式。

## 9. 依赖映射表

下面这张映射关系是本次重构的核心治理规则。

### 9.1 基础单体元素清单

- 按钮：`FocusButton`
- 图标按钮：`TvIconButton`
- 标题：`SectionHeader`
- 辅助文案：`page-helper-text`
- Tab / Chip：`detail-chip`、`library-folder-chip`
- 视频卡片：`MediaCard`
- 输入面板：`SearchComposer`
- 顶栏胶囊：`TopbarProfilePill`
- 摘要标签：`FollowingSummaryChips`
- 进度条：`TvProgressBar`

### 9.2 组合元素依赖

- 左侧菜单栏 -> 图标按钮、品牌区文案、导航容器
- 首页频道切换条 -> 标题、Tab / Chip
- 搜索入口区 -> 标题、基础按钮、输入面板、辅助文案
- 播放器底部控制条 -> 图标按钮、基础按钮
- 播放器设置抽屉 -> 基础按钮、标题 / 标签、辅助文案、信息行

### 9.3 验收规则

后续实现时，每新增一个组合块，都必须先过下面 3 个检查：

1. 这个组合块中可见的 UI 元素，上方是否都已有对应单体来源。
2. 它是不是在展示真实业务组合，而不是为了 Debug 页单独拼一个样板。
3. 它有没有把“内容消费者”误表达成“新的基础来源”。

## 10. 实施建议

### 10.1 组件组织

建议先在 `UiDebugPage.tsx` 内部做一次结构收口：

- 将“基础单体元素”和“组合元素”拆成两个顶层 section
- 每个 section 只负责一个层级
- 给 `ShowcaseCard` 增加一行“由哪些基础元素组成”的说明字段，供组合区使用

### 10.2 命名建议

建议页面里的标题直接使用用户能稳定复述的说法，不要过于抽象：

- `基础单体元素`
- `组合元素`
- `左侧菜单栏`
- `首页频道切换条`
- `搜索入口区`
- `播放器底部控制条`
- `播放器设置抽屉`

### 10.3 可读性建议

- 上方单体元素尽量以“并排对照”为主，方便扫视。
- 下方组合元素尽量以“真实业务片段”为主，避免为了凑满页面把多个组合块硬塞在一起。
- 对组合元素的用途说明，应优先写“用于哪里”，其次才是“来源是什么”。

## 11. 分阶段任务拆分

### Phase 1：信息架构重排

- 把页面重组为“导览区 / 基础单体元素 / 组合元素”
- 调整区块顺序与标题命名
- 删除或合并当前层级不清的旧区块

### Phase 2：基础单体元素补齐

- 补全标题样式、Tab / Chip、视频卡片、输入、轻量信息、进度条等基础资产展示
- 将原本位于下方的基础元素迁移到上方

### Phase 3：组合元素收敛

- 保留高价值组合块
- 给每个组合块补依赖说明
- 删除任何引用了“上方未定义元素”的组合块，或先把所需元素补到基础区

### Phase 4：校验与治理

- 逐项检查组合区依赖是否闭环
- 确认没有 Debug 专用新样式渗入
- 同步补充必要的注释说明，保证后续继续维护时不走样

## 12. 验收标准

完成后，`UI Debug` 页至少应满足以下标准：

### 12.1 结构标准

- 页面顶层只剩“基础单体元素”和“组合元素”两大层。
- 用户不需要阅读代码，也能看懂每个区块处于哪一层。

### 12.2 资产标准

- 用户提到“基础按钮”“标题样式”“Tab”“视频缩略图 / 视频卡片样式”时，都能在上半区直接找到。
- `SectionHeader`、`SearchComposer`、`TopbarProfilePill`、`FollowingSummaryChips`、`TvProgressBar` 等真实可复用元素，都进入基础单体元素区。

### 12.3 依赖标准

- 下半区每个组合块都清楚标出“由哪些基础元素组成”。
- 下半区出现的可见 UI 语义，都能在上半区找到对应来源。

### 12.4 治理标准

- 不新增只服务 `UI Debug` 页的 UI 视觉语义。
- 不把业务消费者错误展示成新的基础来源。
- 后续新增通用 UI 时，维护者知道应先补基础区，再决定是否需要新增组合区案例。

## 13. 风险与应对

### 风险 1：把基础单体元素定义得过细

风险：

- 页面会变成零件手册，失去可读性。

应对：

- 保持“可单独点名修改、具备稳定语义”才进入基础层，不做纯 DOM 级拆分。

### 风险 2：组合元素过少，用户觉得不直观

风险：

- 只看基础件不容易理解真实页面关系。

应对：

- 保留高价值组合块，但要求每个组合块都回指基础件来源。

### 风险 3：未来新增真实页面后再次打乱结构

风险：

- 维护者可能继续按业务页面维度往下追加区块。

应对：

- 将“先补基础层，再补组合层”的规则写入本方案并体现在 issue 验收标准里。

## 14. 建议实施范围

本次重构优先修改：

- `src/features/debug/UiDebugPage.tsx`
- `src/styles/app.css`

必要时配合小范围调整：

- 某些基础组件的展示文案或样例数据

但不应扩散到业务页面真实实现，除非发现某个展示所依赖的基础语义本身就不稳定。

## 15. Task List

1. 压缩顶部说明区，改成轻量导览。
2. 建立“基础单体元素”顶层分区。
3. 将按钮、标题、Tab / Chip、视频卡片、输入、轻量信息、进度条统一迁入基础区。
4. 建立“组合元素”顶层分区。
5. 将左侧菜单栏、首页频道切换条、搜索入口区、播放器底部控制条、播放器设置抽屉整理为组合区。
6. 为每个组合元素增加“由哪些基础元素组成”的依赖说明。
7. 删除或合并所有违反层级规则的旧展示块。
8. 对照本方案自检“组合元素引用的 UI 是否都已在上方单体元素中定义”。
