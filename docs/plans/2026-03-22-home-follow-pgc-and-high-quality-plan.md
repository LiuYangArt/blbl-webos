# 首页登录态频道与高画质能力专项计划

## 1. 背景

当前 `bilibili_webos` 已经完成扫码登录、基础详情页、播放器取流与若干用户资产页，但仍存在两个直接影响可用性的缺口：

1. 登录后首页没有出现用户预期中的登录态频道，尤其缺少：
   - `正在关注`
   - `订阅剧集`
2. 播放器虽然已经走到 `playurl + DASH`，但用户感知上仍主要只能播放到 `720P`，缺少明确的高画质能力与切换入口。

本计划先不直接写实现代码，而是先把参考项目的事实、当前仓库的差距、以及 WebOS 端的落地顺序梳理清楚，避免后续边改边猜。

## 2. 结论先行

### 2.1 关于 1080P / 4K / HDR

先说结论：

- `1080P / 4K`：从接口能力上看，**有机会获取**。
- `HDR / 杜比视界 / HDR Vivid`：安卓参考项目已经在数据模型层承认这些档位，但当前 WebOS 仓库**还没有进入“可稳定获取并可播放”的状态**。

原因如下：

- 当前 WebOS 仓库的 [bilibili.ts](../../src/services/api/bilibili.ts) 已经会请求 `/x/player/playurl`，并带：
  - `fnval=4048`
  - `fourk=1`
- 它也已经能解析：
  - `support_formats`
  - `accept_quality`
  - `dash.video`
  - `dash.audio`
- 但当前 [PlayerPage.tsx](../../src/features/player/PlayerPage.tsx) 调用 `fetchPlaySource(bvid, cid)` 时，没有传入更高的目标 `qn`，所以默认还是走 `quality = 80` 这一档。
- 也就是说，当前代码的问题更接近：
  - “已经知道有更高档位”
  - 但还没有真正把“请求更高档位 + UI 切换 + 权限提示 + 真实播放验证”这条链路做完整

安卓参考项目 `PiliPlus` 在这件事上做得更完整：

- `lib/http/video.dart` 的 `videoUrl()` 会带 `qn`、`fnval=4048`、`fourk=1`
- `lib/models/common/video/video_quality.dart` 明确定义了：
  - `1080P`
  - `1080P 高码率`
  - `1080P 60帧`
  - `4K`
  - `HDR`
  - `杜比视界`
  - `8K`
  - `HDR Vivid`
- `lib/models/video/play/url.dart` 会把 `flac.audio`、`dolby.audio` 也并入音轨解析
- `lib/pages/video/controller.dart` 会按“先选画质、再选 codec”的顺序重新构建播放器数据源
- `lib/pages/video/widgets/header_control.dart` 会把完整画质列表展示给用户，并明确区分“可选”和“当前账号/设备未解锁”

所以本轮可以做出的事实判断是：

- `1080P / 4K`：应该纳入近期落地范围
- `HDR / 杜比视界 / HDR Vivid`：先做数据模型与 UI 识别，真实可播性放到真机专项验证阶段，不在第一轮承诺

### 2.2 关于 “正在关注”

安卓参考项目没有把“正在关注”做成和截图完全一样的首页频道条，但它已经具备对应的数据基础：

- `lib/http/dynamics.dart`
  - `followUp()` -> `/x/polymer/web-dynamic/v1/portal`
  - `followDynamic()` -> `/x/polymer/web-dynamic/v1/feed/all`

这两条接口承担的职责不同：

- `portal`
  - 更适合做“关注对象与更新概览”
  - 会返回 `up_list` 和 `live_users`
- `feed/all`
  - 才是完整动态流
  - 数据会更杂，包含非视频类动态

因此对 TV 端更合适的做法不是照搬移动端动态页，而是：

1. 首页二级频道里提供 `正在关注`
2. 首版只展示“可直接进入播放或详情的关注更新”
3. 需要时再补充“关注中的直播 / 动态详情 / 更多动态类型”

### 2.3 关于 “订阅剧集”

安卓参考项目对这块的实现更直接：

- `lib/http/fav.dart`
  - `FavHttp.favPgc()` -> `/x/space/bangumi/follow/list`
- `lib/pages/pgc/controller.dart`
  - 登录后调用 `queryPgcFollow()`
- `lib/pages/pgc/view.dart`
  - 在页面里渲染 `最近追番 / 最近追剧`

其中关键参数是：

- `type=1` -> 番剧
- `type=2` -> 影视

这说明“订阅剧集”本质上不是推荐流，而是用户 PGC 订阅资产流。  
对 WebOS 端来说，最合理的首版设计是：

- 首页出现 `订阅剧集` 二级频道
- 进入后先展示“最近追番 / 最近追剧”的可播放卡片
- 再按需要扩展“查看全部”

## 3. 当前 WebOS 仓库差距

### 3.1 首页结构差距

当前 [HomePage.tsx](../../src/features/home/HomePage.tsx) 只有：

- Hero
- 首页推荐
- 热门速看

缺少：

- 首页二级频道条
- 登录前后不同的首页频道集
- 频道级焦点记忆与切换状态

### 3.2 路由与信息架构差距

当前 [routes.ts](../../src/app/routes.ts) 与 [SideNavRail.tsx](../../src/components/SideNavRail.tsx) 只维护左侧一级导航。  
这本身不是问题，问题在于首页内部还没有二级频道容器。

专项设计决定如下：

- **不把 `正在关注` / `订阅剧集` 强行塞进左侧一级导航**
- **优先在首页内部补一个 TV 友好的二级频道条**

原因：

- TV 一级导航应该少而稳
- 登录态频道更适合归属于首页，而不是打散全局导航
- 这也更接近安卓参考项目“一级导航克制、页内内容丰富”的结构

### 3.3 播放器能力差距

当前播放器的关键短板不是“完全不会取高画质”，而是：

1. 目标质量默认固定在 `qn=80`
2. 没有手动画质切换 UI
3. 没有把 `accept_quality / support_formats / limit_reason` 反馈给用户
4. 还没有承接 `HDR / 杜比 / FLAC` 等高级元信息
5. Simulator 与真机能力不同，当前不能只靠 Simulator 判定高画质链路是否成立

## 4. 设计原则

### 4.1 首页频道原则

- 首页二级频道必须是横向、清晰、可直接通过遥控器切换的
- 登录态频道只在认证成功后出现
- 登录后频道默认顺序建议为：
  - `个性推荐`
  - `正在关注`
  - `订阅剧集`
  - `热门视频`
  - `排行`
  - `直播`
- 若后续发现频道太多影响 TV 操作效率，可再压缩；但本轮先与用户预期对齐

### 4.2 高画质原则

- 先做“获取与切换”
- 再做“高级格式识别”
- 最后才做“HDR 真机链路优化”

换句话说，本轮优先级应是：

1. 先解决 `1080P / 4K` 的请求、切换、提示
2. 再补齐 HDR / 杜比相关数据模型
3. 最后再决定是否为高级格式做专门播放策略

### 4.3 TV 场景原则

- `正在关注` 首版只展示可播放内容，不做复杂动态互动
- `订阅剧集` 首版只展示最近订阅与继续观看价值高的条目
- 一切以 `5-way 焦点稳定` 和 `可直接播放` 为主，不引入移动端密集信息流

## 5. 实施范围拆分

## 5.1 Phase A: 首页二级频道框架

目标：

- 把首页从“固定三段内容”升级为“可切换频道的内容容器”

任务：

- 在首页新增二级频道条组件
- 频道项支持：
  - 默认焦点
  - 左右切换
  - 记忆上次选中的频道
  - 登录态条件渲染
- 首页内容区改为按频道渲染，而不是固定写死

交付标准：

- 未登录时至少可见：
  - `个性推荐`
  - `热门视频`
  - `排行`
  - `直播`
- 已登录时额外可见：
  - `正在关注`
  - `订阅剧集`

## 5.2 Phase B: `正在关注` 数据接入

目标：

- 首页能展示“我关注的 UP / 关注内容更新”

任务：

- 新增动态相关 API 封装到 `src/services/api/*`
- 至少封装两类数据：
  - `/x/polymer/web-dynamic/v1/portal`
  - `/x/polymer/web-dynamic/v1/feed/all`
- 设计 TV 首版数据裁剪策略：
  - 仅保留可转成视频卡片或可进入详情页的动态项
  - 非视频文本动态先不进入首页主列表
- 可选地将 `portal.up_list` 做成顶部“关注更新头像条”或状态摘要

风险：

- 动态流内容类型复杂，直接全量接入会破坏 TV 信息密度
- 部分动态卡片可能需要额外字段映射，不能直接复用现有 `VideoCardItem`

交付标准：

- `正在关注` 频道可加载出稳定的可播放内容
- 焦点移动不出现“看得见但选不到”的卡片

## 5.3 Phase C: `订阅剧集` 数据接入

目标：

- 首页能展示“最近追番 / 最近追剧”

任务：

- 新增 PGC follow API 封装
  - `/x/space/bangumi/follow/list`
- 设计 WebOS 数据模型：
  - 统一字段适配现有 TV 卡片
  - 保留 `season_id / ep_id / cover / title / badge / progress / publish_time` 等关键字段
- 首页 `订阅剧集` 频道首版建议内部分两段或两个切换子标签：
  - `番剧`
  - `影视`
- 如果实现成本过高，首版也可只先做“番剧优先”，但文档层面以“统一订阅剧集频道”作为目标

交付标准：

- 登录后进入 `订阅剧集` 频道，能稳定看到个人订阅内容
- 卡片点击后能进入后续详情/播放链路

## 5.4 Phase D: 画质切换与高档位请求

目标：

- 把播放器从“默认 720P/1080P 单档请求”升级为“可请求、可切换、可提示多画质”

任务：

- 给 `fetchPlaySource()` 增加显式目标 `qn`
- 在播放器设置面板增加“画质”入口，而不只是“编码策略”
- 请求高档位时：
  - 优先按用户选择的 `qn`
  - 如果接口实际返回更低档位，显示降级原因
- 在 UI 上展示：
  - 当前请求画质
  - 实际返回画质
  - 是否受登录/VIP/版权限制

关键约束：

- 画质切换不能只在本地重排已有轨道；高档位通常需要重新请求 `playurl`
- 高画质切换后要保留当前播放进度并重新装载播放器

交付标准：

- 至少可在 UI 中切换：
  - `720P`
  - `1080P`
  - `1080P+ / 4K`（若接口与账号允许）
- 若接口不返回请求档位，界面能明确告诉用户不是“按钮坏了”，而是“当前权限/资源不可用”

## 5.5 Phase E: HDR / 杜比数据层补齐

目标：

- 先把高级质量档位识别出来，为真机专项验证做准备

任务：

- 扩展当前播放类型，显式支持质量枚举：
  - `HDR`
  - `杜比视界`
  - `HDR Vivid`
  - `4K`
  - `1080P+`
- 扩展音频模型，承接：
  - `dolby.audio`
  - `flac.audio`
- 在播放器设置面板显示“接口宣称存在该档位”，但不默认承诺可播放

注意：

- 这一阶段是“识别与展示”
- 不是“高级格式已经稳定可播”

## 5.6 Phase F: 详情页与 PGC 链路补齐

目标：

- 让首页 `订阅剧集` 与 `正在关注` 进入后的落点不再只有 UGC 播放

任务：

- 评估是否新增：
  - PGC 详情路由
  - season / episode 级别的路由载荷
- 明确 UGC 与 PGC 的播放取流参数差异
- 把 PGC 的详情、选集、继续观看链路做成 TV 友好的骨架

说明：

- 如果不补这一步，`订阅剧集` 频道即使能出卡片，也会在详情/播放阶段丢失语义

## 6. 数据与组件建议

### 6.1 建议新增的数据模型

- `HomeChannelKey`
  - `personalized`
  - `following`
  - `subscriptions`
  - `hot`
  - `ranking`
  - `live`
- `FollowFeedItem`
- `PgcSubscriptionItem`
- `PlayQualityTier`
  - 补齐 `1080P+ / 4K / HDR / 杜比视界 / HDR Vivid`

### 6.2 建议新增的组件

- `HomeChannelTabs`
  - 首页频道条
- `ContentStrip`
  - 统一横向内容带
- `QualitySelector`
  - 播放器画质选择器
- `PlaybackCapabilityNotice`
  - 统一承接“该画质未开放 / 需要会员 / 当前设备不建议”这类提示

## 7. 风险与边界

### 7.1 风险

- `正在关注` 的动态流过于复杂，若不先裁剪内容类型，会导致首页失控
- `订阅剧集` 实际上是 PGC 体系，后续一定会牵扯到 `season/ep` 路由与详情页
- `1080P+ / 4K / HDR` 是否真能播，不仅取决于接口，还取决于：
  - 账号权限
  - 视频本身是否提供该档位
  - webOS 真机播放能力
  - 当前 DASH / MSE 链路的稳定性

### 7.2 本轮不承诺

- DRM 内容播放
- 会员专属能力完整绕通
- HDR / 杜比视界首轮稳定播放
- 动态社区的完整互动能力

## 8. 推荐实施顺序

建议按以下顺序推进：

1. 首页二级频道框架
2. `订阅剧集` 频道接入
3. `正在关注` 频道接入
4. 播放器画质切换
5. 高级质量档位识别
6. PGC 详情/播放链路补齐

这样安排的原因是：

- `订阅剧集` 的数据结构更稳定、TV 适配成本更低
- `正在关注` 虽然用户感知强，但数据裁剪更复杂
- 画质切换虽重要，但如果首页内容入口不完整，整体体验仍然缺块

## 9. 验证策略

完成后至少验证：

- `npm run lint`
- `npm run typecheck`
- `npm run build`

功能验证建议分两层：

### 9.1 Simulator 验证

- 登录成功后首页频道条是否按预期出现
- `正在关注` / `订阅剧集` 是否能进入并正确聚焦
- 画质面板是否能显示更多档位与限制提示

### 9.2 真机验证

- `1080P / 4K` 请求后是否真的拿到更高档位
- 切换画质后是否能继续播放且保留进度
- 高级档位在 `LG C1` 上的可播性是否成立

## 10. 下一步建议

下一步直接进入实现时，建议先做 **Phase A + Phase C**，也就是：

- 先把首页二级频道框架搭起来
- 再先接 `订阅剧集`

理由：

- 用户最直观看到的是首页结构变化
- `订阅剧集` 比 `正在关注` 更容易稳定复用为 TV 卡片
- 做完这两步后，再补 `正在关注` 与画质切换，节奏更稳
