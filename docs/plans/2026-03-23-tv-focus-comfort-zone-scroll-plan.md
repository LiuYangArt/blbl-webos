# WebOS TV 焦点舒适区滚动重构方案

## 1. 文档目标

本文档用于为 `bilibili_webos` 设计一套统一的 TV 端“焦点舒适区滚动”方案。  
目标不是继续修首页、关注页、热门页等单个页面的回滚 bug，而是把“焦点如何驱动页面滚动”提升为焦点引擎的统一能力。

本轮只输出方案与重构计划，不进入代码实现。  
待方案确认后，再按本文档进入开发。

---

## 2. 背景与问题定义

### 2.1 当前现象

目前在首页、关注页等长内容页中，都会出现同类问题：

- 焦点从上方按钮区进入下方视频列表后，页面会向下滚动。
- 当用户再按上键返回时，焦点虽然能回到上方按钮或第一排卡片。
- 但屏幕只把焦点元素卡在顶部边缘附近。
- 该 section 的标题、说明、统计信息、辅助标签等非焦点内容，往往仍然被挤出屏幕。

用户的主观感受就是：

- “焦点回来了，但这个区域上面的信息没回来”
- “页面最高只能回到某一行按钮，视图无法恢复到这个 section 的完整起始位置”

### 2.2 根本原因

根因不在某一个页面，而在当前焦点滚动策略：

- 当前引擎只保证“焦点元素可见”
- 不保证“焦点上下文可读”
- 不知道“第一排焦点应该带出整个 section 顶部”
- 不知道“TV 页面里焦点应停留在屏幕舒适区，而不是长期贴边”

因此，这不是首页特例，也不是简单的 `leaveFor` 配错，而是 **滚动模型层级不够**。

## 3. 调研结论：`jellyfin-webos` 能参考什么

### 3.1 已确认的事实

本地参考仓库：`F:\CodeProjects\bilibili_tv_android\jellyfin-webos`

调研结论如下：

- `frontend/js/index.js` 里的导航逻辑只覆盖连接页的简单 `input / button` 线性切换。
- 仓库主流程是把真正的 Jellyfin Web 页面加载进 `iframe#contentFrame`。
- `frontend/js/webOS.js` 只负责 `NativeShell / AppHost` 适配。
- `frontend/css/webOS.css` 只隐藏 scrollbar，并未实现复杂滚动策略。

### 3.2 对本项目的实际启发

因此，本地这个 `jellyfin-webos` 仓库：

- **不能作为实现级参考**
- **可以作为产品效果级参考**

也就是说，我们能借鉴的是：

- TV 列表浏览时，焦点不要贴着屏幕边缘
- 焦点回到第一排元素时，section 顶部的信息要能一起回来
- 页面滚动应该围绕“用户视线稳定”和“信息可读”组织

不能借鉴的是：

- 直接抄某个现成的 `scrollIntoView` 或焦点算法

## 4. TV 端通用交互原则

这部分是本方案的核心原则，也是后续所有页面共享的统一规则。

### 4.1 焦点不是只要“看得见”就够了

浏览器式的“元素出界了再滚一点”并不适合 TV。

对电视来说，焦点除了必须可见，还要满足：

- 用户能迅速理解自己在当前内容区的什么位置
- 焦点上下文可见
- 页面在连续按上下键时，视线移动要稳定

所以 TV 端更合理的目标是：

- **焦点进入舒适区**
- 而不是仅仅 **没有越界**

### 4.2 视口应有“舒适区”

推荐把可视区分成三部分：

- 顶部保留区：用于露出 section 标题、说明、标签、顶部状态
- 焦点舒适区：焦点优先停留在这个区域
- 底部预读区：让用户提前看到下一行/下一屏内容

具体效果上应表现为：

- 焦点向上回时，不会被卡在屏幕顶边
- 焦点向下浏览时，也不会总是贴在屏幕底边
- 视口滚动更多地围绕“让焦点保持在中上部稳定区域”进行

### 4.3 section 顶部信息必须被当作布局资产

对于 TV 页面，很多关键信息本来就不是可聚焦元素，例如：

- 区块标题
- 描述文案
- 筛选说明
- 数量统计
- 频道提示

这些信息不能因为焦点回到了 section 第一排按钮，就继续被压在屏幕外。

因此：

- 当焦点回到某个 section 的首行焦点元素时
- 页面滚动应优先保证该 section 的头部信息一起可见

## 5. 目标方案总览

推荐把当前“元素级可见性修正”升级为两层模型：

### 第一层：全局焦点舒适区滚动

由焦点引擎统一负责：

- 读取当前焦点元素的 rect
- 读取当前滚动容器的视口范围
- 判断焦点是否落在“舒适区”内
- 若不在，则把页面滚动到更合适的位置

这层解决的是：

- 焦点总贴边
- 上下返回时视口不稳定
- 各页面滚动手感不一致

### 第二层：section 头部保留规则

由 section 配置声明：

- 当前 section 是否需要“头部保留”
- 当前 section 的滚动锚点是 section 自身、头部容器，还是首行焦点元素
- 当焦点处于首行时，视口应优先对齐到哪里

这层解决的是：

- 标题和说明露不出来
- 第一排按钮回来了，但区块起始位置没回来
- 各种首页/关注页/搜索页顶部信息区丢失

## 6. 目标架构设计

### 6.1 焦点引擎新增职责

当前 `src/platform/focus/engine.ts` 负责：

- 焦点定位
- 几何导航
- section 注册
- 焦点恢复

后续建议它再承担一项统一职责：

- **焦点驱动的滚动校正**

建议将滚动相关能力内聚到焦点层，而不是页面层：

- 页面负责声明结构与意图
- 引擎负责统一执行滚动策略

### 6.2 建议新增的配置模型

当前 `FocusSectionConfig` 只有：

- `id`
- `selector`
- `defaultElement`
- `enterTo`
- `leaveFor`
- `disabled`
- `group`

建议新增一个 section 级滚动配置，例如：

```ts
type FocusScrollMode = 'none' | 'comfort-zone';

type FocusScrollAnchor = 'section-start' | 'section-header' | 'focused-element';

type FocusSectionScrollConfig = {
  mode?: FocusScrollMode;
  anchor?: FocusScrollAnchor;
  preserveHeaderWhenFirstRowFocused?: boolean;
  topOffset?: number;
};
```

然后扩展到：

```ts
type FocusSectionConfig = {
  id: string;
  selector?: string;
  defaultElement?: string;
  enterTo?: 'default-element' | 'last-focused';
  leaveFor?: Partial<Record<Direction, string>>;
  disabled?: boolean;
  group?: FocusGroup;
  scroll?: FocusSectionScrollConfig;
};
```

### 6.3 建议新增的全局舒适区配置

除了 section 配置外，还应有一组全局默认值，例如：

```ts
type FocusViewportComfortZone = {
  top: number;
  bottom: number;
  left: number;
  right: number;
};
```

建议默认按 1080p TV 视口理解为：

- 顶部保留更大
- 底部保留次之
- 左右保持轻量边距

其目标不是精确像素，而是形成稳定体验：

- 焦点优先停在屏幕中上部
- 给 section 标题和说明留空间
- 给下一行内容留预读空间

## 7. 建议的滚动策略

### 7.1 全局策略：舒适区优先

每次焦点变化后，引擎不再只做“越界修正”，而要执行：

1. 读取当前焦点元素位置
2. 读取当前滚动容器可视区域
3. 判断焦点是否落在舒适区内
4. 如果不在，则滚动到舒适区目标位置

关键区别：

- 不是 `看不见了才滚`
- 而是 `不舒服就滚`

### 7.2 section 首行策略：优先保头部

当满足以下条件时：

- 当前 section 开启 `preserveHeaderWhenFirstRowFocused`
- 当前焦点位于 section 的首行或首个可交互带

则优先让视口对齐到：

- `section-start` 或 `section-header`

这样能保证：

- 焦点回到 tabs / 首行卡片时
- 该 section 的标题、描述、统计标签一起露出来

### 7.3 非首行策略：维持舒适浏览节奏

当焦点已进入列表主体时：

- 不需要每次都把 section 顶部硬拉回来
- 只需要维持焦点在舒适区内

这样可以避免：

- 页面频繁大幅抖动
- 每按一下都出现过强回弹
- 中段浏览时视图跳来跳去

### 7.4 滚动方式：保持 `auto`

由于 webOS 真机性能有限，本方案建议继续坚持：

- 不做平滑滚动动画
- 不做缓动回正
- 一律采用 `behavior: 'auto'`

目标是：

- 稳定
- 清楚
- 可预期

而不是浏览器端那种强调视觉平滑的滚动。

## 8. 页面层如何接入

### 8.1 页面层不再手写 `scrollIntoView()`

后续页面不应继续在：

- `onFocus`
- `onFocusCapture`
- `useEffect`

里各自补一套 `scrollIntoView()`。

原因：

- 容易互相打架
- 页面多了以后难以维护
- 行为不一致

页面层应只声明：

- 这是哪个 section
- 这个 section 是否需要头部保留
- 首行焦点回来时应以哪个锚点回滚

### 8.2 首批应接入的页面

这套能力不是首页专属，建议首批覆盖：

- 首页频道 tabs + 频道内容区
- 关注页顶部说明区 + 第一排卡片
- 热门页第一块内容区
- 搜索页顶部操作区 + 热搜区
- 历史页顶部说明区 + 第一条内容
- 收藏/稍后/个人中心的第一块主内容区

### 8.3 页面声明粒度建议

不是整个页面只设一个滚动规则，而是按 section 声明：

- 顶部操作区
- 列表区
- 横向泳道区
- 说明区

让引擎知道：

- 何时只保证焦点舒适区
- 何时需要带出整块头部信息

## 9. 与现有焦点系统的关系

本方案不是推翻当前 `Section + leaveFor + last-focused` 模型，而是在其上补齐滚动层。

也就是说：

- 几何导航与 section 跳转继续保留
- 焦点恢复继续保留
- 遥控器按键映射继续保留

新增的是：

- 焦点变化后的统一滚动校正
- section 级滚动意图声明

这意味着后续实现应优先复用当前：

- `src/platform/focus/engine.ts`
- `src/platform/focus/types.ts`
- `src/platform/focus/FocusSection.tsx`

而不是重新造一套并行焦点系统。

## 10. 实施阶段建议

### Phase 1：补齐焦点滚动配置模型

目标：

- 在 `types.ts` 中定义 section 级滚动配置
- 在 `FocusSection.tsx` 中支持声明滚动意图
- 保证旧页面不配置时仍按兼容策略运行

### Phase 2：在引擎中实现舒适区滚动

目标：

- 将 `ensureElementVisible()` 升级为“舒适区滚动”
- 默认仍走 `auto`
- 支持 section 级覆盖

### Phase 3：实现 section 头部保留规则

目标：

- 当焦点处于首行时，允许使用 section 锚点滚动
- 保证顶部标题/说明/统计一起回来

### Phase 4：首批页面接入

目标：

- 首页
- 关注页
- 热门页
- 搜索页
- 历史页

### Phase 5：真机联调与规则收敛

目标：

- 在模拟器和 LG C1 真机上验证
- 调整舒适区参数
- 收敛 section 配置写法

## 11. 验收标准

方案落地后，至少应满足：

1. 从任意长列表向上返回时，焦点不会长期贴在屏幕顶边。
2. 当焦点回到某个 section 的首行按钮或首行卡片时，该 section 的标题和说明可见。
3. 首页、关注页、热门页、搜索页、历史页表现一致。
4. 页面层不再散落手写 `scrollIntoView()` 补丁。
5. 焦点框与滚动都不使用平滑动画，优先保证稳定。

---

## 12. 风险与边界

### 12.1 风险

- 如果舒适区参数过大，可能导致每次移动都滚动，产生“晃动感”。
- 如果 section 头部保留规则写得过于激进，可能让中段浏览也频繁回顶。
- 某些横向列表与纵向列表混合页面，可能需要更细粒度的 section 划分。

### 12.2 非目标

本方案不解决以下问题：

- 重新设计各页面视觉布局
- 重写几何导航评分算法
- 引入新的第三方焦点库
- 播放器页沉浸模式的特殊 overlay 滚动

这些内容若需要，可在后续单独规划。

---

## 13. 推荐结论

推荐后续不要再继续按页面逐个修：

- 首页补一个 `scrollIntoView`
- 关注页补一个 `scrollIntoView`
- 搜索页再补一个 `scrollIntoView`

这种方式短期能止血，但长期会把滚动逻辑散落到各页面。

更合理的做法是：

- 以当前 `Section + Focus Engine` 为基础
- 在引擎层补齐“TV 焦点舒适区滚动”
- 在 section 层补齐“头部保留规则”
- 页面层只声明意图，不再手写滚动

这才是一套能适配所有页面、并适合后续长期演进的统一方案。
