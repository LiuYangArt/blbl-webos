# 2026-03-23 TV 焦点舒适区滚动与滚动根误判 Postmortem

## 结论

这次问题表面上看是：

- 焦点框已经随着方向键上下移动
- 但屏幕并没有跟着焦点滚动
- 最终焦点会直接跑出屏幕外

真正的根因不是“焦点导航失效”，也不是“舒适区参数不对”，而是：

1. 我们先把滚动策略从“元素可见”升级成了“舒适区 + 首行保头部”
2. 但初版实现仍然默认把 `window / document.scrollingElement` 当成滚动目标
3. 在 webOS Simulator 这类 TV 容器场景里，真正应该滚动的是应用主内容区，而不是浏览器窗口

所以用户看到的异常会变成一种很迷惑的状态：

- 焦点逻辑其实已经工作
- 视口却完全不动
- 看起来就像“滚动方案没生效”

最终稳定方案是两步一起成立：

- **应用壳显式提供 TV 主内容滚动容器**
- **焦点引擎优先滚最近的滚动根，而不是默认滚 window**

## 用户可见现象

用户在首页视频区连续按上下键时看到的是：

1. 焦点框会继续往下切到新的卡片
2. 但屏幕停在原地
3. 新焦点会直接移出当前视口
4. 用户主观感受就是“屏幕根本没有跟着滚”

这和之前 2026-03-21 那次“`preventScroll: true` 把默认滚动禁掉”的问题很像，但又不是同一个根因。

上一次的问题是：

- 引擎主动阻止了浏览器默认滚动

这一次的问题是：

- 引擎确实在发滚动命令
- 只是滚到了错误的对象

## 根因拆解

### 根因一：焦点导航和舒适区判断其实已经在工作

从实际表现看，方向键继续按下时没有出现：

- 焦点回到左侧导航
- 焦点停在原位
- 焦点丢失

这说明：

- 空间导航候选选择是正常的
- `FocusSection` 的接入也是正常的
- 焦点变化后的舒适区逻辑至少已经被触发

问题不在“找不到下一个焦点”，而在“找到了之后滚给谁看”。

### 根因二：初版实现默认滚 `window`

焦点引擎里初版舒适区实现默认用的是：

- `document.scrollingElement`
- `window.scrollTo`

这在普通浏览器文档流页面里通常没问题。

但当前项目的 TV UI 壳层结构并不是“让整个浏览器窗口自然滚动”：

- 左侧导航固定在一侧
- 顶部状态条固定在主内容上方
- 实际长内容应该在主内容区内部滚动

一旦页面运行环境更偏向 app 容器，而不是普通网页，继续把 `window` 当滚动目标就会失效。

### 根因三：应用壳没有显式声明“谁是滚动根”

这次也暴露了一个结构层面的缺口：

- 页面虽然有 `.tv-page-content`
- 但它之前并没有被明确建模成焦点系统可识别的滚动根

于是焦点引擎只能“猜”：

- 要不要滚 window
- 还是滚某个未知容器

这种猜测在桌面浏览器里可能凑巧可用，但在 Simulator / 真机这种 TV 运行环境里不够稳。

## 修复方案

### 1. 主内容区显式成为滚动容器

在应用壳中，把：

- `.tv-page-content`

标记为：

```html
data-focus-scroll-root="true"
```

并且在样式层明确它承担：

- `flex: 1`
- `min-height: 0`
- `overflow-y: auto`

这一步的意义不是“为了能滚”，而是：

- 明确告诉焦点引擎，TV 页面里真正应该滚的是谁

### 2. 焦点引擎优先滚最近滚动根

在焦点引擎里新增了“滚动根”概念：

1. 先从当前焦点元素向上找最近的 `[data-focus-scroll-root="true"]`
2. 如果找到了，就滚这个容器
3. 只有没有找到时，才回退到 `window`

这样修复后：

- 浏览器开发态仍可兼容普通窗口滚动
- Simulator / 真机则会稳定滚动主内容区

### 3. 舒适区与首行锚点都改成基于滚动根视口计算

这一步很关键。

如果只是把滚动命令从 `window` 改成容器，但舒适区仍然按整个窗口高度算，仍然会出现：

- 焦点判断边界不准
- section 首行回滚位置漂移

所以最终修复里，以下信息全部改成基于滚动根视口计算：

- 视口 frame
- 舒适区上下左右边界
- section 锚点回滚偏移

也就是说，焦点引擎现在不再关心“整个浏览器窗口多大”，而是关心：

- **当前真正负责滚动的 TV 内容区多大**

## 为什么这个问题容易误判

这次问题很容易被误判成下面几类：

1. “舒适区算法失效”
2. “section 首行保头部没生效”
3. “焦点切换成功但滚动命令没发出”

但真正情况是第四类：

4. **滚动命令发出了，只是发给了错误的滚动对象**

这类问题如果不先确认滚动容器，很容易一直在：

- 改 comfort zone 参数
- 改 top offset
- 改首行判断阈值

最后越改越偏，因为真正的问题不在参数，而在目标对象。

## 经验沉淀

后续继续维护 TV 焦点滚动时，必须记住：

1. 只要页面不是纯浏览器文档流，就不能默认 `window` 是滚动目标。
2. TV 应用必须显式建模“滚动根”，不要让焦点引擎靠猜。
3. 舒适区、section 锚点、可见性修正，必须基于同一个滚动根视口计算，不能一半按 window、一半按容器。
4. 当用户反馈“焦点能动但屏幕不动”时，优先检查滚动命令发给了谁，而不是先改导航算法。
5. 对 TV 页面来说，左导航、顶部状态栏、主内容滚动区最好从结构层就固定职责，否则焦点滚动会在不同运行环境下漂移。

## 后续约束

以后只要新增或重构以下内容，都必须同步检查滚动根是否仍然正确：

1. `AppShell` 布局
2. `.tv-page-content` 的 overflow / flex 结构
3. 新的沉浸页与非沉浸页切换逻辑
4. 任意新增的弹层、抽屉、局部滚动区

验证清单里应明确包含：

1. 焦点是否正确切换
2. 焦点离开当前可视区时，是否滚动了正确的容器
3. 首行回滚时，section 标题与说明是否一起回来
4. 浏览器开发态、Simulator、真机三者是否保持一致
