# 2026-03-22 webOS 焦点高亮缺失与旧包残留 Postmortem

## 结论

这次“电视和 Simulator 上遥控器移动时看不到任何焦点高亮”的问题，最终不是单一原因，而是两层问题叠在一起：

1. 之前把焦点高亮画在按钮自身样式上的方案，在 webOS 环境里表现不稳定
2. 真机安装阶段还叠加了一个更隐蔽的问题：电视长时间继续运行旧包，导致“模拟器有效、电视无效”的结论一度失真

最终稳定可用的方案是：

- 不再依赖按钮本体的 `:focus` / `box-shadow` 视觉
- 改为由焦点系统直接驱动一个独立的白框 overlay
- 真机部署时必须以电视文件系统里的实际入口 JS hash 为准，而不是只看 CLI 的 `Success`
- 当同版本号 IPK 出现覆盖异常时，直接提升 `appinfo.json.version`

## 复现现象

用户最初反馈是：

- 遥控器左右移动时，没有任何高亮或提示
- 首页按钮、视频卡片、播放器 OSD、菜单都一样
- 电视上和 Simulator 上都看不到预期焦点效果

在中途排查时还出现过一个误导性现象：

- Simulator 上有时能看到焦点框
- 电视上始终没有

这让问题一度看起来像：

- “真机 CSS 不兼容”

但后面证明并不完全如此。

## 已确认事实

### 1. 焦点系统本身一直在工作

通过在页面里增加焦点调试 HUD 和独立白框 overlay，可以确认：

- 当前焦点元素会随方向键正常移动
- `focusId / sectionId / pressed` 状态都能正确更新
- 问题不在于“焦点逻辑失效”

也就是说，之前用户“完全看不到高亮”，并不代表没有焦点，而是：

- 焦点逻辑和视觉渲染脱钩了

### 2. 按钮本体样式在 webOS 上不可靠

此前尝试过几类实现：

- `:focus-visible + box-shadow`
- `data-focus-active + box-shadow`
- `border + background + brightness`

这些方案在桌面浏览器里都合理，但在 webOS Simulator / 真机上表现都不稳定，用户肉眼看到的结果仍然是：

- 没有明显焦点

说明这类视觉如果直接挂在按钮本体上，会受到 webOS 浏览器实现细节影响。

### 3. 真机一度一直在跑旧包

这是这次排障里最关键的事实。

本地最新构建时，`build/webos/index.html` 的入口已经是：

- `index-legacy-Cg_6cjGV.js`

但直接读取电视文件系统时，电视上的实际入口长时间仍然是：

- `index-legacy-DZO80Qi0.js`

这说明：

- 即便 CLI 安装返回 `Success`
- 真机也可能继续跑旧前端产物

因此中途出现的“电视没效果、模拟器有效”，有一部分其实不是同一版代码的对比。

## 根因

这次问题的根因可以拆成两层：

### 1. 视觉实现根因

> 焦点高亮过度依赖按钮本体的 CSS 状态，在 webOS 环境里不够稳定，导致焦点逻辑正常但视觉反馈缺失。

### 2. 部署验证根因

> 真机部署阶段把 CLI 的安装成功误当成“电视已经运行新代码”，没有第一时间用电视文件系统里的入口 JS hash 交叉验证，导致旧包残留长时间混入判断过程。

## 修复方案

### 1. 使用独立焦点框 overlay

新增了一个独立的焦点覆盖层组件：

- `src/components/FocusOverlay.tsx`

它的职责是：

- 读取当前焦点元素
- 根据元素实时位置绘制一个固定定位的白框
- 按下确认键时同步显示 pressed 态

这样做的好处是：

- 焦点框不再依赖按钮本体的渲染表现
- 即便按钮自身样式在 webOS 上不稳定，焦点框依然可见

### 2. 保留调试 HUD，但默认关闭

为了提高后续排障效率，保留了：

- `debugFocus=1`

这个调试入口。

开启时会显示：

- 当前 `focusId`
- `section`
- `pressed` 状态
- 焦点矩形坐标

默认正式运行不会显示 HUD，只保留白框 overlay。

### 3. 焦点引擎补充显式状态

在焦点引擎里补充了：

- `data-focus-active`
- `data-focus-pressed`

这样 overlay 不需要依赖浏览器对 `:focus` / `:active` 的实现差异，而是直接读取应用自己的焦点状态。

### 4. 真机安装必须核对入口 hash

后续真机联调必须固定执行：

1. `package / reinstall`
2. 等待几秒
3. 直接读取电视上的 `index.html`
4. 对比 `index-legacy-*.js` hash 是否与本地 `build/webos/index.html` 一致

这一步必须成为真机判断前置条件，不能省略。

### 5. 同版本覆盖异常时直接升版本

这次已经证明：

- 同版本 `0.1.1` 重装时，电视仍可能保留旧入口 hash

把版本提升到：

- `0.1.2`

之后，再 `package + reinstall`，电视上的入口才真正切到最新构建。

## 为什么 overlay 方案更适合当前项目

当前项目的关键约束是：

- 遥控器 5-way 导航必须稳定
- 电视端必须一眼看清当前焦点
- 真机行为优先于桌面浏览器体验

基于这些约束，overlay 的优势非常明显：

1. 焦点可见性和组件自身样式解耦
2. 不需要每个按钮、卡片、OSD 分别调焦点 CSS
3. 对 webOS 真机和 Simulator 更容易保持一致
4. 后续如果需要更换样式，只改一处即可全局生效

## 验证结果

最终验证分两步完成：

### 1. Simulator

在 Simulator 中，独立白框 overlay 已可见，说明：

- 焦点状态
- overlay 坐标同步
- pressed 态

都已正常工作。

### 2. 真机

真机第一次看不到效果时，最终确认是旧包残留。

在版本提升到 `0.1.2` 并重新安装后，电视文件系统中的入口已切到：

- `index-legacy-Cg_6cjGV.js`

这与本地最新构建一致，证明白框 overlay 代码已真正上电视。

用户最终反馈为：

- “OK，我看到有白框了。”

## 教训

### 1. 电视端 UI 问题，不能只盯 CSS

当用户说“完全没有焦点效果”时，不能默认等价于：

- “焦点逻辑失效”

更高效的做法是第一时间把：

- 当前焦点元素
- 当前焦点矩形
- 当前 pressed 状态

直接可视化出来。

### 2. CLI 的 `Success` 不是最终真相

对于 webOS 真机：

- 安装成功
- 启动成功

都不等于电视一定运行了新包。

真正可信的依据是：

- 电视文件系统中的入口 JS hash

### 3. 真机关键联调时，不要依赖同版本覆盖

如果需要高频验证关键修复，尤其是 UI 或播放器这类“用户一眼就能感知”的问题：

- 优先 `remove/reinstall`
- 如仍异常，直接升版本号

这比反复猜“是不是没生效”更节省时间。

### 4. webOS 上优先选择更“笨但稳”的方案

在桌面浏览器里优雅的按钮焦点实现，到了 webOS 环境未必稳定。

当前阶段更应该优先：

- 简单
- 显眼
- 可验证
- 可全局复用

overlay 正好满足这几点。

## 后续建议

### 1. 把入口 hash 核对固化进真机流程

后续每次真机重装后，优先自动输出：

- 本地 `build/webos/index.html` 入口 hash
- 电视文件系统中的入口 hash

避免再次把旧包误判成新包。

### 2. 保留 `debugFocus` 能力

即使平时不用 HUD，也建议保留这条开关。

后续只要再遇到：

- 焦点不见了
- 焦点位置错了
- 按下态不一致

都可以快速打开排查，不需要重新临时加调试代码。

### 3. 后续只在 overlay 上继续打磨视觉

既然 overlay 已被验证为稳定方案，之后如果还要继续优化：

- 白框粗细
- 圆角跟随
- pressed 态
- 过渡速度

都优先继续在 overlay 上打磨，而不要再回到“每个按钮各自实现焦点视觉”的方向。
