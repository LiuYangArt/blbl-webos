# 2026-03-22 首页公共内容容错、webOS 滚动闪烁与推荐瀑布流 Postmortem

## 结论

这轮首页问题表面上看起来分散：

- 从播放器返回首页时偶发 `-352`
- 匿名态更容易复现，登录后明显好转
- 首页向下滚动时会整屏黑一下
- 推荐内容太少，继续加载时焦点又容易丢

但它们背后其实是三个不同层级的问题叠在了一起：

1. **数据层**：首页把多个公共接口当成整页强依赖，任一路被风控拦截都会拖死整个首页。
2. **渲染层**：首页 Hero 的大图、光斑和毛玻璃与整页滚动叠加，在 webOS 上容易触发全屏重绘闪烁。
3. **交互层**：推荐区仍然停留在“首屏少量卡片 + 手动扩展”的思路，没有真正按 TV 长列表去设计焦点和预加载。

最终有效的方案不是继续针对某一个报错打补丁，而是把首页分别按 **容错、减重、连续浏览** 三个方向重构。

## 用户可见问题

### 1. 首页从播放器返回后偶发直接报错

用户稳定复现路径是：

1. 进入 app
2. 打开一个视频
3. 返回首页

然后首页会出现：

- `首页推荐失败：风控校验失败（-352）`
- 或 `首页排行失败：风控校验失败（-352）`

并且用户补充了一个很关键的现象：

- **登录账号后再返回，问题明显减少甚至消失**

这说明问题不是我们本地页面状态乱了，而是 **匿名态公开接口更容易被 B 站风控拦截**。

### 2. 首页滚动比热门页更容易闪屏

用户又给出了一个很有价值的对照：

- 首页个性推荐往下走时，屏幕会偶发整页黑一下
- 热门页也有很多视频，但向下走时几乎不闪

这让我们能把排查方向从“焦点算法错误”转成“首页特有的渲染负担”。

### 3. 推荐加载后的焦点体验不稳定

后续继续做“加载更多推荐”后，用户又指出两个体验问题：

- 初始推荐条数仍然偏少
- 新内容加载完后焦点会飞掉

这表明推荐区不能再用“普通页面里临时扩一块”的心态做，而要按 TV 连续浏览内容流来设计：

- 预加载要提前发生
- 焦点要么保持当前位置，要么精确落到新增内容第一项
- 不能因为异步数据回来就让用户失去上下文

## 根因拆解

### 根因一：首页公共内容被设计成了整页强依赖

首页同时依赖：

- 推荐
- 热门
- 排行

这三条都属于“公共内容源”，理论上任何一条短暂失败都不应该让首页完全不可用。

但旧实现把它们放进一组并发请求里，只要其中一个抛错，整个首页就进入错误态。

这在匿名态尤其脆弱，因为 B 站部分公开接口偶发会返回 `-352`，而登录态 cookie 又会让风控表现不同。

### 根因二：首页 Hero 比热门页重得多

首页和热门页虽然都用了卡片网格，但首页额外多了：

- 顶部大封面图
- 叠加渐变遮罩
- 额外的发光 orb
- 内容区毛玻璃

这些视觉层在桌面浏览器里问题不大，但在 webOS 上和整页滚动叠加时很容易触发大面积重绘。

热门页没有这套顶部重层，所以成为了非常清晰的对照样本：

- **不是列表滚动本身会闪**
- **而是首页独有的重视觉层会放大滚动成本**

### 根因三：推荐区没有按 TV 连续内容流来设计

旧推荐区有两个典型问题：

1. 初始只展示少量内容，用户很快就触底。
2. 异步补内容时，没有明确规定“焦点加载完成后应该落在哪里”。

这导致两种糟糕体验：

- 用户会不断撞到底部再等待内容
- 内容回来后焦点丢失或跳转不可预测

TV 长列表更合适的模型应该是：

- 初次进入先把一屏以上内容准备好
- 剩余内容快到底时提前异步补下一批
- 手动重试这类显式操作再做焦点跳转
- 预加载本身不打断当前浏览位置

## 修复内容

### 1. 首页公共内容改成分区级降级

推荐、热门、排行都改成：

- 各自失败时保留本分区提示
- 不再拖死整个首页
- 只有三路公共内容全部不可用时，才显示整页错误

这样匿名态即使碰到 `-352`，首页也还能正常浏览其他区域。

### 2. 首页公共内容加入短 TTL 缓存

为推荐 / 热门 / 排行加入了：

- `3` 分钟内的新鲜缓存直读
- 最长 `30` 分钟的失败回退缓存

这让“从播放器返回首页”这种高频路径少打一轮请求，也降低了被风控时整页退化的概率。

### 3. Windows 下 `webos:verify-install` 改为可稳定校验

仓库脚本原本在 Windows 下调用 `ares-novacom --run "cat ..."` 时会被引号链路打坏。

最终改成：

- 在 Windows 捕获场景下走 `PowerShell`
- 保证 `npx -> node@16 -> ares-novacom --run` 的参数能完整透传

这样真机部署不再需要手工读取电视上的 `index.html` 才能确认是否装上新包。

### 4. webOS 上禁用焦点滚动的平滑动画

焦点引擎原本在滚动跟随时使用：

```ts
scrollIntoView({ behavior: 'smooth' })
```

在 webOS 上这会把滚动过程放大成整屏重绘，因此改成：

- webOS: `behavior: 'auto'`
- 其他环境：继续 `smooth`

### 5. 首页在 webOS 上切成轻量视觉模式

为首页单独做了 webOS 轻量化：

- 去掉 Hero orb
- 弱化 Hero 背景层
- 关闭首页内容区毛玻璃

后续又进一步直接移除了首页大 Hero，首页改成：

- 频道 tabs
- 推荐列表
- 其他频道内容

这进一步减少了首页滚动时的大层重绘。

### 6. 推荐区改成真正的连续加载流

当前推荐区的策略变成：

1. 首批先拉一整批推荐
2. 初次进入首页后，后台再异步补一批
3. 当剩余内容约 `6` 张时，自动预加载下一批
4. 如果自动预加载失败，只显示底部重试入口
5. 只有用户明确点“重试加载更多推荐”这种手动动作时，加载完成后才把焦点落到新批次第一张卡

这和 TV 端的浏览心智更接近：

- 正常浏览时尽量无感补内容
- 失败时保留可恢复入口
- 不让后台预加载突然把焦点抢走

## 经验沉淀

1. TV 首页的“公共内容源”不能做成整页硬依赖，任何单个公共接口失败都应该先局部降级。
2. webOS 上滚动性能问题往往不是列表本身，而是“重视觉层 + 滚动”叠加后的整页重绘。
3. 在 TV 长列表里，预加载比“触底后再加载”更重要；剩余量阈值应显式控制，而不是等用户撞到底。
4. 自动预加载和手动加载的焦点策略不能混为一谈：
   - 自动预加载：不打断当前焦点
   - 手动重试：可以在成功后把焦点带到新内容第一项
5. 真机部署脚本如果不能稳定校验入口 hash，会让问题定位成本成倍增加；部署链路可验证本身就是生产力功能。

## 后续约束

后续再做 TV 长列表或瀑布流时，默认遵守下面的规则：

1. 初始内容量至少覆盖一屏以上。
2. 在剩余内容耗尽前就触发预加载，而不是等用户撞到底。
3. 预加载失败只做局部提示和重试入口，不影响已拿到的内容。
4. 后台预加载不得主动抢焦点。
5. 任何真机部署改动都必须继续保证 `webos:verify-install` 可用，不允许回到“CLI 看着成功、电视其实还是旧包”的状态。
