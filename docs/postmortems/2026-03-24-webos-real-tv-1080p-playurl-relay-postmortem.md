# 2026-03-24 webOS 真机 1080P 排障收口与 playurl relay 转向 Postmortem

## 结论

这轮排障最后确认的结论很明确：

1. 问题不是 `LG C1` 没有 `1080P / 4K / HEVC / AV1` 播放能力。
2. 问题也不是“B 站接口完全不给我们 `1080P` 数据”。
3. 现在真正卡住的是：`webOS 真机 Web App` 这一层，没法像原生客户端或受控代理那样稳定控制 `playurl` 请求上下文。
4. 因此，继续只在前端里硬调 `fnval / platform / high_quality / UA / 回退顺序`，收益已经明显递减。
5. 下一阶段应该转向：`把拿 playurl 的动作放到一个最小 relay 服务里做，TV 端保留当前直连作为 fallback`。

一句话总结：

- `电视硬件能播高清`。
- `B 站有时也会给 1080 数据`。
- `但纯 webOS Web App 这一层，不能稳定拿到并跑通那条高清链路`。

## 背景

Issue `#9` 的目标，是解决“用户明明请求 `1080P`，真机却长期落到 `720P` 或根本起不来”的问题。

这轮工作前半段已经做过一轮修复：

1. 拆开 `请求画质 / 接口返回画质 / compatible 回退画质`
2. 修正 `DASH` 与 `compatible` 的高层排序
3. 保证 `Simulator` 和真机高层策略保持一致

这些改动有价值，它们解决了“语义污染”和“错误回退顺序”问题。

但在继续追真机 `1080P` 时，又暴露出一个更底层的新结论：

- **就算播放器内部逻辑已经理顺，真机端依然不一定能稳定拿到或播通真正可用的 `1080P` 播放地址。**

## 这轮排障里确认过的事实

### 1. 真机不是拿不到 `1080P` 元数据

针对测试视频 `BV1Sf4y1q7H9`，真机 telemetry 多次证明：

1. `requestedQuality = 80`
2. `returnedQuality = 80`
3. `videoStreamCount = 8`
4. `DASH manifest` 可以成功生成和加载

这说明至少在“接口返回的 DASH 元信息”这一层：

- 不是完全没有 `1080P`
- 也不是我们本地把 `1080P` 解析丢了

### 2. 真机 `1080 DASH` 的问题出在后续媒体分片

继续打点后，能稳定看到：

1. `manifest request` 能出去
2. `segment request` 也能出去
3. 但 `segment response` 经常等不到

这意味着：

- `1080 DASH` 不是一开始就失败
- 而是进入媒体分片阶段后卡住

也就是说，真机问题不只是“接口参数不对”，还涉及：

- 真机 WebApp 的媒体请求链路
- CDN 返回形态
- 或 B 站对这类请求上下文的处理差异

### 3. 真机稳定能播的，仍然是 `720 html5 / T_0_0`

这轮反复验证后，最稳定的成功路线仍然是：

1. `compatible`
2. `quality = 64`
3. `platform = html5`
4. `f = T_0_0`

而用户最在意的 `1080 compatible`，在真机上更常见的是：

1. `quality = 80`
2. `platform = pc`
3. `f = u_0_0`

这条路线在当前真机环境下并不稳定，甚至经常直接起不来。

### 4. 本机终端能拿到的结果，和真机 Web App 看到的不一致

本机终端直接请求 B 站 `playurl` 时，可以拿到：

1. `quality = 80`
2. `platform = html5`
3. `f = h_0_0`
4. `mid = 0`

说明 B 站服务端今天依然会给某些请求上下文返回更高质量的 html5 兼容流。

但同一时期在真机 App 中，即使参数看起来相同，telemetry 看到的仍然常常是：

1. `64 / html5 / T_0_0`
2. 或 `80 / pc / u_0_0`

这说明真正的问题不是“我们不会拼 URL”，而是：

- **真机 Web App 发出的请求，在 B 站眼里并不等于终端脚本发出的请求。**

## 这轮尝试过、但最终失败的方向

### 1. 单纯继续调 `fnval / platform / high_quality`

这条路已经做到比较极限：

1. `DASH` 请求能力位做过多轮尝试
2. `compatible` 也按不同参数组合拉过
3. `html5 + high_quality` 也单独测过

结论是：

- 这些参数仍然重要
- 但它们已经不足以单独解决真机高清问题

因为现在矛盾不再只是“参数不够激进”，而是“同样参数在不同运行上下文里结果不同”。

### 2. 在前端里做“匿名/访客请求”实验

这轮专门验证过两种做法：

1. `fetch(..., { credentials: 'omit' })`
2. `XMLHttpRequest + withCredentials = false`

目的很直接：

- 想看真机能不能像终端那样，拿到更像 `访客 / mid=0` 的 `html5 1080` 结果

最终结论：

- **在真机 webOS WebApp 里，这两种做法都没有稳定把请求真正“匿名化”。**

telemetry 里虽然能看到我们“尝试 omit”的意图，但结果仍然没有变成预期的那条高清 html5 链路。

这说明继续在纯前端里硬抠“匿名请求”成功率已经不高了。

### 3. 继续把真机伪装成 PC Chrome

这轮也确认了一件事：

- 改 `User-Agent` 不是根因级方案。

原因是 B 站看起来不只判断 `UA`，还会综合：

1. `Cookie`
2. 登录态
3. `Referer`
4. `Origin`
5. 请求参数组合
6. 后续媒体请求链路

所以：

- 你可以伪装成 PC Chrome
- 但服务端不一定就真的把你当成 PC Chrome

### 4. 把问题继续解释成“LG 文档没支持”

这条路也被排除了。

LG 官方文档和设备能力本身只能说明：

1. 电视支持高分辨率视频播放
2. 电视支持 `HEVC / AV1` 等解码能力

但它们不能保证：

- 第三方 Web App 一定能从某个站点稳定拿到对应的高清播放地址
- 也不能替代站点自己的鉴权和播放策略

## 参考项目给出的启发

### `JKVideo`

`JKVideo` 的 Web 侧很关键，因为它没有假装“浏览器前端能完全掌控这些请求”。

它直接用了本地 relay：

1. `http://localhost:3001/bilibili-api`
2. `http://localhost:3001/bilibili-passport`

这等于承认了一个现实：

- **Web 环境本身不适合完全掌控 `Cookie / Referer / Origin` 这类上下文。**

### `PiliPlus`

`PiliPlus` 更接近原生客户端路线。

它能比我们这个 webOS Web App 更自由地控制：

1. 请求头
2. 登录态
3. UA
4. 客户端参数

所以它的成功经验不能直接推导成：

- “我们这个纯 Web App 也应该只靠前端做到一样的事”

更准确的理解是：

- `原生客户端` 和 `Web App` 在这件事上的可控度不是一个量级。

## 为什么现在要转向 relay

到这里，问题已经收敛得比较清楚：

1. 真机硬件和系统播放器能力不是主要瓶颈
2. 当前前端逻辑里的明显误判已经修过一轮
3. 纯前端继续强拧请求上下文，收益越来越低
4. 参考项目里，Web 方案本身也在用 relay

所以最合理的下一步不是继续在前端里追加更多实验开关，而是：

- **把 `playurl` 这一步单独挪到受控服务端去做。**

而且这一步不需要做成大而全的后端：

- 只代理 `playurl`
- 不代理媒体文件
- 电视仍然直连 B 站 CDN 播放

这样就能最小化改造成本，同时最大化验证根因。

## 本次收口时做的代码处理

为了避免把排障期的实验代码长期留在主线里，这次收口把两类临时改动清掉了：

1. `credentials: omit / XHR withCredentials=false` 这套匿名请求实验代码
2. 仅用于追 Shaka 请求流水的临时 `network` 调试事件

保留下来的，是仍然有长期价值的部分：

1. `returnedQuality / compatibleQuality` 语义拆分
2. 真机 `DASH / compatible` 排序与回退稳定性改进
3. 请求追踪里对返回质量、host、URL hint 的记录

这意味着主线代码会更干净：

- 不再带着已经证伪的实验路径
- 但仍保留后续定位问题真正有用的诊断信息

## 经验总结

以后再碰类似“电视能播，但这个 Web App 拿不到高清”的问题，可以直接先套用这几条经验：

1. 先分清楚是“硬件能力问题”，还是“站点返回给这个 WebApp 的播放地址问题”。
2. 不要把“终端脚本能拿到高清地址”直接等同于“真机 Web App 也能拿到同样结果”。
3. 单纯改 `UA` 通常不够，真正关键的是整套请求上下文。
4. 当前端已经无法稳定控制请求上下文时，不要继续在前端里无限加实验开关，应尽快转向 relay 或原生层能力。
5. relay 的最小闭环不是“代理整个视频流”，而是“只代理拿播放地址这一步”。
6. 电视端一定要保留 fallback，避免 relay 不可用时把现有 `720P` 播放能力也弄丢。

## 下一步

下一阶段建议按下面顺序推进：

1. 先实现最小 `playurl relay`
2. TV 端接入“relay 优先、直连 fallback”
3. 在真机上验证：同一视频是否能稳定拿到更高质量的 `compatible` 或 `DASH`
4. relay 跑通后，再讨论 `HEVC / AV1` 优先和用户默认编码持久化
