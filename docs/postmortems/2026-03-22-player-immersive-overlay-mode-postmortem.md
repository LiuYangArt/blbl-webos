# 2026-03-22 播放器沉浸式全屏与横条 Overlay Postmortem

## 结论

这次播放器重构暴露出来的核心问题，不是某一个按钮没接上，而是播放器仍然带着“普通页面”的结构惯性：

- 还有应用壳留下来的左侧导航和顶部状态条
- 相关推荐仍然按页面内容块来思考
- OSD、推荐区、分P 区各自用独立布尔量管理

这些做法在浏览器页面里勉强可用，但在电视播放器场景里会直接破坏沉浸感和遥控器交互预期。

最终有效的修复，不是继续给旧页面结构打补丁，而是把播放器明确改成：

- **背景永远是固定的全屏视频**
- **前景只允许出现有限的 overlay 层**
- **所有返回行为都走统一的 overlay 优先关闭链路**

## 用户可见问题

本轮用户连续指出了几类典型症状：

1. 进入播放器后仍然能看到左侧菜单栏和顶部条，不是真正的电视全屏。
2. 推荐视频区虽然能左右选中，但按下键时不能“翻到下一屏 6 个”。
3. 播放器仍然残留详情页式思路，会先看到封面或初始页，而不是直接开始播。
4. 推荐区弹出后如果仍依赖页面滚动，会让背景视频跟着布局流被挤走。

这些问题表面上分散，实际都指向同一个根因：**播放器没有被当成一个独立的沉浸式状态机来设计。**

## 根因拆解

### 根因一：播放器不应该复用普通内容页布局

普通页面允许：

- 页面外边距
- 卡片圆角容器
- 顶部状态条
- 左侧导航长期存在

但播放器是 TV 端最高优先级场景，进入后应该立即把空间让给视频本身。  
只要还沿用普通内容页壳，用户就会觉得“视频只是页面里的一个模块”，而不是“我已经进入播放器模式”。

### 根因二：推荐区和分P 区不能再做成页面内内容块

推荐视频和分P/选集都属于“播放过程中的临时浏览层”，不是新的主页面正文。

如果把它们继续做成页面里的普通 section，就会天然产生两个问题：

1. 背景视频会跟着页面布局一起被推挤或滚动。
2. 用户按上下键时，焦点系统会尝试做普通页面导航，而不是“翻屏”。

TV 播放器里更自然的模型其实是：

- 视频固定在底层
- 推荐/分P 区固定在下半屏
- 左右在当前屏内选
- 上下负责切上一屏/下一屏

### 根因三：多个布尔量会让返回链路和显示链路失控

旧思路里如果同时维护：

- `isSettingsOpen`
- `isRecommendationsOpen`
- `isEpisodesOpen`
- `isChromeVisible`

很容易出现互斥关系不清楚的问题，例如：

- 推荐区开着时普通 OSD 还在
- 返回键不知道先关谁
- 某个 overlay 关掉后焦点回不到正确按钮

这次收敛成统一的 `overlayMode` 之后，逻辑才真正稳定下来：

- `none`
- `settings`
- `recommendations`
- `episodes`

这样可以直接派生：

- 是否显示普通 OSD
- 当前应该 capture 哪个 section
- 返回键下一步应该关闭哪个层

## 修复内容

### 1. 播放器切到沉浸式全屏壳

播放器路由单独启用 `immersive` 的 `AppShell`，进入后不再渲染：

- 左侧导航
- 顶部状态栏
- 常规内容区 padding

并在样式层把播放器改成真正铺满视口的全屏视频层。

### 2. 普通 OSD 默认隐藏，只在遥控器输入后出现

进入播放器时直接开始播，不再显示详情式初始界面。  
当普通 OSD 隐藏时，首次方向键或确认键只负责“唤起 OSD 并把焦点带到控制条”，不直接触发原动作。

同时只在“正在播放 + 没有 overlay 打开”时启用 3 秒自动隐藏。

### 3. 推荐视频与分P 改成底部固定横条 overlay

这两个区域都改成固定在底部半屏的横条层：

- 背景视频不移动
- 页面不滚动
- 一屏固定 6 项
- 左右切当前屏
- 上下切屏

这比“让页面继续往下长”更符合电视端的使用预期。

### 4. 返回键链路统一收口

播放器现在统一遵循：

1. 推荐横条 / 分P 横条 / 设置抽屉先关闭
2. 只有普通 OSD 时先隐藏 OSD
3. 都没有时再退出播放器

这个顺序和电视端用户的心智模型一致，也更方便后面继续叠加新 overlay。

## 经验沉淀

1. TV 播放器必须被当成“模式切换”，而不是“页面的一部分放大版”。
2. 播放器里的临时浏览内容，优先考虑 fixed overlay，而不是页面内 section 继续往下堆。
3. 对电视横条内容区，`左右选项 + 上下翻屏` 通常比“在同一布局里继续做空间导航”更稳定。
4. 只要有多个 overlay，必须尽早收敛成统一状态机，否则返回链路和焦点恢复一定会变乱。
5. 真正的“全屏”不只是视频元素 `width: 100%`，而是整条应用壳都要进入沉浸式模式。

## 后续约束

后续播放器再加新面板时，默认先问自己三个问题：

1. 它是不是一个临时 overlay，而不是新页面正文？
2. 它打开后背景视频是否仍然固定在原地？
3. 它是否已经被纳入统一的返回链路和焦点恢复链路？

如果这三个问题里有任一答案是否定的，就说明实现方向大概率又开始偏回“普通页面思路”了。
