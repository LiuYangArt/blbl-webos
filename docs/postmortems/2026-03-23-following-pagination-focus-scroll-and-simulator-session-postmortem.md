# 2026-03-23 关注分页、焦点滚动与 Simulator 旧会话误判 Postmortem

## 结论

这次问题不是单点故障，而是三类问题叠在一起：

1. 关注页分页接口和通用分页协议没有真正对齐
2. 焦点滚动修复虽然方向对了，但触发时机仍然依赖了不稳定的浏览器事件链路
3. Simulator 本地验证时还混入了旧窗口、旧 DevTools、旧代理会话，制造了“像旧包”的假象

表面上用户看到的是：

- 关注页只能加载一次，继续往下没有新内容
- 焦点有时能移动，但页面不跟着滚
- 删掉的调试面板偶尔又重新出现

真正收敛后的结论是：

- **列表页的无限加载必须走统一分页协议，而不是页面自己猜接口语义**
- **焦点移动后的滚动校正必须在聚焦成功后主动执行，不能只等 `focusin`**
- **Simulator 重启前必须先清旧实例，否则很容易把旧会话误判成新构建**

## 用户可见现象

这轮用户反馈实际分成了四段：

1. 首页个性推荐已经能继续加载，但关注页下翻后没有继续补内容
2. 历史记录在已登录状态下仍显示不可用
3. 焦点在部分场景下会移出屏幕，屏幕没有稳定跟着滚
4. 即使代码已经改掉了调试面板，重新开 Simulator 仍偶尔看到顶部面板残留

这些现象如果分开看，很容易被误判成：

- 关注接口单独坏了
- 历史接口没权限
- 焦点引擎回退了
- 构建目录没刷新

但最后确认并不是四个互不相关的问题。

## 根因一：关注页接口返回的是 `offset`，通用 hook 却按 `cursor` 协议前进

问题位置：

- `src/services/api/bilibili.ts`
- `src/features/shared/usePagedCollection.ts`
- `src/features/following/FollowingPage.tsx`

关注动态接口 `/x/polymer/web-dynamic/v1/feed/all` 返回的是下一页 `offset`。  
但我们新抽出的通用分页 hook `usePagedCollection` 统一消费的是：

- `items`
- `hasMore`
- `cursor`

如果数据适配层没有把 `offset` 映射成 `cursor`，页面表面上虽然也能拿到第一页数据，但后续 `loadMore()` 实际拿不到下一页游标，结果就会停在“只成功一次”。

这次说明一个很重要的约束：

- **页面不应该直接理解每个接口自己的分页字段名**
- **所有分页差异都应该在 API 适配层里收口成统一协议**

否则页面一旦开始自己认 `offset`、`next`、`cursor`、`page`，统一模板就会名义存在、实际分裂。

## 根因二：cursor 流里“这次没新增卡片”不等于真的到底

问题位置：

- `src/features/shared/usePagedCollection.ts`

通用 hook 初版有一个对传统页码流成立、但对 cursor 流不一定成立的假设：

- 如果本次请求没有合并出新卡片，就把 `hasMore` 直接打成 `false`

这个判断对热门页、搜索页这种普通翻页数据通常问题不大。  
但关注动态流里存在一种真实场景：

1. 服务端返回一页动态
2. 这页里很多项被过滤掉，因为不是可直达的视频卡片
3. 本轮映射后可能新增 `0` 条视频
4. 但服务端的游标其实还在继续推进，后面仍然可能有可展示的视频

也就是说：

- **集合没有增长**
- 不等于
- **数据流已经走到结尾**

这次最终修正成：

- 只要 `cursor` 还在推进，就视为 collection 还在继续前进

这也是通用分页层必须承担的职责，而不是让关注页单独打补丁。

## 根因三：焦点滚动不能只依赖 `focusin`

问题位置：

- `src/platform/focus/engine.ts`

之前我们已经把“滚动哪个容器”这件事修到比较正确了：

- 主内容区显式作为滚动根
- 舒适区计算也基于滚动根

但后续又出现“有时会滚，有时不会滚”的不稳定现象，根因在于：

- 焦点滚动校正之前主要依赖 `focusin` 事件触发
- 在 Simulator 里通过程序化 `focus()` 切换元素时，这条事件链并不总是稳定

因此最终修复不再把“滚动校正”绑定在浏览器是否完整派发 `focusin` 上，而是改成：

- **`focusElement()` 只要成功完成聚焦，就立即主动调用 `ensureFocusedElementComfort(element)`**

这一步把焦点系统从“依赖环境事件时序”改成了“依赖自己的聚焦成功事实”，因此比原来稳定很多。

## 根因四：Simulator 旧会话会制造“像旧包”的假性回归

问题位置：

- `scripts/webos-cli.mjs`
- `.agents/skills/lg-webos-deploy/SKILL.md`
- `AGENTS.md`

这轮还有一个非常容易误判的问题：

- 代码里已经移除了关注调试面板
- 构建目录里的入口脚本 hash 也是新的
- 但重新打开 Simulator，顶部却仍然像跑着旧 UI

最后排查下来，本地构建目录其实已经是新的。问题更像是：

1. 旧 Simulator 进程没有真正退出
2. 旧 DevTools 子窗口仍然挂在旧会话上
3. 旧 `simulator-media-proxy` 还留着

结果就是用户看到的不是“当前构建”，而是混着旧会话痕迹的运行现场。

因此这次把规则明确固化为：

- 启动新 Simulator 之前，必须先清掉旧 Simulator 进程树
- 同时清理旧 `simulator-media-proxy`
- 等旧会话退出后，再拉起新实例

这不是体验优化，而是**调试正确性约束**。

## 修复方案

### 1. 分页统一收口到通用模板与通用 hook

当前视频列表页统一收敛到：

- 通用列表模板：`src/components/VideoGridSection.tsx`
- 通用分页 hook：`src/features/shared/usePagedCollection.ts`

页面只负责两件事：

1. 把各自接口适配成统一的 `items / hasMore / cursor`
2. 把数据映射成统一视频卡片

这样关注页、历史页、热门页、搜索页、收藏页、订阅页都不再各自重复造“滚到底后怎么补下一页”。

### 2. 关注页接口适配层显式输出 `cursor`

`fetchFollowingFeedPage()` 现在不再把服务端原始字段直接散给页面，而是统一返回：

- `items`
- `hasMore`
- `cursor`

并且在遇到“当前 offset 这一页没有可展示视频，但游标还在推进”的情况时继续向后探测，直到：

- 命中可展示视频
- 或者真的没有下一页

### 3. 历史记录改为 cursor 接口

历史页这次也顺手从旧读取方式切到：

- `/x/web-interface/history/cursor`

并统一通过 `max/view_at` 游标继续向后加载。  
这样历史页不再是“登录页风格的独立特例”，而是和其他列表页走同一套通用分页逻辑。

### 4. 聚焦成功后立即做滚动舒适区校正

焦点引擎现在在 `focusElement()` 内部直接保证：

1. 程序化调用 `focus({ preventScroll: true })`
2. 聚焦成功后立刻执行 `ensureFocusedElementComfort(element)`

因此无论浏览器后续是否稳定派发 `focusin`，只要聚焦切换成功，滚动校正就会发生。

### 5. Simulator 启动前先清旧会话

`scripts/webos-cli.mjs` 的 `simulator` 分支现在会先：

1. 解析当前版本对应的 Simulator 可执行文件
2. 清理旧 Simulator 进程树
3. 清理旧 `simulator-media-proxy`
4. 等待旧会话退出
5. 再启动新的 proxy 和新的 Simulator

同时，这条规则也已经写入项目级 skill 和仓库规则，避免以后再次只留在口头记忆里。

## 为什么这个问题容易误判

这次最容易踩坑的地方有三个：

1. **把“映射后没新增卡片”误判成“服务端没有下一页”**
2. **把“焦点能动但页面不滚”误判成“舒适区参数不对”**
3. **把“Simulator 看起来像旧 UI”误判成“构建目录没更新”**

这三个误判有共同点：

- 表象都发生在页面层
- 但真正根因分别在分页协议、焦点引擎触发时机、模拟器会话管理

如果不先把层级划分清楚，就会一直在页面上打补丁，最后越修越散。

## 经验沉淀

后续继续做 TV 列表页和焦点流时，必须保留下面这些约束：

1. 任何视频列表页都优先复用 `VideoGridSection`，不要再页面内各自实现一套“滚动到底再补数据”。
2. 分页差异一律在 API 适配层收口成统一协议，不允许页面自己理解 `offset`、`cursor`、`max/view_at`。
3. cursor 流里“这页无新增可见项”不等于“真正到底”，要看游标是否继续推进。
4. 焦点滚动校正必须绑定在“聚焦成功”这个事实本身，而不是只依赖浏览器事件链路。
5. Simulator 只要用于重新验证最新代码，就必须先清旧会话；否则看到的现象不具备可信度。

## 后续约束

以后只要继续改下面这些区域，都要一并复查：

1. `VideoGridSection` 的加载更多触发时机
2. `usePagedCollection` 对 cursor / page 两类数据流的兼容性
3. `focusElement()` 与 `ensureFocusedElementComfort()` 的调用关系
4. `scripts/webos-cli.mjs` 的 simulator 生命周期管理

验证清单里也应该明确包含：

1. 关注页能否连续加载多页
2. 历史页在已登录状态下是否能拿到云端数据
3. 焦点进入屏幕下方卡片时，页面是否稳定跟着滚
4. 重新启动 Simulator 后，是否还会出现旧调试面板或旧窗口残留
