# 2026-03-21 webOS Simulator DASH + Shaka 播放不稳定 Postmortem

## 结论

本轮 `webOS TV 6.0 Simulator` 联调，不能再把“某一个视频在某一次尝试中成功起播”当成方案已成立。

截至 2026-03-21 晚间，这条 `B 站 playurl -> 前端生成 MPD -> Shaka + MSE -> <video>` 链路的真实状态应定性为：

- **实验性原型**
- **样本覆盖严重不足**
- **在 Simulator 上远未达到可用**

更直接地说：

1. 当前方案没有证明“绝大多数公开视频都能稳定播放”
2. 只能证明“少量样本、少数时刻可能起播”
3. 在用户实际回归里，依然表现为：
   - 绝大部分视频播不了
   - 控制台持续出现 `403`
   - 播放页停在“正在切换备选线路”
   - 工程判断和用户真实体验明显脱节

这不是一个“再补一点小兼容”就能宣布成功的状态。

## 用户可见症状

本轮用户反馈与截图确认了这些现象：

1. **手动启动 Simulator 后，绝大部分视频仍然播不了**
2. 播放页会显示：
   - 标题
   - 当前画质
   - 当前 codec
   - `DASH`
   - `当前地址不可用，正在切换备选线路 #2`
3. 控制台持续出现：
   - `HEAD blob:file://... ERR_METHOD_NOT_SUPPORTED`
   - `GET https://upos-...bilivideo... 403 (Forbidden)`
   - `Payload length does not match range requested bytes`
4. 之前由脚本触发的 Simulator 启动，用户多次观察到白屏
5. 当前播放器页**没有直接显示 BV 号 / CID**，导致用户无法快速反馈“是哪一条视频失败”

这意味着当前链路在“可观测性”和“可播放性”两个维度都不合格。

## 已确认事实

### 1. `blob HEAD` 报错不是主问题

控制台里的：

- `HEAD blob:file://... ERR_METHOD_NOT_SUPPORTED`

更像是 Simulator / 浏览器对 `blob` 清单做探测时的附带噪音。

它会污染判断，但目前没有证据说明它是“绝大多数视频无法播放”的主根因。

### 2. 真正拦住播放的是大量 `403`

用户截图里，真正导致失败的是：

- 多个 bilivideo CDN 地址连续 `403`
- 播放器不断切到下一条候选
- 最终仍无法起播

这说明当前主问题不在“能不能生成 MPD”，而在：

- **生成后的媒体请求是否真的能被当前环境稳定接受**

### 3. “有些视频能播”不能代表路线已稳定

这轮曾出现过 Simulator 某些视频可播的现象，因此一度容易得出“主路线已成立”的乐观判断。

但用户后续大样本回归直接推翻了这个结论：

- 单条样本成功，不代表总体成功
- 一次成功，不代表稳定成功
- 只能把它当作“局部可行信号”，不能当作“方案闭环完成”

### 4. 当前播放器缺少最基础的排障信息

本轮暴露出一个很实际的问题：

- 用户想反馈失败视频
- 但页面上没有地方直接看到 `BV 号`

结果就是：

- 用户难以把“具体哪条视频失败”准确传回来
- 工程侧难以做定点复现

这属于明显的可观测性设计缺失。

## 目前最可信的判断

基于本轮现象，当前最可信的判断不是“codec 选择逻辑还有一点小问题”，而是下面这些更底层的问题至少还有一个没有解决：

1. **Simulator 环境下，Shaka / MSE 拉取 bilivideo 分片时的网络行为与预期不一致**
2. **B 站媒体分片访问对当前环境的请求头、来源或探测方式更敏感**
3. **现有候选线路切换还不够，仍然没有覆盖真正稳定的请求模式**
4. **当前 `file://` 运行环境与真实安装态 / 真机态的表现可能存在重要差异**

其中第 4 点目前仍然只是推断，不应写成既成事实。

## 本轮误判

### 误判一：过早把“个别样本成功”当作“方案已跑通”

这轮最核心的误判，是在样本量明显不够的时候，过早把方案状态提升成“基本可用”。

正确做法应该是：

1. 至少验证一批不同类型视频
2. 记录成功率
3. 记录失败是否集中在某些 codec / 某些清晰度 / 某些 CDN
4. 成功率足够高后，才允许把方案定性为“成立”

### 误判二：把大量精力放在 codec 选择，而不是整体请求链路

本轮虽然在：

- `AVC / HEVC / AV1`
- 实际返回 codec
- 接口宣称 codec
- 音视频候选排序

这些层面做了不少改动，但用户最终反馈说明：

- **主矛盾仍然是请求链路整体不稳**

如果请求链路本身大面积 `403`，那么 codec 选择再精细，也只能在失败路径里更精细地失败。

### 误判三：缺少“先补可观测性，再继续调试”的纪律

播放器在没有展示：

- `BV 号`
- `CID`
- 当前候选 URL host
- 当前候选序号 / 总数

这些最基本信息之前，就继续大量排障，导致沟通成本高、复现效率低。

这属于流程顺序错误。

## 经验沉淀

后续继续处理 webOS 播放问题时，必须遵守下面这些规则：

1. **禁止再用单条成功样本宣布方案成功**
2. **先补观测信息，再继续深入调试**
3. **如果用户无法一眼看到 BV 号，这就是产品和调试设计缺陷**
4. **Simulator 成功只能算辅助信号，不能替代样本化验证**
5. **面对持续 `403` 时，优先怀疑请求链路整体，而不是继续微调 codec 排序**

## 后续约束

在当前仓库里，后续任何人继续推进播放器之前，先满足这些前置条件：

1. 播放页必须直接展示：
   - `BV 号`
   - `CID`
   - 当前候选线路索引
   - 当前候选 host
2. 需要有一份最小样本表，至少记录：
   - 能播的视频
   - 不能播的视频
   - codec
   - 清晰度
   - 失败现象
3. 在没有拿到“多数公开视频可播”的证据前，不再把当前路线写成“已验证完成”
4. Simulator 脚本启动白屏问题要单独记录，不能和播放失败混成同一个问题

## 直接行动项

1. 把本次路线状态从“已跑通”降级回“实验性原型”。
2. 后续先补播放器页的 BV / CID / 当前候选 host 等调试信息。
3. 对 Simulator 白屏与播放器 `403` 失败分开建问题，不再混排。
4. 如果后续继续做播放器方案判断，必须先用一批样本统计成功率，再决定是否继续沿这条路线投入。
