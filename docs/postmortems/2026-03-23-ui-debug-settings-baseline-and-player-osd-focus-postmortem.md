# 2026-03-23 UI Debug 设置基线收口与播放器 OSD 自动抢焦点 Postmortem

## 结论

这次其实收敛了两条很关键、而且以后会反复踩到的规则：

1. `UI Debug` 只能展示“真实可复用的 UI 样式来源”，不能把所有业务内容消费者都摆上去。
2. 播放器是沉浸式场景，允许在一段时间内“没有任何真实焦点”；不能沿用普通内容页那套“只要长出新可聚焦元素就自动补焦点”的机制。

最终有效的修复不是继续调按钮顺序或焦点映射，而是分别把这两层规则明确下来：

- `UI Debug` 里只保留一个“设置抽屉基线”样本，不再单独陈列字幕设置内容。
- `App` 级内容区自动补焦点策略对播放器路由禁用，避免 OSD 一出现就把焦点塞到底部按钮区。

## 用户可见现象

用户连续指出了两类现象：

1. `UI Debug` 里同时出现“播放设置”和“CC 字幕设置”两块，看起来像是两套抽屉样式。
2. 播放器进入后按左右键，会先正常快进/后退一下，但随后焦点立刻掉到底部 OSD 按钮上，继续按左右键就变成切按钮，而不是继续 seek。

这两个问题表面上分别属于：

- 一个是 Debug 展示语义问题
- 一个是播放器遥控器问题

但它们背后的共同点是：**代码里某些“通用规则”越界到了不该管的地方。**

## 根因一：UI Debug 把“样式来源”和“业务消费者”混在了一起

问题位置：

- `src/features/debug/UiDebugPage.tsx`

用户指出的关键判断是对的：

- 字幕设置并没有引入新的 UI 原语
- 它只是复用了设置类抽屉已有的外壳、section 节奏、chip 按钮与焦点规则

如果 `UI Debug` 里仍把字幕设置单独摆出来，就会制造一种错误暗示：

- 好像这里有两套需要并列维护的抽屉样式

但真实情况不是这样。

`UI Debug` 真正应该回答的是：

- “这套样式来源是什么？”

而不是：

- “有哪些页面内容正在使用它？”

因此这次需要明确区分：

1. **样式来源**：应该展示
2. **内容消费者**：只有在它带来新的 UI 原语时才值得展示

当前字幕设置不属于第二种情况，所以应该删掉。

## 根因二：播放器不应该继承普通页面的 Mutation 自动补焦点

问题位置：

- `src/App.tsx`
- `src/app/focusPolicy.ts`

之前应用壳里有一条很合理的普通页面规则：

- 如果进入页面时暂时还没有内容焦点，就监听内容区 DOM 变化
- 一旦后续长出可聚焦元素，就自动 `focusFirst()`

这对普通页面很好用，比如：

- 列表异步加载后补焦点
- 首屏 skeleton 替换成真实卡片后补焦点

但播放器不是普通页面。

播放器里恰好存在一种合法状态：

- 页面已经进入播放态
- OSD 可能暂时隐藏
- 焦点可以不存在
- 左右键默认应该直接控制 seek

这时一旦第一次左右键把 OSD 显示出来，底部控制条也会一起出现在 DOM 里。  
旧的 MutationObserver 会立即把焦点自动补到底部 OSD 默认按钮，于是用户看到的行为就变成：

1. 左右键先触发一次 seek
2. OSD 出现
3. 焦点被 `App` 层自动塞到底部按钮
4. 后续左右键开始在按钮间移动

所以错的不是播放器内部遥控器模式判断，而是 `App` 层那条“通用自动补焦点规则”越界接管了沉浸式播放器。

## 根因三：程序化 pause 会伪装成“用户暂停”

问题位置：

- `src/features/player/PlayerPage.tsx`
- `src/features/player/playerChromePause.ts`

这次顺带还补掉了另一个容易混淆的问题：

- 播放器重置媒体源时会触发一次程序化 `pause`
- 如果把它误判成用户暂停，就会提前拉起 OSD

这会让“刚进入播放器就看到 OSD / 焦点异常”的问题更难判断。

因此这次也把这条规则沉淀下来：

- **程序化 pause 不应触发 OSD 展开**
- **只有真正的播放态 pause，才应该显示 OSD**

## 修复内容

### 1. UI Debug 只保留一个“设置抽屉基线”样本

本次删除了：

- `CC 字幕设置抽屉片段` 展示块

并把播放器模块说明改成：

- 重点看 `OSD`
- 重点看 `设置抽屉基线`

这样 `UI Debug` 现在表达的是：

- 这是设置类抽屉的统一视觉来源
- 真实字幕设置页是它的消费者，而不是另一套样式

### 2. 把播放器从全局 Mutation 自动补焦点里排除

新增：

- `src/app/focusPolicy.ts`
- `src/app/focusPolicy.test.ts`

现在应用壳会显式判断：

- 普通页面：允许在后续 DOM 变更时自动补内容焦点
- 播放器页面：禁止这条自动补焦点规则

这意味着播放器进入后：

- 可以暂时无焦点
- 左右键继续负责 seek
- 只有按上/下时才进入 OSD 控件区

### 3. 给程序化 pause 加抑制逻辑

新增：

- `src/features/player/playerChromePause.ts`
- `src/features/player/playerChromePause.test.ts`

播放器在重置媒体源时，会先标记“下一次 pause 属于程序化 pause”，从而避免误拉起 OSD。

## 经验沉淀

### 关于 UI Debug

以后往 `UI Debug` 补内容时，先问三个问题：

1. 这是不是一个真实可复用的 UI 原语或样式来源？
2. 它是不是给业务页面提供了新的视觉语义？
3. 如果把它放进来，会不会把“消费者”误导成“新的来源”？

只要第 3 个问题答案是“会”，就不应该展示。

### 关于播放器焦点

以后处理播放器遥控器时，先问三个问题：

1. 当前问题到底是播放器内部逻辑导致的，还是 `App` 层通用焦点规则越界接管？
2. 播放器当前是不是允许“暂时无焦点”？
3. 新出现的 OSD / overlay 元素，是否会触发全局自动补焦点副作用？

如果没有先回答这三个问题，就很容易继续在播放器组件内部误修。

## 后续固定检查项

以后只要改 `UI Debug`，必须同时检查：

1. 展示的是不是“样式来源”而不是“业务消费者”
2. 有没有把业务页面复用关系错误表达成多套平级样式
3. 新增展示块是否真的对应新的 UI 原语

以后只要改播放器 OSD，必须同时检查：

1. 程序化 `pause` 会不会误触发 OSD
2. `App` 级自动补焦点会不会把焦点抢进 OSD
3. 左右键默认 seek、上下键进入 OSD 的规则是否仍然成立
