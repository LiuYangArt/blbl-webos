# 2026-03-23 播放器 effect 依赖扩散导致重复重载与静音错觉 Postmortem

## 结论

这次回归的直接触发条件是：

1. 播放器新增了“当前 P 播完自动播放下一 P”能力
2. 自动续播逻辑直接写在播放器主加载 effect 的闭包里
3. 为了访问下一 P 与路由跳转回调，把 `currentEpisodeIndex`、`episodeEntries`、`onOpenPlayer` 一并带进了该 effect 的依赖数组

结果是播放器不再只在“播放源切换”时重建，而是会在上层页面重新 render、路由回调引用变化或分 P 数据联动时也执行 teardown / re-init。

在 Simulator 上的用户可见表现是：

1. 视频总在前几秒反复重播
2. 看起来像一直起播失败
3. 声音有时也会一起消失，容易被误判成音轨或代理问题

这次真正的根因不是字幕、不是 OSD、也不是 CSS；是**播放器主 effect 的职责边界被自动续播逻辑污染了**。

## 用户可见现象

用户反馈主要有两类：

1. 分 P 面板里卡片下方仍有大量空隙，说明文案也显得冗余
2. 视频播放有明显问题，会在前几秒循环，并且没有声音

其中第二条很容易把排查带偏，因为“无声音”通常会让人先怀疑：

1. DASH manifest 音轨拼接有误
2. Simulator 代理缺请求头，媒体流异常
3. 浏览器 / Simulator 的静音状态被改了

但这次这些都不是第一根因。真正先坏掉的是播放器生命周期。

## 根因分析

问题位置：

- `src/features/player/PlayerPage.tsx`

核心问题：

1. 播放器主 effect 负责创建 `<video>` 监听、装载媒体源、销毁旧实例
2. 新增 `handleEnded` 时，直接引用了 `currentEpisodeIndex`、`episodeEntries`、`onOpenPlayer`
3. React 为了保证闭包拿到最新值，只能把这些变量加入 effect 依赖
4. 一旦父层 render 产生新的 `onOpenPlayer` 引用，或相关派生值变化，播放器主 effect 就会重新执行
5. effect cleanup 会 `pause()`、移除事件、重新 `load()`，导致当前播放在刚起播后又被打断

因此表面上像“前几秒循环”，本质上是：

- 播放器实例被重复销毁并重新创建

“无声音”之所以会同时出现，是因为媒体还没稳定播放到正常状态就被重建，最终看上去像音轨缺失或代理失效。

## 修复方案

### 1. 把自动续播数据从主 effect 依赖链剥离

修复原则：

1. 播放器主 effect 只依赖真正影响媒体加载的状态
2. 自动续播所需的“下一 P 跳转目标”通过 `ref` 保存最新值
3. 路由跳转回调 `onOpenPlayer` 也通过 `ref` 持有最新引用

这样 `video ended` 事件仍然能拿到最新的下一 P 信息，但不会再迫使播放器主 effect 跟着重建。

### 2. 保持 `handleEnded` 只做事件时机处理

`handleEnded` 现在只负责：

1. 打开 chrome
2. 清空续播进度
3. 读取 `nextEpisodeRef`
4. 调用 `onOpenPlayerRef.current(...)`

它不再直接把“分 P 派生状态”和“路由回调”耦合进播放器加载生命周期。

### 3. 顺手收紧分 P 抽屉密度

这次用户同时指出分 P 面板视觉上过松，所以一并做了两点：

1. 去掉顶部那行说明文字，只保留 badge 与分页信息
2. 缩小 `player-strip-card--episode` 的最小高度和内边距

这部分不是故障根因，但和这轮播放器迭代强相关，适合一起沉淀。

## 验证结果

本次修复后完成了下面验证：

1. `npm run lint` 通过
2. `npm run typecheck` 通过
3. `npm run test` 通过，20 文件 88 测试通过
4. `npm run build` 通过
5. 重新执行 `npm run webos:simulator`，在 Simulator 中确认视频已恢复正常连续播放

## 经验沉淀

以后处理播放器与长生命周期组件时，遵守以下规则：

1. 主播放 effect 只允许依赖“真正决定媒体源/播放实例”的状态，不能把导航回调、展示派生数据随手带进去
2. 事件回调如果只需要“最新值”，优先用 `ref` 同步，不要用扩大 effect 依赖的方式强行保持闭包新鲜
3. 当用户反馈“循环 + 无声音”同时出现时，先检查播放器是否被重复重建，再排查音轨、代理和编解码
4. 新增“自动续播 / 自动跳转 / 自动恢复”这类功能时，要先检查它是否会污染播放器生命周期边界
5. 分 P / 设置 / 字幕这类 TV 抽屉界面，删掉冗余说明后要同步复查 header 占位与卡片高度，避免出现新的空白区域

## 后续约束

后续继续改播放器时，至少做下面两件事：

1. 凡是给播放器主 effect 新增依赖，都先问一句：“它是真的影响媒体加载，还是只是为了拿到最新 UI / 跳转数据？”
2. 只要出现播放刚开始就反复重置的症状，第一优先级检查 effect 依赖扩散与 cleanup 是否误触发，而不是先从网络层开始猜
