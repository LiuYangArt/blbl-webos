# 2026-03-22 webOS Simulator B 站媒体 403 与本地代理修复 Postmortem

## 结论

这次 `webOS TV 6.0 Simulator` 上打开 `BV1Sf4y1q7H9` 时看到的“黑屏”，根因不是应用白屏，也不是播放器页面没有挂载，而是：

1. 播放器页本身默认就是黑底沉浸式全屏
2. 这条视频在 Simulator 内发起的媒体请求被 `bilivideo` CDN 拒绝，返回了 `403`
3. 因为视频层没有成功出画，用户肉眼看到的效果就像“整个模拟器是黑屏”

最终修复方案不是继续改 codec 选择，也不是继续怀疑首页/路由挂载，而是：

- **仅在 `webos-simulator` 下**，把视频媒体请求改走本机轻量代理
- 由代理补齐 B 站 CDN 所需的关键请求头：
  - `Referer: https://www.bilibili.com/`
  - 桌面浏览器 `User-Agent`
- 前端播放器仍保持当前 `React + <video>` 主链路
- 真机与普通浏览器路径不受影响

修复后，这条 HDR 样本视频已经在 Simulator 中完成了实际播放验证：

- 实际出画成功
- 自动化 telemetry 收到 `loadedmetadata -> play -> progress`
- 视频元信息为 `1920x1080`
- 当前可稳定验证的是：**1080P AVC 兼容流播放成立**

## 复现现象

用户给出的样本视频：

- BV：`BV1Sf4y1q7H9`
- CID：`240681376`
- 标题：`【4KHDR】可能是效果最好的HDR视频-杜比视界官方演示片`

在 Simulator 中的最初现象：

- 直接进播放器时屏幕看上去是全黑
- 用户按一下遥控器后，播放器 UI 会浮出来
- 页面标题、控制条、画质信息都能显示
- 但视频本身没有画面

控制台进一步显示：

- 媒体资源请求命中 `bilivideo` 地址
- 该请求返回 `403`

这说明：

- App 已启动
- 播放器页已挂载
- 不是 React 首屏白屏
- 真正失败的是**媒体资源直连**

## 已确认事实

### 1. “黑屏”其实是播放器黑底，不等于应用白屏

用户按一下遥控器后，播放器控制条和标题就会显示，说明：

- 当前页面已经在播放器页
- 焦点系统和遥控器响应正常
- 问题不在启动链路

也就是说，这次看到的“黑屏”只是“视频层没有成功出画”的视觉结果。

### 2. 这条视频的 1080P 数据本身能拿到

对 `playurl` 真实接口做了验证：

- 请求 `qn=80`
- 返回 `code=0`
- 返回的兼容流 `format=mp4`
- `accept_quality` 至少包含 `80`
- 可拿到 `1080P 高清` 对应的实际媒体地址

因此问题不是“未登录所以完全拿不到 1080P 数据”。

### 3. 403 的关键在请求头，不在 URL 是否存在

对同一条真实媒体地址做主机侧复现验证，结论非常明确：

- 不带额外头，直接请求媒体 URL：`403`
- 带 `Referer + 浏览器 User-Agent`：`206`

同样的现象也出现在 DASH 分片地址上：

- 不带头：`403`
- 带头：`206`

这说明当前根因不是：

- 链接过期
- 画质请求错误
- CID 错误

而是：

- **Simulator 里的媒体请求上下文不满足 B 站 CDN 要求**

### 4. 之前“Simulator 优先走兼容 MP4”只能绕开一部分问题

此前播放器已经做过：

- Simulator 识别
- 优先走兼容 MP4
- 避免优先走 DASH + Shaka

这能绕开一部分 Simulator 对 DASH 的不稳定问题，但不能解决：

- 兼容 MP4 地址本身仍然会因为缺少 `Referer/User-Agent` 被 `403`

所以仅仅“换一条兼容流”并不能从根上解决这次问题。

## 根因

本次问题的根因可以准确表述为：

> `webOS Simulator` 以 `file://` 应用页面加载播放器时，`HTMLVideoElement` 对 `bilivideo` 媒体地址发起的请求缺少 B 站 CDN 所需的来源信息，导致媒体资源返回 `403`，播放器无法出画，最终在 TV UI 上表现成持续黑底。

## 修复方案

本次采用的是**严格限域**的 Simulator 方案，而不是把代理能力直接推广到所有运行环境。

### 方案内容

新增一个本地轻量代理：

- 文件：`scripts/simulator-media-proxy.mjs`
- 默认监听：`http://127.0.0.1:19033`

代理只做一件事：

- 接收播放器传入的真实 `bilivideo` 媒体 URL
- 向上游请求时补齐：
  - `Referer`
  - `User-Agent`
  - `Origin`
  - `Range`
- 再把上游响应透传回 `<video>`

前端播放器新增一层 URL 重写：

- 文件：`src/features/player/playerMediaProxy.ts`
- 仅当 `deviceClass === webos-simulator` 时生效
- 把 `candidate.videoUrl / audioUrl` 改写为本地代理地址

Simulator 启动脚本同步升级：

- 文件：`scripts/webos-cli.mjs`
- 执行 `simulator` 时自动拉起本地媒体代理

### 为什么这次方案可接受

因为它满足了三个边界：

1. **只影响 Simulator**
2. **不改真机主链路**
3. **不引入之前那套会干扰 UI 启动的 webOS JS Service 原型**

这和此前被暂停的“把 media relay 打进 webOS 应用/服务包里”的实验路线不是同一件事。

## 验证结果

这次修复后，自动化验证脚本已经从“只能收到 environment / attempt-switch”推进到：

- `loadedmetadata`
- `play`
- `progress`

关键验证事实：

- 当前样本：`BV1Sf4y1q7H9 / CID 240681376`
- Simulator 中实际播放模式：`1080P 高清 / AVC / 兼容流`
- 视频分辨率：`1920x1080`
- `decodedVideoFrames` 已增长
- `progress` 事件在 2 秒时成功回传

这说明：

- 媒体请求链路已经打通
- 不是“只显示元信息但没真的播放”
- 当前可以把这条视频作为 Simulator 下的 1080P 功能验证样本

## 教训

### 1. 先分清“应用黑屏”和“视频黑屏”

这次如果只凭肉眼看“整个屏幕是黑的”，很容易继续往：

- 启动链路
- React 挂载
- legacy bundle
- boot route

这些方向误判。

实际上用户一按遥控器，UI 就出来了，这已经足够说明问题更偏向播放器媒体层。

### 2. 只验证 API 返回不够，必须验证媒体 URL 的实际可读性

此前我们已经知道：

- `playurl` 能返回 `1080P`
- 甚至能返回 DASH/兼容流地址

但这并不代表 `<video>` 真能在当前环境把资源拉下来。

这次真正锁定根因的关键，是直接验证：

- 同一个媒体 URL
- 不同请求头组合
- 对应的 `403 / 206`

### 3. Simulator 问题必须和真机问题分层处理

这次的本地代理修复是一个 **Simulator 验证增强方案**，它解决的是：

- “如何让 Simulator 能稳定验证一条 B 站视频在当前播放器里的基础播放闭环”

它不自动等价于：

- 真机以后也必须走代理
- 真机 4K/HDR 正式能力已经被证明

Simulator 和真机仍然要分别验证。

## 后续建议

1. 保留当前 Simulator 本地代理方案，作为后续播放器功能开发的稳定验证底座。
2. 在文档和口径上明确区分：
   - “已验证 Simulator 下 1080P 可播”
   - “尚未证明真机 HDR/4K 可稳定播”
3. 后续如果继续推进 HDR / 4K / DASH 轨道验证，优先先回答两个问题：
   - 当前账号是否真正拿到了对应画质轨
   - 真机路径是否也会受请求头与媒体执行层限制
4. 不要把这次 Simulator 本地代理成功，误解成此前那套 webOS service relay 方案也应恢复。
