# 2026-03-21 Spatial Focus 焦点滚动不跟随 Postmortem

## 结论

本次问题不是“按下键时没有找到下一个焦点”，而是：

- **焦点已经正确移动到了下一张卡片**
- **但页面滚动被我们自己禁掉了**

更准确地说，问题根因是新焦点引擎在切换焦点时统一使用了：

```ts
element.focus({ preventScroll: true })
```

这会阻止浏览器默认的“当焦点元素跑到视口外时，把页面滚到可见区”的行为。

因此用户在详情页“相关推荐”区域看到的真实现象是：

1. 焦点系统能正常算出下一目标
2. 下一目标实际上已经被 focus 到
3. 但屏幕停在原位不动
4. 用户视觉上就会误以为“向下键失效”或“焦点没继续往下走”

最终修复不是改导航算法，而是补回 **TV 焦点切换后的视口跟随机制**。

## 用户可见症状

- 在详情页“相关推荐”第一排卡片上按遥控器“向下”
- 正确预期：页面向上滚动，露出下面一排内容，同时焦点继续停在新的卡片上
- 实际表现：页面不滚动，看不到下面的卡片

这个现象很容易让人第一反应怀疑：

- section `leaveFor` 没写对
- 空间导航算法没找到候选
- 页面没有可继续聚焦的下一个元素

但这些都不是这次的主因。

## 根因拆解

### 根因一：焦点导航本身是工作的

本轮重构后，详情页“相关推荐”区域已经接入新的：

- `Section`
- `focusId`
- 空间几何候选计算

从实际表现看，按方向键时没有出现：

- 焦点回到左导航
- 焦点丢失
- 焦点停在原位不变

这说明“焦点目标选择”本身不是主要问题。

### 根因二：我们在引擎里统一禁掉了浏览器默认滚动

问题位置：

- `src/platform/focus/engine.ts`

为了避免切换焦点时页面出现突兀跳动，焦点引擎里统一写成了：

```ts
element.focus({ preventScroll: true })
```

这在“焦点目标本来就在当前视口内”的情况下是合理的。

但一旦目标已经落到当前视口外，浏览器原本负责的自动滚动也一起被禁止了。

于是我们得到了一种很有迷惑性的中间态：

- 状态层：焦点已经切换成功
- 视觉层：页面完全没动

用户只看视觉层，自然会认为行为不对。

### 根因三：TV 焦点系统不能只负责“算焦点”，还必须负责“把焦点带进视口”

这轮重构暴露出一个很重要的设计约束：

- 在 TV 场景里，“焦点切换”与“滚动跟随”不是两个可以分开的体验问题

只要页面存在纵向超出一屏的内容区，焦点系统就必须保证：

1. 目标元素可聚焦
2. 目标元素被聚焦后能进入用户可见区域

否则用户会遇到一种非常糟糕的体验：

- 焦点逻辑其实是对的
- 但屏幕没有反馈
- 看起来就像遥控器失灵

## 修复内容

### 1. 保留 `preventScroll`

这一步没有回退。

原因是如果直接删掉 `preventScroll`，浏览器默认滚动行为在不同页面和不同容器上可能不够稳定，容易出现：

- 滚动距离过大
- 触发错误容器滚动
- 焦点切换时画面突兀跳动

所以修复原则不是“完全恢复默认行为”，而是：

- **继续由引擎掌控焦点**
- **在需要时手动补可视区滚动**

### 2. 在焦点切换后补 `ensureElementVisible`

最终在 [engine.ts](/F:/CodeProjects/bilibili_webos/src/platform/focus/engine.ts) 中改成：

1. 先执行 `focus({ preventScroll: true })`
2. 再检测目标元素是否已经跑出当前视口
3. 如果超出视口，则执行：

```ts
element.scrollIntoView({
  block: 'nearest',
  inline: 'nearest',
  behavior: 'smooth',
})
```

### 3. 采用最小滚动策略，而不是整页强制对齐

这里特意选择：

- `block: 'nearest'`
- `inline: 'nearest'`

而不是：

- `block: 'start'`
- `block: 'center'`

原因是 TV 页面卡片密度较低、卡片尺寸较大，强制把每次焦点都滚到顶部或中间，会造成画面频繁大跳。

`nearest` 的目标更符合电视端预期：

- 只在必要时滚
- 只滚到刚好看得见
- 不打断用户对当前页面结构的空间记忆

## 为什么这个问题容易误判

这类问题非常容易被误判成“导航算法错误”，原因有三个：

1. 用户操作是按方向键，直觉上最先怀疑焦点算法。
2. 视觉结果是“看不到下一项”，很像“没跳过去”。
3. 如果不看 DOM 实际焦点，只看屏幕，很难分辨：
   - 是“焦点没变”
   - 还是“焦点变了但屏幕没跟”

所以后续再遇到“按键后看起来没移动”的问题时，必须先把这两层分开判断：

1. 焦点目标有没有变
2. 视口有没有跟随到可见区

## 经验沉淀

后续继续维护 TV 焦点系统时，必须记住下面这些规则：

1. `preventScroll: true` 不能单独使用；只要禁掉浏览器默认滚动，就必须补一层显式的可视区跟随逻辑。
2. 电视端焦点系统不只是“找下一个元素”，还必须负责“确保用户看得到这个元素”。
3. “焦点正确但画面不动”在用户体验上几乎等价于“焦点失效”，不能把它当成小问题。
4. 出现“按下键没反应”的反馈时，先区分是导航失败还是滚动跟随失败，不要一上来就改空间导航评分。
5. 对 TV 列表和网格，优先选择最小滚动策略，避免每次焦点切换都造成整屏大跳。

## 后续约束

后续只要新增以下任一场景，都必须同步检查焦点滚动跟随：

1. 超过一屏的详情页
2. 搜索结果页长列表
3. 历史记录与收藏列表
4. 播放器相关推荐和设置抽屉
5. 任何会在同一页面内纵向推进的新内容区

如果未来再对焦点引擎做重构，验证清单里应明确包含：

1. 焦点能否正确切换
2. 焦点离开视口后页面是否自动滚动
3. 滚动是否只发生必要的最小距离
