# 2026-03-26 TV 视频列表动态加载与 FocusOverlay 性能优化 Postmortem

## 结论

这轮用户反馈的核心不是“接口太慢”，而是：

1. 视频列表在接近底部并动态补内容时，主线程会同时承受列表渲染、图片解码、焦点白框跟随和滚动测量的压力
2. 电视端的卡顿更容易来自 DOM / 布局 / 图片 / overlay，而不是纯计算
3. 在当前阶段，`Web Worker / WASM` 不是第一优先级；更高收益的做法是先把主线程上不必要的 UI 开销降下来

最终采用的策略是：

- 先优化列表增量渲染与图片解码策略
- 再对 `FocusOverlay` 做低风险减负
- 明确不改 `focus engine` 的导航规则、滚动根、section 语义

这条路线在真机上已经拿到用户正反馈：

- 列表动态加载“有改善”
- 在继续做 `FocusOverlay` 优化后，仍保持当前焦点链路稳定

## 用户可见现象

用户最早的主诉是：

- 电视上视频列表动态加载时仍然比较卡
- 希望确认是否还可以继续优化
- 也主动提到了“多线程 / wasm”这类方案

这类描述如果只看表面，很容易直接把方向带去：

- 把接口处理搬进 Worker
- 或者开始找适合上 WASM 的计算热点

但结合当前仓库结构和真机场景，真正高风险的不是“算不动”，而是：

- 一次 append 新卡片时 DOM 同帧增量太大
- 老卡片跟着一起重渲染
- 一批封面图同时解码
- 焦点白框 overlay 在列表增长时重复测量和更新

## 根因判断

### 1. 首要瓶颈是主线程 UI 成本，而不是纯计算

检查当前实现后可以确认：

- 首页推荐流会持续 append 新卡片
- 焦点引擎本身需要维护 section 和几何信息
- `FocusOverlay` 会监听全局变化并读取当前焦点元素位置
- 卡片海报使用普通 `<img>`，在电视芯片上同批解码成本明显

这类组合在桌面浏览器上可能只是“有点顿”，但在 TV 端会更快放大成：

- 动态加载一瞬间掉帧
- 白框跟随观感发紧
- 用户主观觉得“页面卡了一下”

### 2. `Worker / WASM` 不是当前最高收益解法

这次很重要的一个判断是：

- `Worker` 更擅长数据预处理、去重、排序、解析
- `WASM` 更适合重计算、编解码、图像处理或底层算法

但这轮主观卡顿主要发生在：

- React 提交
- DOM 插入
- 样式计算与布局测量
- 图片解码
- 焦点白框 overlay 的同步更新

这些工作绝大部分仍然在主线程上，因此：

- 即便先上 `Worker / WASM`
- 也未必能直接改善“动态补列表时卡一下”的真机体感

结论是：

- 先优化主线程 UI 路径
- 再决定是否还有必要引入更重的技术方案

### 3. 焦点相关改动必须遵守既有复盘边界

仓库里已有多篇焦点复盘已经证明：

- 白框 overlay 方案本身是当前最稳的方向
- `focus engine` 的 `focusElement -> ensureFocusedElementComfort` 链路不能随意碰
- `FocusSection`、滚动根、返回链路、section 语义都属于高敏感区域

因此这轮优化从一开始就明确约束：

- 不改导航算法
- 不改 section 规则
- 不改滚动根识别
- 只在列表层和 overlay 层减负

这条边界是这次能“先提速、又不把 focus 搞坏”的关键。

## 修复方案

### 1. 列表增量渲染先减峰值

对 `VideoGridSection` 的处理分两层：

1. 把一次性 reveal 改成分帧揭示，避免同一帧插入过多卡片
2. 把单个网格项拆成 memo 化组件，降低旧卡片被无关状态带着重渲染的概率

这一步的目的不是“让列表变得更花哨”，而是：

- 降低 append 时的瞬时主线程峰值
- 让动态加载的卡顿从“一下子挤爆”变成“更平滑地展开”

### 2. 首页列表项映射收敛成稳定引用

首页原本会在 render 中频繁把原始数据重新映射为统一的 `video list item`。

这会带来一个典型问题：

- 即使只是 `isLoadingMore`、错误信息或焦点辅助状态变化
- 前面已经显示的整批卡片也可能一起重新参与 render

这次改成：

- 用 `useMemo` 收敛各频道的视频项映射
- 让真正变化的频道数据才触发对应列表项重建

同时首页推荐继续 append 时，使用：

- `startTransition`

把新增一批推荐的 UI 更新降成低优先级，尽量少抢当前焦点移动与遥控器操作反馈。

### 3. 图片解码策略改为“近屏更积极、远处更保守”

卡片海报加上了：

- `decoding="async"`
- 根据位置区分 `eager / lazy`

这样做的目标很直接：

- 首批可见卡片尽快稳定显示
- 后续尚未进入用户主要视野的卡片，不要和当前帧抢太多解码预算

在 TV 端，这类图片策略往往比“继续优化请求并发”更接近用户体感。

### 4. `FocusOverlay` 只做低风险减负

这一轮没有回头改焦点规则，而是只在 overlay 自己的实现上减负：

1. 只有 snapshot 真变化时才触发状态更新
2. 生产态不再为 HUD 额外采集大量调试文本
3. MutationObserver 尽量观察当前焦点所在的滚动/section 根，而不是一直盯整棵 `document.body`
4. 保留已有跟随帧策略，但让它在更少的无效变化上工作

这一步的价值是：

- 不影响现有焦点导航和滚动修正逻辑
- 但能减少白框自己带来的额外测量和重渲染

## 为什么这轮方案更合适

这次最关键的经验是：

- **不要把“感觉卡”自动翻译成“先上多线程”**

在 TV 应用里，尤其是 React + 焦点系统 + 大图卡片列表的组合场景里，更高收益的路径往往是：

1. 先减少一帧里要新增多少 DOM
2. 先减少旧节点的重复 render
3. 先减少图片解码冲击
4. 先减少 overlay 自己的观测与更新成本

只有当这些都做完后，仍然能明确看到“纯数据处理很重”，才值得继续把 Worker / WASM 拉进来。

## 经验沉淀

1. TV 列表页卡顿优先排查主线程 UI 成本，不要直接跳到 `Worker / WASM`
2. 只要涉及焦点链路，优先区分“导航引擎问题”与“overlay 观感问题”，不要混着改
3. 对焦点系统的性能优化，优先从 overlay 自己减负，而不是先动核心导航算法
4. 列表追加优化要优先控制“每帧新增多少 DOM”，而不是只盯请求耗时
5. 真机上只要出现“感觉还是旧行为”，必须继续坚持入口 hash 核对，不要凭肉眼猜

## 后续建议

如果后面还要继续做列表页性能优化，建议按下面顺序推进：

1. 继续观察 `FocusOverlay` 在高频滚动和大列表下的真机体感，必要时再补专项测试
2. 如果列表仍会随滚动越积越多，下一步优先做“行级窗口化 / 受控 DOM 数量”
3. 只有当 UI 路径已经明显收敛后，再评估是否存在值得挪到 Worker 的数据预处理热点
4. 除非后续出现明确的重计算场景，否则不要为了“听起来更高级”而引入 WASM
