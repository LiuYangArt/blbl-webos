# 2026-03-25 焦点框可见性回归与视频列表加载门控 Postmortem

## 1. 背景

本轮在 `issue #19` 的 TV 端交互优化过程中，出现了两类连续反馈：

1. 焦点白框在遥控器操作下不可见，且上下键滚动体验异常
2. 焦点白框恢复后，首屏加载阶段仍能看到布局位置变化，导致白框跟随过程不稳定、观感生硬

这两类问题都直接影响 TV 端可操作感和首屏稳定性。

## 2. 现象与影响

### 2.1 焦点白框不可见 / 焦点残留

- 左侧菜单收起后，白框偶发停留在已不可见区域
- 遥控器上下键可触发焦点变化，但页面滚动感知不明显

影响：

- 用户无法稳定判断当前可操作目标
- 5-way 导航信心下降，交互反馈出现断层

### 2.2 首屏加载阶段布局变化可见

- 首页刚进入时，频道按钮与内容区会在加载中发生位移
- 白框初始套在按钮上，但按钮位置变化时白框跟随不稳

影响：

- 首屏“在用户可见阶段发生重排”，造成明显抖动感
- 白框视觉质量下降，放大“卡顿/掉帧”的主观感受

## 3. 根因分析

### 3.1 焦点可见性链路被打断

- `FocusOverlay` 在真实 `activeElement` 与标记元素切换时，存在白框追踪窗口不足的问题
- 菜单收起与焦点流转时，`data-focus-active` 清理不及时，导致旧焦点残留
- 焦点移动后“舒适区滚动”链路缺失时，出现“焦点变了但页面没跟上”的体验落差

### 3.2 首屏布局稳定性策略不足

- 当前列表页虽然有 loading 文案，但仍会让真实页面结构提前进入可见区
- 数据与登录态是异步收敛，导致页面在可见状态下逐步重排
- 焦点框在重排过程中被迫追踪移动，表现为“跟随滞后”

## 4. 已确认的有效做法

1. 恢复生产态焦点白框显示，HUD 仍由 debug 开关控制
2. 恢复焦点后的舒适区滚动逻辑，保证遥控器上下导航可见反馈
3. 增加 `focusout` 清理与重标记逻辑，降低菜单收起后的焦点残留
4. 在 `FocusOverlay` 增加轻量跟随帧，降低布局短时变化导致的白框掉队

## 5. 新策略决策：统一 Loading Gate

针对“首屏可见重排”，决定采用统一方案而非页面级补丁：

- 所有视频列表页在关键数据未就绪前，仅显示统一 Loading 页面
- 不提前渲染真实列表结构，不让用户看到加载过程中的位置变化
- 数据与关键登录态稳定后，再一次性切换到正式页面

对应计划文档：`docs/plans/2026-03-25-video-list-loading-gate-plan.md`

## 6. 过程经验（可复用）

1. TV 端问题优先级应按“可操作反馈”排序：焦点可见 > 焦点可达 > 页面滚动可见 > 视觉细节
2. 对“首屏抖动”类问题，优先考虑“可见阶段隔离”，而不是继续打磨白框跟随算法
3. 用户明确要求“先方案后实现”时，应先冻结代码改动，仅输出可执行文档，再进入实现阶段
4. 焦点引擎与视觉层联动问题需要双向排查：只看焦点逻辑或只看 CSS 都容易误判

## 7. 后续行动

1. 按 Loading Gate 方案统一接入首页、热门、历史、关注、收藏/稍后、搜索结果、订阅、作者页
2. 完成后在 `LG C1` 真机复测“首屏可见重排”是否消失
3. 若仍有个别页面抖动，再定位页面内部二次布局来源（图片尺寸、异步模块、tabs 高度变化）

## 8. 二次排障补充

首轮按方案接入后，用户在真机和 Simulator 上继续反馈两类问题：

1. Loading 黑层没有盖住右侧主区域的完整边框与顶部 profile 区
2. Loading 消失后，列表页仍会出现轻微二次位移

这说明“已有 loading gate”并没有真正作用在用户看到的最终层级，也没有覆盖首屏稳定性的完整条件。

## 9. 最终确认的根因

### 9.1 Loading 挂载层级错了

- 之前的 `VideoListLoadingPage` 只是页面组件自己返回的一个黑块
- 它只能盖住 `page-shell` / `tv-page-content` 内部内容
- 右侧主区域外圈的安全边距、顶部 profile 区、背景层仍属于 `tv-app-main`
- 所以用户看到的是“黑块在页面里面”，不是“右侧主区域整体进入 loading”

最终修正：

- 把 loading 提升到 `AppShell` 层
- 由 `RouteLoadingOverlayProvider` 统一管理右侧主区域 overlay
- 规则固定为：**只有左侧菜单栏保持在 loading 上面，右侧其余内容全部在 loading 下面**

### 9.2 “接口成功”不等于“首屏稳定”

- 列表接口返回成功时，首屏封面图仍可能未完成解码/渲染
- 首页 `个性推荐` 还会在首屏后继续自动补一批内容
- 这两种情况都会导致 loading 虽然已经消失，但页面仍继续长高或重排

最终修正：

- 首屏 loading 关闭条件增加“首批封面图预加载完成”
- 首页把当前频道首屏需要的图片集合纳入 gate
- 自动补更多内容不再作为 loading 消失后的第一帧可见变化来源

## 10. 可复用经验

1. TV 端“全屏 loading”要先确认它覆盖的是**用户感知区域**，不是某个页面组件自己的内容盒子
2. 遇到“loading 看起来没盖全”的问题，先检查父容器层级、padding、安全边距、sticky/expanded 侧栏，不要只改 loading 自身高度
3. 遇到“loading 消失后仍抖一下”，优先怀疑：
   - 首屏图片未就绪
   - 首屏卡片数仍在增长
   - tabs / section / helper text 在首帧后才进入布局
4. 对 TV 列表页，`ready` 更准确的定义应是：
   - 关键数据已稳定
   - 关键登录态已稳定
   - 首屏所需图片已就绪
   - 首帧不会再发生可见结构扩张
