# Task Plan: PiliPlus 功能迁移规划

## Goal
梳理参考项目 `PiliPlus` 的能力边界，结合当前 `bilibili_webos` 的真实基线，产出一份可长期执行的 WebOS 端功能迁移计划文档。

## Phases
- [x] Phase 1: 审视当前 WebOS 仓库结构与已有能力
- [x] Phase 2: 调研参考项目的功能地图、入口结构与关键模块
- [x] Phase 3: 设计适合 TV 场景的迁移分期与模块映射
- [x] Phase 4: 编写正式计划文档并同步研究笔记
- [ ] Phase 5: 按计划逐阶段实施与验收

## Key Questions
1. `PiliPlus` 中哪些能力属于 WebOS MVP 的主路径，应该优先迁移？
2. 哪些能力明显带有移动端/桌面端特征，不适合早期照搬到 TV？
3. 当前 `bilibili_webos` 缺的首先是页面，还是更底层的路由、数据、播放器、鉴权与焦点基础设施？
4. 如何把“照着抄”变成可执行的里程碑，而不是失控的大而全需求？

## Decisions Made
- 不按 `PiliPlus/lib/pages/*` 一页页机械移植，而按业务域和 TV 关键路径拆分阶段。
- 迁移优先级遵循：播放闭环 > 遥控器可用 > 焦点稳定 > 内容资产 > 社交长尾功能。
- 先覆盖首页、搜索、详情/播放器、登录、历史、收藏/稍后再看、个人中心等 TV 主路径。
- 动态、私信、发布、DLNA、离线缓存、复杂设置等重功能先后置，不在首批闭环内承诺。

## Errors Encountered
- 使用 `rg` 搜索 `video` 目录能力点时，PowerShell 将 `|` 解释为管道，导致命令失败；后续改为更小粒度的文件定向读取规避。

## Status
**Currently in Phase 5** - 迁移规划文档已写入，下一步可按文档从首页数据化与路由扩展开始实施。

# Task Plan: 首页登录态分区与高画质能力规划

## Goal
基于当前 `bilibili_webos` 仓库与安卓参考项目 `PiliPlus` 的真实实现，产出一份针对“首页正在关注/订阅剧集入口”以及“1080P/4K/HDR 能力” 的专项计划文档，指导后续 WebOS 端落地。

## Phases
- [x] Phase 1: 审视当前 WebOS 首页、播放器与登录态能力
- [x] Phase 2: 调研安卓参考项目在播放取流、画质切换、关注更新、PGC 订阅上的实现
- [x] Phase 3: 归纳 TV 端可复用能力与当前差距
- [x] Phase 4: 编写专项计划文档并同步研究笔记
- [ ] Phase 5: 按计划逐阶段实施首页二级频道与播放器画质能力

## Key Questions
1. 当前 WebOS 仓库为什么登录后仍主要只能走到 `720P`？
2. 安卓参考项目是否已经具备 `1080P+ / 4K / HDR / 杜比视界` 等能力的数据入口与切换逻辑？
3. `正在关注` 与 `订阅剧集` 在安卓参考项目里分别依赖哪些接口与页面结构？
4. WebOS 端应把这些能力做成左侧一级导航，还是首页里的二级频道条？

## Decisions Made
- 首页的 `正在关注` / `订阅剧集` 优先做成首页二级频道，而不是左侧新增一级导航，保持 TV 一级导航克制。
- 高画质规划拆成两层：先解决 `1080P / 4K` 的“可获取、可切换、可提示”，再评估 `HDR / 杜比 / HDR Vivid` 的真实播放链路。
- `正在关注` 不直接照搬移动端动态页，而是优先抽取可在 TV 首页稳定展示的“可播放内容更新”子集。
- `订阅剧集` 以 `PGC follow list` 为主数据源，首版聚焦“最近追番/追剧”的可播放内容入口。

## Errors Encountered
- PowerShell / shell tool 在带有 `|` 的正则检索上多次把关键词误解释为管道命令；后续改为单关键词、小粒度检索与定向文件读取规避。

## Status
**Currently in Phase 5** - 专项计划文档已写入，下一步可按文档优先实现首页二级频道框架与播放器画质切换入口。

# Task Plan: TV 焦点舒适区滚动重构规划

## Goal
调研 `jellyfin-webos` 的 TV 端表现与当前 `bilibili_webos` 的焦点引擎现状，产出一份统一的“焦点舒适区滚动”重构文档，为后续所有页面提供一致的上/下导航与滚动策略。

## Phases
- [x] Phase 1: 审视当前 WebOS 仓库的焦点与滚动现状
- [x] Phase 2: 调研 `jellyfin-webos` 仓库，确认可借鉴的是产品效果还是具体实现
- [x] Phase 3: 设计适合本仓库的统一 TV 焦点舒适区滚动方案
- [x] Phase 4: 编写正式重构文档并同步研究笔记
- [ ] Phase 5: 等用户确认方案后再进入编码实施

## Key Questions
1. 当前“焦点回来了，但 section 顶部信息没回来”的根因是页面个例，还是全局滚动策略问题？
2. `jellyfin-webos` 的仓库代码里，是否真的包含可直接移植的 TV 长列表焦点滚动实现？
3. 我们应不应该继续按页面打补丁，还是把焦点引擎升级为统一的 TV 视口舒适区模型？
4. 这套方案如何兼容首页、关注页、热门页、搜索页、历史页等所有长页面？

## Decisions Made
- 本轮只做研究与方案文档，不做代码实现。
- `jellyfin-webos` 只能作为“产品效果参考”，不能直接作为当前问题的实现参考。
- 后续应从焦点引擎层统一处理“焦点舒适区滚动”，而不是在各页面继续补 `scrollIntoView()`。
- 页面层只声明 section 的滚动意图与头部保留规则，不再各自定义一套滚动行为。

## Errors Encountered
- 使用 `rg` 在 PowerShell 下检索带 `|` 的多关键词模式时，多次被解释为管道命令；后续改为 `Select-String` 与单模式检索规避。
- `jellyfin-webos` 仓库并不包含完整内容页前端，初看容易误判为可直接参考其焦点实现；进一步读取后确认它主要是 iframe 壳层。

## Status
**Currently in Phase 5** - 统一方案文档已写入，等待用户确认后再开始实现。
